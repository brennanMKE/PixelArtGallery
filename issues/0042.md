## Description

`MARKETING_VERSION` (1.0) and `CURRENT_PROJECT_VERSION` (1) currently live as target-level settings inside `project.pbxproj` тАФ the kind of duplicated, easy-to-forget location that ships DMGs self-reporting the wrong version. Move the marketing version to a single xcconfig location, and have the release pipeline derive the build number automatically so Sparkle always sees a monotonically increasing value.

## Expected behavior

- `MARKETING_VERSION` lives in exactly one place тАФ `Configuration/Build.xcconfig` тАФ and the pbxproj carries no per-target override (per the #0023 precedence lesson, a leftover pbxproj line would silently win).
- Bumping a release version is a one-line change committed as `Bump version to <X.Y.Z>` (SemVer).
- `scripts/release.sh` (#0037) overrides `CURRENT_PROJECT_VERSION` at archive time with today's UTC date in `YYYYMMDD` form тАФ the build number Sparkle compares тАФ and prints it at the end of the run. The static value left in the project is only seen by Xcode-driven debug builds.
- Both platform builds still report correct `CFBundleShortVersionString`/`CFBundleVersion`.

## Notes

- Verify with `xcodebuild -showBuildSettings` (both destinations) and by dumping the built bundles' Info.plists.
- The preflight (#0043) enforces the discipline: fails when a `MARKETING_VERSION` line reappears in `project.pbxproj`, and fails when the version isn't strictly greater than the newest item already in `website/appcast.xml`.

## Root cause

`MARKETING_VERSION = 1.0;` and `CURRENT_PROJECT_VERSION = 1;` lived as target-level settings in both the Debug and Release configurations of `PixelArtGallery.xcodeproj/project.pbxproj`. Per the #0023 precedence lesson, target-level pbxproj settings silently beat anything in the base xcconfig, so any future xcconfig-side version bump would have been ignored and DMGs could ship self-reporting the wrong version. The value was also `1.0` (two-part), while the changelog/appcast convention (#0041) is SemVer three-part, so the generated changelog link `#v1-0` would have missed the existing `id="v1-0-0"` anchor.

## Fix

- Deleted `MARKETING_VERSION` and `CURRENT_PROJECT_VERSION` from both target-level configurations in `project.pbxproj` (grep for either key now returns zero hits in that file).
- Added to `Configuration/Build.xcconfig` (already wired as `baseConfigurationReference` for both configurations): `MARKETING_VERSION = 1.0.0` and `CURRENT_PROJECT_VERSION = 1`, with a comment noting that release bumps are a one-line change here and that `scripts/release.sh` overrides the build number at archive time.
- Normalized the marketing version 1.0 -> 1.0.0 (SemVer), aligning with the `#v1-0-0` changelog anchor.
- No script changes needed: `scripts/release.sh` reads `MARKETING_VERSION` from `xcodebuild -showBuildSettings` and passes `CURRENT_PROJECT_VERSION=$(date -u +%Y%m%d)` as a command-line override to `xcodebuild archive`, and command-line settings outrank both pbxproj and xcconfig. `PixelArtGallery/Info-macOS.plist` already uses `$(MARKETING_VERSION)` / `$(CURRENT_PROJECT_VERSION)` substitution (#0039), so the new values flow through unchanged.

## Verification

- `grep -n 'MARKETING_VERSION\|CURRENT_PROJECT_VERSION' PixelArtGallery.xcodeproj/project.pbxproj` -> no hits (exit 1).
- `xcodebuild -showBuildSettings` for `platform=macOS` and `platform=iOS Simulator,name=iPhone 17 Pro` -> both report `MARKETING_VERSION = 1.0.0`, `CURRENT_PROJECT_VERSION = 1`.
- macOS build (signed) and iOS Simulator build (`CODE_SIGNING_ALLOWED=NO`) -> both `** BUILD SUCCEEDED **`; `plutil -extract CFBundleShortVersionString raw` on both built Info.plists -> `1.0.0` (CFBundleVersion `1`).
- `./scripts/release.sh --dry-run` -> archive/export/codesign all passed; exported app reports `CFBundleShortVersionString = 1.0.0`, `CFBundleVersion = 20260706` (today's UTC date), proving the command-line override still beats the xcconfig static value.
- `cd PixelArtGalleryKit && swift test` -> 92 tests, 0 failures.
- `website/changelog.html` anchor `id="v1-0-0"` now matches the `#v1-0-0` link #0041's generator emits for version 1.0.0 -> no changelog change needed.

## Files changed

- `PixelArtGallery.xcodeproj/project.pbxproj` тАФ removed `MARKETING_VERSION` and `CURRENT_PROJECT_VERSION` from both target-level configurations.
- `Configuration/Build.xcconfig` тАФ added `MARKETING_VERSION = 1.0.0` and `CURRENT_PROJECT_VERSION = 1` as the single source of truth.

## Gotchas

- If Xcode's target editor is ever used to change the version, it writes `MARKETING_VERSION` back into `project.pbxproj`, which silently overrides the xcconfig again. Bump versions only by editing `Configuration/Build.xcconfig`; the #0043 preflight fails the release when a `MARKETING_VERSION` line reappears in the pbxproj.
- The static `CURRENT_PROJECT_VERSION = 1` in the xcconfig is intentional: Xcode-driven debug builds need some value, and the release pipeline never uses it (command-line override at archive time wins over every file-based setting).
