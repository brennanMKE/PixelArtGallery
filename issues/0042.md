# 0042 — Single source of truth for versioning: MARKETING_VERSION in xcconfig, date-derived build number

| | |
|---|---|

## Description

`MARKETING_VERSION` (1.0) and `CURRENT_PROJECT_VERSION` (1) currently live as target-level settings inside `project.pbxproj` — the kind of duplicated, easy-to-forget location that ships DMGs self-reporting the wrong version. Move the marketing version to a single xcconfig location, and have the release pipeline derive the build number automatically so Sparkle always sees a monotonically increasing value.

## Expected behavior

- `MARKETING_VERSION` lives in exactly one place — `Configuration/Build.xcconfig` — and the pbxproj carries no per-target override (per the #0023 precedence lesson, a leftover pbxproj line would silently win).
- Bumping a release version is a one-line change committed as `Bump version to <X.Y.Z>` (SemVer).
- `scripts/release.sh` (#0037) overrides `CURRENT_PROJECT_VERSION` at archive time with today's UTC date in `YYYYMMDD` form — the build number Sparkle compares — and prints it at the end of the run. The static value left in the project is only seen by Xcode-driven debug builds.
- Both platform builds still report correct `CFBundleShortVersionString`/`CFBundleVersion`.

## Notes

- Verify with `xcodebuild -showBuildSettings` (both destinations) and by dumping the built bundles' Info.plists.
- The preflight (#0043) enforces the discipline: fails when a `MARKETING_VERSION` line reappears in `project.pbxproj`, and fails when the version isn't strictly greater than the newest item already in `website/appcast.xml`.
