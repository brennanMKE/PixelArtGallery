## Description

Running the package test suite pollutes the user's real gallery storage: `FileStorageManagerTests` exercises `FileStorageManager` against its default location, which outside the app sandbox is `~/Library/Application Support/PixelArtGallery/Images`, and tests like `testSaveAndLoadImageData` never delete the files they write. Every `swift test` run leaves small fixture `.dat`/image files behind in the directory the real app reads, and could in principle collide with real user data. Discovered while verifying #0033.

## Expected behavior

Tests run against an isolated temporary directory (e.g. `FileManager.temporaryDirectory` subfolder created per test/suite and removed in teardown), or at minimum delete every file they create. A `swift test` run leaves the user's Application Support untouched.

## Actual behavior

Fixture files accumulate in `~/Library/Application Support/PixelArtGallery/Images` after each test run.

## Notes

- `PixelArtGalleryKit/Tests/PixelArtGalleryKitTests/Persistence/FileStorageManagerTests.swift` — the leaking tests.
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Persistence/FileStorageManager.swift` — check whether it can be initialized with a custom base directory; if not, adding an injectable base URL (defaulting to the current behavior) is the clean fix and keeps production behavior unchanged.

## Root cause

`FileStorageManager` had a single `init()` hard-wired to `Application Support/PixelArtGallery/Images`, so `FileStorageManagerTests` had no way to point it anywhere else, and `testSaveAndLoadImageData` never deleted the file it wrote (the round-trip test's `defer { Task { … delete } }` cleanup was also racy). In addition, `GalleryCoordinatorTests` was leaking through the same path: `GalleryCoordinator.createGalleryItem` lazily creates `FileStorageManager()` internally, so every coordinator test that imported an image wrote a fixture PNG into the real directory too. 366 leaked `.dat` files had accumulated.

## Fix

- `FileStorageManager.init(imageDirectory: URL? = nil)` — an injectable storage directory. `nil` (the default) preserves the exact production behavior, so the only production call site (`GalleryCoordinator.fileStorageManager()`, still `try FileStorageManager()`) is unchanged.
- `GalleryCoordinator.init(fileStorage: FileStorageManager? = nil)` — lets tests inject a store; all existing `GalleryCoordinator()` call sites (GalleryListView, previews, DisplayMergeTests) are unchanged and keep the lazy default.
- `FileStorageManagerTests`: per-test unique temp directory (`FileManager.default.temporaryDirectory/FileStorageManagerTests-<UUID>`) created in `setUp`, removed in `tearDown`; removed the racy defer-Task cleanup; added `testInitWithCustomDirectoryStoresFilesThere` (suite goes 81 → 82 tests).
- `GalleryCoordinatorTests`: same per-test temp directory pattern with a `makeCoordinator()` helper that injects `FileStorageManager(imageDirectory:)`.
- Cleaned up the 366 previously leaked files: a one-off script regenerated the exact fixture payloads (`"Test image data"`, `"Existence test"`, and the solid-color PNGs at 7×11, 16×16, 24×24, 400×300, 400×200) and deleted only byte-identical matches. All 366 files matched exactly; the directory now contains no user data loss risk — nothing non-fixture was present or touched.

## Gotchas

- The issue title only names `FileStorageManagerTests`, but `GalleryCoordinatorTests` was the bigger leaker (255 of the 366 files were its 16×16 PNG imports) because `createGalleryItem` builds the default `FileStorageManager` internally — fixing only the storage tests would not have stopped the leak.
- Every one of the 366 leaked files byte-matched a regenerated fixture payload, confirming the real directory contained no actual user images; deletion was restricted to exact content matches as a safety guarantee.
