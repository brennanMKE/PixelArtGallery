# 0053 — Stopping a send should clear the layer by sending an all-black frame

## Description

When a continuous send stops (Stop button, or leaving the view per #0052), the app just stops pushing frames — the last image lingers on the display until the FT layer times out. Stopping should actively clear the layer by sending one final frame of all-black pixels, which the FT server treats as transparent on overlay layers, erasing the image immediately.

## Steps to reproduce

1. Send a variant continuously to a display (layer 1–15).
2. Tap **Stop Sending** (or navigate away).
3. The image stays on the display until the server's layer timeout (~15s) instead of clearing right away.

## Expected behavior

Stopping the send immediately clears the painted layer: a final all-black frame (same dimensions, same layer) is sent, so the overlay disappears at once.

## Actual behavior

Stopping only cancels the refresh loop; no clearing frame is sent, so the last image persists until the layer times out.

## Long Description

FlaschenTaschen treats black as transparent on layers above the background. Verified in the server compositor (`server/composite-flaschen-taschen.cc`):

```cpp
if (col.is_black()) {
    // Transparent pixel. Find closest stacked below us that is not.
    for (/**/; layer > 0; --layer) {
        if (!screens_[layer]->At(x, y).is_black()) break;
    }
    delegatee_->SetPixel(x, y, screens_[layer]->At(x, y));
}
```

The app only ever paints layers 1–15 (#0047, never 0), so an all-black frame sent to that layer erases it. On layer 0 black would be opaque, but the app never uses layer 0.

## Notes

Implementation plan:

- Capture the active send's parameters (host, port, width, height, layer) when the loop starts, so the clear targets the exact layer/endpoint that was painted — even if the display picker changed mid-send.
- On stop (both the Stop button and `.onDisappear` from #0052), after cancelling the loop, fire a **detached** task that sends an all-black frame (`Data(count: width*height*4)` → RGB 0,0,0) to the captured host/port/layer. Detached so it survives view teardown and isn't killed by the loop's cancellation.
- Send the clear frame a few times (UDP is lossy) as a best-effort erase; ignore errors.
- Only clear on user-initiated stop, not on a network error that already ended the loop.

## Resolution notes

> 🟢 Resolved 2026-07-07 — Stopping a send (Stop button or leaving the view) now sends a final all-black frame to the painted layer, which the FT server composites as transparent, erasing the overlay immediately. Pending on-network confirmation.

## Root cause

`stopSending`/`onDisappear` only cancelled the refresh loop; nothing erased the layer, so the last frame lingered until the FT server's layer timeout (~15s).

## Fix

- Verified against the FT server compositor (`server/composite-flaschen-taschen.cc`): `if (col.is_black()) { … find first non-black below … }` — black is transparent on every layer > 0. The app only paints layers 1–15 (#0047), so an all-black frame erases exactly its overlay.
- `VariantDetailView` now captures the send parameters (host, port, width, height, scaleFactor, layer) into a `Sendable` `ActiveSend` struct when the loop starts, so the clear targets the exact endpoint/layer painted even if the display picker changed mid-send.
- A new `stopSending()` cancels the loop and then, in a **detached** task (so it survives view teardown and the loop's cancellation), sends `Data(count: width*height*4)` (all-zero RGBA → RGB 0,0,0) to the captured layer, repeated 3× as a best-effort erase against UDP loss. Both stop paths — the Stop button and `.onDisappear` (#0052) — route through it.
- The loop's `defer` also nils `activeSend`, so a network error that ends the loop doesn't leave stale params that a later `onDisappear` would use to fire a spurious clear. The clear only runs on user-initiated stop.

## Review

Self-reviewed. Clearing is best-effort and idempotent; targeting captured params (not live `selectedDisplay`) avoids clearing the wrong display if the picker changed during a send.

## Gotchas

- Black is transparent **only** on layers > 0; on layer 0 it would paint opaque black. Safe here because the app never uses layer 0 (#0047 clamps to 1–15), but a future feature that sends to layer 0 could not clear this way.
- The clear must run **detached**: the loop's Task is cancelled at stop, and `onDisappear` tears the view down, so any clear tied to either would be killed before the packet goes out.
