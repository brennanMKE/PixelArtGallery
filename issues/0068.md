# 0068 — Retire the superseded display-first navigation and stretching legacy paths

## Description

Cleanup phase for the popover flow ([#0065](0065.md)). Once the popover ([#0067](0067.md)) is the entry point, remove the now-superseded surfaces: the "Send to Display" button and `DisplaySendPickerView` (from #0064), the pushed `GalleryDetailView` if fully replaced (or reduce it to whatever the popover doesn't cover), the manual **"Custom Size…"** creation path (`VariantCreationView`) which could still stretch, and the unused `PixelationEngine.processFitting` if it remains uncalled. The goal is the end state from #0065: **just the gallery grid and the popover**, with no path that produces a distorted (stretched) variant.

## Long Description

The prior display-first work (#0064) added a pushed `DisplaySendPickerView` reached via a "Send to Display" toolbar button; #0067 replaces that entry with a popover, leaving those screens dead. The manual `VariantCreationView` ("Custom Size…") still calls the stretching `process(...)` at arbitrary dimensions and is not part of the intended end state. This issue deletes/repurposes the dead surfaces and removes any remaining path that can distort an image, verifying nothing else depended on them.

## Plan

> **This plan was written against the code as of #0064/#0057 (before #0066 and #0067 landed).** It is a **verify-then-remove checklist**, not a list of blind deletions: #0066/#0067 may have already renamed, removed, or started using some of these symbols. Before every deletion, re-grep for the symbol across `PixelArtGalleryKit/Sources`, `PixelArtGalleryKit/Tests`, and the `PixelArtGallery/` app target, and only delete when the grep shows no remaining references outside the file being deleted. Keep the build green after each step.

Reference state observed at planning time (line numbers approximate, pre-#0066/#0067):

- `DisplaySendPickerView.swift` — referenced only by `GalleryDetailView.swift:99` (`navigationDestination(isPresented: $isPickingDisplay)`) and its own `#Preview`. UI-only caller of `coordinator.createFittedVariant` (line 195).
- `GalleryDetailView.swift` — referenced only by `GalleryListView.swift:121` (`navigationDestination(for: GalleryItem.self)`). Hosts: "Send to Display" toolbar button + `isPickingDisplay` push, "Custom Size…" (`startVariantCreation()` → `coordinator.selectedItem` + `coordinator.showVariantCreation`), Rename, the variant list with Edit-Dimensions/Duplicate/Delete actions, and `navigationDestination(for: Variant.self)` → `VariantDetailView`.
- `VariantCreationView.swift` — referenced only by `GalleryListView.swift:285–296` (the `showVariantCreation` sheet, which calls `coordinator.createVariant(for:width:height:associatedDisplayId:)` at arbitrary typed dims — the stretching manual path) plus doc-comment mentions in `VariantEditDimensionsView.swift:9` and `DisplayEditorView.swift:117`.
- `PixelationEngine.processFitting(...)` (`PixelationEngine.swift:59`) — called by **nothing in Sources**; only `PixelationEngineTests.swift:117–156` (4 tests) and doc comments (`AspectFit.swift:58`, `GalleryCoordinator.swift:293`) reference it.
- `GalleryCoordinator.createFittedVariant` (`GalleryCoordinator.swift:303`, always-persist) — UI caller is only `DisplaySendPickerView`; tests at `GalleryCoordinatorTests.swift:380–456` (3 tests). #0066 reworks/replaces this.
- `GalleryCoordinator` dead UI state: `showNewVariantSheet` (uncalled anywhere already), `selectItem(_:)` / `selectVariant(_:)` (uncalled), `showVariantCreation` + `selectedItem` (only the manual-creation path plus internal nil-out in `deleteGalleryItem`), `selectedVariant` (only internal nil-out in `deleteVariant`).
- `DisplayRow.swift` — used by `DisplayRegistryView.swift:48` **and** `DisplaySendPickerView.swift:62`; the popover's dropdown/empty-state (#0067) may add a third use.
- The stretch origin: `PixelationEngine.pixelate` draws full-frame (`context.draw(image, in: CGRect(0,0,targetW,targetH))`, `PixelationEngine.swift:161`). That primitive stays — it is distortion-free whenever the caller passes aspect-fit dims. The cleanup goal is that **no caller for a display path passes non-fit dims**.
- App target (`PixelArtGallery/ContentView.swift`, `PixelArtGalleryApp.swift`) references none of the candidates.

### Checklist

**1. Delete `DisplaySendPickerView.swift`** (`PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/DisplaySendPickerView.swift`)
- Gate: `grep -rn DisplaySendPickerView` shows only the file itself + `DisplayRow.swift:5` doc comment (and `GalleryDetailView`, which is also being deleted). #0067 stops routing to it, so post-#0067 this should hold.
- Also update the `DisplayRow.swift` doc comment that names it (see step 6).

**2. Remove `GalleryDetailView.swift` entirely** (decision: **remove**, not reduce)
- Rationale: #0067's popover shows the saved-variants list and each row opens the existing `VariantDetailView`, which already has its own Duplicate (`VariantDetailView.swift:113`) and Edit-Dimensions sheet (`:135–139`); Rename/Pin/Delete for the item already exist in `GalleryListView`'s cell context menu. Nothing `GalleryDetailView` offers survives uniquely, so a reduced "saved-variant editor host" role is redundant — `VariantDetailView` *is* that editor.
- Gate: after #0067, `grep -rn GalleryDetailView` must show no live navigation to it (if #0067 left `GalleryListView.swift:121`'s `navigationDestination(for: GalleryItem.self)` in place as dead routing, remove that too; the popover must be the only cell-tap response). Confirm the popover's saved-variant rows reach `VariantDetailView` some other way (per #0067) before deleting — if #0067 routed rows *through* `GalleryDetailView`, stop and reconcile with the reviewer instead of deleting.

**3. Remove the manual "Custom Size…" path** (decision: **remove**; it is the last deliberate stretch path)
- Delete `VariantCreationView.swift`.
- Remove the `showVariantCreation` sheet block from `GalleryListView.swift` (lines ~285–296) — this is the only UI call of `coordinator.createVariant` at arbitrary dims.
- `GalleryDetailView.startVariantCreation()` goes away with step 2.
- Gate: `grep -rn "VariantCreationView\|showVariantCreation\|startVariantCreation"` returns only doc comments, then fix those (step 6).
- **Owner note:** if the product owner later wants arbitrary custom sizing back, it must be re-added as an aspect-fit (letterbox) creation, never a stretch — file that as a follow-on issue rather than keeping this view. Do not retain it "just in case".

**4. `processFitting` disposition — conditional on what #0066 actually did**
- Check first: `grep -rn processFitting PixelArtGalleryKit/Sources`.
  - **If #0066's `fittedPreview(for:display:)` calls it** → keep `processFitting` and its 4 tests; the #0062 dead-code concern is retired. Done.
  - **If it is still test-only** (likely — #0066's plan leans toward `AspectFit.fit` + `process(...)` at fit dims because the dedup/cache key needs the fit dims *before* pixelation) → **remove** `processFitting` from `PixelationEngine.swift` and delete its test block (`PixelationEngineTests.swift`, the `// MARK: - processFitting (#0062)` section: `processFittingProducesFitSizedGrid`, `processFittingHonorsExifOrientation`, `processFittingRejectsInvalidDisplayDimensions`, ~lines 117–156). Also fix the stale doc references in `AspectFit.swift:58` and the `createFittedVariant` doc comment (if that method still exists). Do not leave tested-but-dead code.

**5. `GalleryCoordinator` reconciliation**
- `createFittedVariant(for:display:)`: after #0066's transient rework and step 1's deletion of its only UI caller, grep for remaining callers. If #0066 replaced it (popover uses `fittedPreview`/`saveVariant`), remove the method and its 3 always-persist tests (`GalleryCoordinatorTests.swift:380–456`: `createFittedVariantUsesFitDimensionsNotDisplayDimensions`, `…DedupsForSameItemAndDisplay`, `…CreatesSeparateVariantPerDisplay`) **unless** #0066 already repurposed them for the save path — check before deleting; the fit-dims-not-display-dims regression guard should survive in some form (against `fittedPreview`/`saveVariant`).
- `createVariant(for:width:height:associatedDisplayId:)`: **keep only if still called** (e.g. by #0066's `saveVariant` or a surviving `createFittedVariant`); its tests (`createVariantRecordsAssociatedDisplayId`, etc.) stay with it. If, post-cleanup, nothing calls it, remove it and its tests too — but expect it to survive as the persistence primitive.
- Dead UI state to delete (each gated on its own grep): `showNewVariantSheet` (already uncalled today), `showVariantCreation`, `selectItem(_:)`, `selectVariant(_:)`. `selectedItem`/`selectedVariant` become internal-only (nil-out in the delete methods) — either keep them as internal bookkeeping or drop them if the delete-method nil-outs are their only remaining use and nothing observes them; prefer dropping what nothing reads.
- `updateVariantDimensions` / `duplicateVariant` / `VariantEditDimensionsView`: **keep** — still used by `VariantDetailView` (the saved-variant editor the popover routes to). See the invariant note below about the edit-dims stretch exposure.

**6. `DisplayRow.swift`: keep** — `DisplayRegistryView.swift:48` uses it regardless of this cleanup (and the popover dropdown may too). Update its doc comment (line ~5) to stop citing `DisplaySendPickerView`. Sweep all stale doc comments naming deleted types: `VariantEditDimensionsView.swift:9` (`VariantCreationView`), `DisplayEditorView.swift:117` (`VariantCreationView`), `AspectFit.swift:58` (`processFitting`, if removed), `DisplayRow.swift:5`.

**7. Assert the no-stretch invariant**
- Grep every caller of `PixelationEngine.process(` and `pixelate(` in Sources. After this cleanup the complete set should be: #0066's fitted-preview path (fit dims from `AspectFit.fit` — aspect-correct by construction), possibly `createVariant` invoked only from a fit-dims caller, and `updateVariantDimensions` (user-typed dims on a *saved* variant via `VariantDetailView`'s Edit Dimensions sheet).
- The full-frame `context.draw` at `PixelationEngine.swift:161` must therefore be reachable **for display sends** only via fit dims. Confirm the popover's Send uses the fitted grid + centering offset (from #0066/#0067), not display dims.
- **Residual exposure to flag, not silently fix:** `VariantDetailView` → `VariantEditDimensionsView` → `updateVariantDimensions` still lets a user retype a saved variant to arbitrary (distorting) dims. #0067 explicitly keeps `VariantDetailView` as the saved-variant editor, so removing this here would be scope creep. Note it in `## Gotchas` and ask the owner: either accept it as an explicit user edit, or file a follow-on to make the edit sheet aspect-locked/aspect-fit. It must be recorded, not ignored.

**8. Verification (mandatory, both platforms + running app)**
- `cd PixelArtGalleryKit && swift test` — full suite passes, and the deleted types' tests are gone (no compile references to removed symbols); confirm tests actually ran (non-zero count).
- `xcodebuild -project PixelArtGallery.xcodeproj -scheme PixelArtGallery -destination 'platform=macOS' build` and `-destination 'platform=iOS Simulator,name=iPhone 17 Pro' build` — both green, proving nothing (including the app target) depended on the deleted code.
- **Run the app** (macOS at minimum, iOS simulator if feasible): tap a gallery cell → the popover opens (no pushed detail screen appears anywhere); confirm there is no "Send to Display" button, no display-picker screen, and no "Custom Size…" anywhere; confirm a saved-variant row still opens `VariantDetailView` and Send still works from the popover. The old entry points being *gone in the running app* is the acceptance test — a green build alone shipped the wrong UI once already (#0064).
- Final grep sweep: `grep -rn "DisplaySendPickerView\|GalleryDetailView\|VariantCreationView\|processFitting\|showVariantCreation\|startVariantCreation\|showNewVariantSheet"` across the repo returns nothing in Sources/Tests (issue markdown mentions are fine).
