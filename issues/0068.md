## Description

Cleanup phase for the popover flow ([#0065](0065.md)). Once the popover ([#0067](0067.md)) is the entry point, remove the now-superseded surfaces: the "Send to Display" button and `DisplaySendPickerView` (from #0064), the pushed `GalleryDetailView` if fully replaced (or reduce it to whatever the popover doesn't cover), the manual **"Custom Size…"** creation path (`VariantCreationView`) which could still stretch, and the unused `PixelationEngine.processFitting` if it remains uncalled. The goal is the end state from #0065: **just the gallery grid and the popover**, with no path that produces a distorted (stretched) variant.

## Long Description

The prior display-first work (#0064) added a pushed `DisplaySendPickerView` reached via a "Send to Display" toolbar button; #0067 replaces that entry with a popover, leaving those screens dead. The manual `VariantCreationView` ("Custom Size…") still calls the stretching `process(...)` at arbitrary dimensions and is not part of the intended end state. This issue deletes/repurposes the dead surfaces and removes any remaining path that can distort an image, verifying nothing else depended on them.

## Root cause

Not a bug fix — a scheduled cleanup. The popover flow (#0067) already superseded the display-first navigation (#0064) and the pushed item-detail screen, but the superseded surfaces (and one lingering manual stretch path) were left in the tree: `DisplaySendPickerView` and `GalleryDetailView` had no live navigation into them, `VariantCreationView`'s "Custom Size…" sheet was the last path that could pixelate to arbitrary, non-aspect-fit dimensions, and `GalleryCoordinator` carried UI-selection state (`selectedItem`/`selectedVariant`, `selectItem(_:)`/`selectVariant(_:)`, `showNewVariantSheet`/`showVariantCreation`) and a persistence wrapper (`createFittedVariant`) that only those dead surfaces used.

## Fix

Re-validated the plan's checklist against the current (post-#0066/#0067) code by grepping every symbol before touching it, per the constraint that the plan predates #0066/#0067:

1. **Deleted `DisplaySendPickerView.swift`.** `grep -rn DisplaySendPickerView` showed its only caller was `GalleryDetailView.swift:99`, which is also deleted (plus its own `#Preview` and a doc-comment mention in `DisplayRow.swift`, fixed below).
2. **Deleted `GalleryDetailView.swift` entirely.** `grep -rn GalleryDetailView` showed no live navigation to it — `GalleryListView` already carried only a doc comment noting the retirement (#0067 had already removed the `navigationDestination(for: GalleryItem.self)` route). Confirmed the popover's saved-variant rows reach `VariantDetailView` directly: `GallerySendPopoverView.swift` has its own `.navigationDestination(for: Variant.self) { VariantDetailView(...) }` (line 69), not routed through `GalleryDetailView`. Its Rename/Delete/Pin actions already live in `GalleryListView`'s cell context menu (`togglePin`/rename alert/delete confirmation, added in #0067); its variant Edit-Dimensions/Duplicate/Delete already live on `VariantDetailView`. Nothing was uniquely lost.
3. **Removed the manual "Custom Size…" path.** Deleted `VariantCreationView.swift`; removed the `.sheet(isPresented: $coordinator.showVariantCreation)` block from `GalleryListView.swift` (the only UI caller of `createVariant` at arbitrary typed dims). `GalleryDetailView.startVariantCreation()` went with step 2. Final grep for `VariantCreationView|showVariantCreation|startVariantCreation` returns only doc comments, now fixed (see below).
4. **`processFitting`: kept.** `grep -rn processFitting PixelArtGalleryKit/Sources` confirms `GalleryCoordinator.fittedPreview(for:display:)` (line 342) calls `pixelationEngine.processFitting(...)` directly — it has a live production caller, so the #0062 dead-code concern is retired. Its 4 tests in `PixelationEngineTests.swift` are untouched.
5. **`GalleryCoordinator` reconciliation:**
   - Removed `createFittedVariant(for:display:)` — its doc comment already called it a "legacy always-persist path... kept as a thin wrapper so `DisplaySendPickerView`, its sole remaining production call site, keeps compiling... until that view (and this wrapper) are retired in #0068." After deleting `DisplaySendPickerView`, `grep -rn createFittedVariant` showed only its 3 dedicated tests in `GalleryCoordinatorTests.swift` (`createFittedVariantUsesFitDimensionsNotDisplayDimensions`, `createFittedVariantDedupsForSameItemAndDisplay`, `createFittedVariantCreatesSeparateVariantPerDisplay`) — removed the method and all 3. The regression guards they covered (fit-dims-not-display-dims, and dedup) already live on `fittedPreview`/`saveVariant`: `fittedPreviewReturnsFitDimensionsAndDoesNotPersist` and `saveVariantPersistsPreviewExactlyOnceAndDedups`, both still passing.
   - `createVariant(for:width:height:associatedDisplayId:)`: kept. It now has zero production callers (its only caller, `VariantCreationView`'s sheet, was just removed), but it's still directly tested and used as the fixture-creation primitive by `duplicateVariantCopiesDataAndIncreasesCount`, `updateVariantDimensionsRegeneratesPixelData`, and `deleteGalleryItemRemovesItemAndCascadesVariants`. Removing it was out of this issue's assigned scope (not listed among the symbols to reconcile) and would have required rewriting several unrelated tests' fixtures — left as-is per "expect it to survive as the persistence primitive." See Gotchas.
   - Removed dead UI state, each gated on its own grep showing zero remaining references once the above were gone: `showNewVariantSheet` (already uncalled before this issue), `showVariantCreation`, `selectItem(_:)`, `selectVariant(_:)`, `selectedItem`, `selectedVariant` (including their nil-out remnants in `deleteGalleryItem`/`deleteVariant`, which became dead once nothing set those properties anymore). `GalleryListView`'s own local `@State private var selectedItem` (the popover's selection, unrelated to the coordinator's now-removed property) was untouched.
   - `updateVariantDimensions`/`duplicateVariant`/`VariantEditDimensionsView`: kept — still the saved-variant editor path via `VariantDetailView`.
6. **`DisplayRow.swift`: kept** — `DisplayRegistryView.swift:48` is its only remaining user (`DisplaySendPickerView.swift:62`, its other caller, is deleted). Fixed its doc comment to stop naming `DisplaySendPickerView`. Also fixed stale doc mentions of the deleted `VariantCreationView` in `VariantEditDimensionsView.swift:9` and `DisplayEditorView.swift:117`.
7. **No-stretch invariant asserted.** `grep -rn "context.draw" PixelArtGalleryKit/Sources` shows the one full-frame draw is in `PixelationEngine.pixelate` (line 161), reached only via `pixelate(...)` calls inside `PixelationEngine.process`/`processFitting` themselves. `grep` for `.process(` callers shows exactly two: `GalleryCoordinator.createVariant` (line 258, now with zero production callers per step 5) and `GalleryCoordinator.updateVariantDimensions` (line 501, the saved-variant Edit Dimensions path). The popover's Send/Save path (`fittedPreview`/`saveVariant`) uses `processFitting`, which is fit-dims by construction. See Gotchas for the one residual stretch-capable path that's out of this issue's scope to remove.

## Gotchas

- **`VariantDetailView` → `VariantEditDimensionsView` → `updateVariantDimensions` remains stretch-capable.** A user can retype a saved variant's dimensions to arbitrary (non-aspect-fit) values on the Edit Dimensions sheet. This is deliberate per #0067 (kept as the saved-variant editor) and out of this issue's scope to change — flagging per the plan rather than silently fixing. Owner decision needed: accept as an explicit user edit, or file a follow-on to make the edit sheet aspect-locked/aspect-fit.
- **`GalleryCoordinator.createVariant(for:width:height:associatedDisplayId:)` now has zero production callers.** It was previously reachable only through the just-deleted `VariantCreationView` manual sheet. It's kept (per the plan's "expect it to survive as the persistence primitive" guidance) because it's still directly unit-tested and used as a fixture by `duplicateVariantCopiesDataAndIncreasesCount`, `updateVariantDimensionsRegeneratesPixelData`, and `deleteGalleryItemRemovesItemAndCascadesVariants`. It remains stretch-capable (arbitrary width/height, no aspect-fit enforcement) — currently unreachable from any UI so the no-stretch invariant holds today, but if a future feature ever wires a UI caller back to it, that caller must pass fit dims, not arbitrary ones. Not removed here since it wasn't in this issue's assigned symbol list and removing it would have meant rewriting several unrelated tests' fixtures.
- Live app-run confirmation of the removed entry points being gone (no "Send to Display" button/screen, no "Custom Size…") was not performed in this pass — recommended as a quick manual check before closing.
