## Description

Sparkle stays dormant until the built app's Info.plist carries the appcast URL and the EdDSA public key. The target uses `GENERATE_INFOPLIST_FILE = YES`, and Xcode's generated plist doesn't accept arbitrary keys — so the macOS build needs a real Info.plist file (or equivalent) carrying the Sparkle keys, without disturbing the iOS build's generated plist.

## Acceptance criteria

- Built `PixelArtGallery.app/Contents/Info.plist` (macOS) contains:
  - `SUFeedURL` — the appcast URL on the site from #0040 (e.g. `https://<site-domain>/appcast.xml`).
  - `SUPublicEDKey` — the base64 EdDSA public key matching the private key in the developer's keychain.
  - `SUEnableAutomaticChecks` = true.
- EdDSA key pair generated with Sparkle's `generate_keys --account PixelArtGallery` (private key stays in the keychain / an offline backup, never in the repo; the public key is what's embedded).
- All existing Info.plist values (bundle ID, versions, display name, copyright, photo-library usage description on iOS, etc.) are preserved; iOS keeps its generated plist.
- "Check for Updates…" (#0038) is enabled on a fresh macOS build.
- Both platform builds succeed; package tests pass.

## Notes

- Recommended shape: create `PixelArtGallery/Info-macOS.plist` with the synthesized keys plus the Sparkle ones, then set `INFOPLIST_FILE[sdk=macos*]` and `GENERATE_INFOPLIST_FILE[sdk=macos*] = NO` in `Configuration/macOS.xcconfig` — remembering the #0023 lesson that any target-level pbxproj `GENERATE_INFOPLIST_FILE` line would override the xcconfig.
- Verify by dumping the built bundle's plist (`plutil -p`) on both platforms.
- Depends on #0038 (controller/menu) and on the site domain decided in #0040; key generation depends on the Sparkle artifacts from the SPM checkout.

## Root cause

Feature gap: the target used `GENERATE_INFOPLIST_FILE = YES` (set at the target level in `project.pbxproj`, overriding any xcconfig value per the #0023 precedence lesson), and Xcode's generated Info.plist cannot carry arbitrary keys — so there was nowhere to put `SUFeedURL`, `SUPublicEDKey`, and `SUEnableAutomaticChecks`, and #0038's `UpdaterController` stayed dormant (`isConfigured == false`, menu item disabled).

## Fix

- **EdDSA key pair** generated with Sparkle 2.9.4's `generate_keys --account PixelArtGallery` (from `PixelArtGalleryKit/.build/artifacts/sparkle/Sparkle/bin/`). No key existed beforehand (`generate_keys -p` reported none). The **private key lives in the login keychain** (account `PixelArtGallery`) and is **not** in the repo; the user should export an offline backup (`generate_keys -x <file>`) — to be documented in SPARKLE.md (#0044). The embedded public key is the **real** one: `tQZNcNn4+PYvilca5jBXoPx6XHxM6mCYrwWh3pzl+H0=`.
- **New `PixelArtGallery/Info-macOS.plist`** replicating the synthesized macOS keys via build-setting substitution (`$(PRODUCT_BUNDLE_IDENTIFIER)`, `$(MARKETING_VERSION)`, `$(CURRENT_PROJECT_VERSION)`, `$(EXECUTABLE_NAME)`, `$(PRODUCT_NAME)`, `$(DEVELOPMENT_LANGUAGE)`, `$(MACOSX_DEPLOYMENT_TARGET)`, `NSPrincipalClass` = `NSApplication`) plus `SUFeedURL` = `https://pixelartgallery.sstools.co/appcast.xml`, `SUPublicEDKey`, and `SUEnableAutomaticChecks` = true. `CFBundleDisplayName` is `$(INFOPLIST_KEY_CFBundleDisplayName)` so `Build.xcconfig` stays the source of truth.
- **`Configuration/macOS.xcconfig`**: `GENERATE_INFOPLIST_FILE[sdk=macos*] = NO`, `INFOPLIST_FILE[sdk=macos*] = PixelArtGallery/Info-macOS.plist`.
- **`Configuration/iOS.xcconfig`**: `GENERATE_INFOPLIST_FILE[sdk=iphone*] = YES` (needed because the target-level line was removed and the setting defaults to NO).
- **`project.pbxproj`**: removed the target-level `GENERATE_INFOPLIST_FILE = YES;` lines from Debug and Release so the per-SDK xcconfig values actually apply (#0023 lesson).

## Gotchas

- **`INFOPLIST_KEY_*` settings are NOT merged into a file-based Info.plist.** With `GENERATE_INFOPLIST_FILE = NO`, `INFOPLIST_KEY_CFBundleDisplayName` (and the copyright/photo-library keys) were silently dropped from the built macOS plist — verified empirically. Any needed value must be in the file; `$(INFOPLIST_KEY_…)` substitution inside the file keeps the xcconfig as the single source of truth. Asset-catalog-driven keys (`CFBundleIconFile`/`CFBundleIconName`, `NSAccentColorName`) are still injected by plist post-processing.
- **`GENERATE_INFOPLIST_FILE` defaults to NO**, so after deleting the target-level `YES` lines, iOS needed an explicit `GENERATE_INFOPLIST_FILE[sdk=iphone*] = YES` in `iOS.xcconfig`.
- The private EdDSA key exists only in this Mac's login keychain (account `PixelArtGallery`). Losing it means shipped apps can never verify another update — back it up offline (`generate_keys -x`); SPARKLE.md (#0044) should walk through this.
- `NSPhotoLibraryAddUsageDescription` and `UISupportedInterfaceOrientations` no longer appear in the macOS plist (they were iOS-oriented spillover from shared `INFOPLIST_KEY_*` settings); iOS still gets both via its generated plist.
