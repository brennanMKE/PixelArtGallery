## Description

Pixelating a photo that carries an EXIF orientation tag (e.g. a portrait shot from a phone camera) produces a rotated pixel grid. The decode path reads the raw pixel buffer and never applies the image's orientation metadata, so any image whose stored pixels differ from its displayed orientation comes out rotated (and, for 90°/270° cases, with width/height swapped).

## Steps to reproduce

1. Import a photo taken in portrait orientation (one with a non-default EXIF orientation).
2. Pixelate it into a variant.
3. Observe the variant is rotated relative to how the source photo displays.

## Expected behavior

The pixelated variant matches the upright, as-displayed orientation of the source image — EXIF orientation is applied before downsampling.

## Actual behavior

The variant is rotated (and mis-proportioned for 90°/270° orientations) because the raw pixel buffer is used as-is.

## Notes

Root cause in `PixelArtGalleryKit/Sources/PixelArtGalleryKit/ImageProcessing/PixelationEngine.swift`:

- `process(imageData:...)` calls `CGImageSourceCreateImageAtIndex(source, 0, nil)` (~line 42), which returns the raw decoded pixels and **does not** apply `kCGImagePropertyOrientation`.
- The orientation tag is available via `CGImageSourceCopyPropertiesAtIndex(source, 0, nil)`; the fix is to read it and apply the corresponding transform to produce an upright `CGImage` before calling `pixelate`.

Cross-platform note: the package is UIKit-free (used on macOS too), so the fix should apply the orientation with Core Graphics (a `CGContext` + `CGAffineTransform` derived from the orientation value), not `UIImage`. All 8 EXIF orientation values should be handled; orientation 1 (up) is the no-op fast path.

## Resolution notes

> 🟢 Resolved 2026-07-06 — Pixelation now decodes through ImageIO's orientation-aware thumbnail path, so photos with any EXIF orientation are pixelated upright.

## Root cause

`PixelationEngine.process(imageData:)` decoded the source with `CGImageSourceCreateImageAtIndex(source, 0, nil)`, which returns the raw stored pixel buffer and does **not** apply the `kCGImagePropertyOrientation` EXIF tag. Any image whose stored pixels differ from its displayed orientation (every phone-camera portrait shot) was therefore downsampled in its stored orientation — rotated, and for 90°/270° cases mis-proportioned.

## Fix

Added `PixelationEngine.uprightImage(from:)`, which decodes the primary image via `CGImageSourceCreateThumbnailAtIndex` with `kCGImageSourceCreateThumbnailWithTransform: true` — the ImageIO path that bakes the EXIF orientation into the returned pixels. `kCGImageSourceThumbnailMaxPixelSize` is pinned to the source's largest dimension so no resolution is lost before downsampling (the cap never upscales). `process` now uses this upright image, falling back to the raw decode only if ImageIO returns nothing. The approach is UIKit-free and works identically on iOS and macOS, avoiding hand-rolled `CGAffineTransform` orientation math.

## Review

Self-reviewed (single session). The regression test encodes identical pixels under orientation 1 (up) and 3 (down/180°) and asserts the processed grids are 180° rotations of each other, with a guard assertion that fails if orientation is ignored — so it genuinely exercises the fix rather than just compiling.

## Gotchas

- `CGImageSourceCreateImageAtIndex` never applies orientation — this trap recurs anywhere raw pixels are read from an image source. The thumbnail-with-transform path is the fix; the other option (`CIImage` respects orientation, or a manual `CGAffineTransform`) is heavier.
- Orientation only survives in formats that carry the tag. The test uses lossless **TIFF** (not PNG, which doesn't reliably round-trip the EXIF orientation tag through ImageIO) so the encoded orientation is actually present on read-back.
