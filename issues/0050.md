# 0050 — Send button should stay active while sending and let the user tap to stop

|---|---|

## Description

While a send is in flight the Send button is disabled, so the user can't cancel a send that is slow or stuck (e.g. an unreachable display timing out). The button should stay active during sending, switch to a "Stop"/"Cancel" affordance, and tapping it again should cancel the in-flight send.

## Steps to reproduce

1. Open a variant's detail view and pick a display.
2. Tap **Send Now**.
3. While it shows "Sending…", observe the button is disabled — there's no way to stop it. If the display is unreachable, the user must wait for the connection timeout.

## Expected behavior

- The button remains tappable while a send is in progress.
- During a send it reads as a stop action (e.g. "Stop" / "Cancel", with a stop icon).
- Tapping it while sending cancels the in-flight send promptly and returns the UI to the idle "Send Now" state (no success/error flash for a user-initiated cancel, or a neutral "Send cancelled" message).

## Actual behavior

The button is `.disabled(isSending || selectedDisplay == nil)` (`UI/VariantDetailView.swift`), so it is inert for the whole send. The send runs in a detached `Task` in `handleSendToDisplay` whose handle isn't retained, so there is nothing to cancel.

## Notes

Implementation sketch:

- Retain the send `Task` in `@State` (e.g. `sendTask`) so it can be cancelled; toggle behavior in `handleSendToDisplay` — if a send is active, call `sendTask?.cancel()` instead of starting a new one.
- Keep the button enabled while `isSending` (still disable when `selectedDisplay == nil`); swap the label/icon on `isSending`.
- `FTDisplayClient.send` must honor cancellation: check `Task.isCancelled` / use cancellation-aware awaits around the `NWConnection` setup and send, and cancel the connection on cancel. Map a cancelled send to a distinct outcome (not `sendFailed`) so the UI can show a neutral "cancelled" state rather than an error.
- The layer stepper is currently disabled during send (#0047); decide whether it stays disabled while a cancellable send is in flight (likely yes).

## Resolution notes

> 🟢 Resolved 2026-07-06 — The Send button stays active during a send, turns into a red "Stop Sending" action, and tapping it cancels the in-flight send, which now reports a distinct `.cancelled` outcome shown as a neutral banner.

## Root cause

Feature gap. The button was `.disabled(isSending || selectedDisplay == nil)` and the send ran in a fire-and-forget `Task` whose handle wasn't retained, so there was nothing to cancel; `FTDisplayClient.send` also had no cancellation checkpoints, so even a cancelled task would run the connect/send to completion (or to the 5s timeout).

## Fix

- `FTDisplayClient`: added `FTDisplayError.cancelled` and made `send` cancellation-aware — cancellation checkpoints (`throwIfCancelled`) before encoding and before connecting, and a `withTaskCancellationHandler` around the connect/send region that cancels the `NWConnection` on cancel. Any error thrown while `Task.isCancelled` is translated to `.cancelled`, so a user stop is never mislabelled as a network failure.
- `VariantDetailView`: the send `Task` is retained in `@State var sendTask`. The button stays enabled while sending (only disabled when no display is selected), switches to a red "Stop Sending" label/icon, and tapping it while sending calls `sendTask?.cancel()`. The catch handles `.cancelled` / `CancellationError` as a neutral outcome.
- Added a neutral `.info` kind to `StatusBanner` and an `infoMessage`/`flashInfo` path so a cancel shows "Send cancelled" as an info banner rather than a green success or red error.

## Review

Self-reviewed (single session). The cancel path is translated in exactly one place (`send`'s catch, gated on `Task.isCancelled`) so a genuine `connectionFailed`/`timedOut` is never masked as a cancel. The button's "never 0"/disabled logic for the layer stepper (#0047) is unchanged.

## Gotchas

- UDP `NWConnection` has no handshake, so a send to a *routable* bogus address can go `.ready` and "succeed" immediately — that's why the cancellation test targets an unresolvable `.invalid` host (stays pending, never `.ready`/`.failed`) to deterministically hold the connection open until the cancel lands.
- The cancel→`.cancelled` translation is gated on `Task.isCancelled` *after* the awaits return, because the `NWConnection` state handler fires on a dispatch queue where `Task.isCancelled` isn't the task's; checking it back in the async context is what makes the translation reliable.
