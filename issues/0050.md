# 0050 — Send button should stay active while sending and let the user tap to stop

| | |
|---|---|
| **Status** | in-progress |
| **Module** | Networking / UI |
| **Platform** | All |
| **First seen** | 2026-07-06 |
| **Branch** | issue/0050 |

## Description

Tapping Send should begin an **ongoing** send that keeps pushing the image to the display and stays active until the user taps **Stop** — it should not send once and stop on its own. The button stays active while sending, reads as "Stop Sending", and tapping it ends the loop.

> ⚠️ Corrected 2026-07-06 after user testing: the first implementation treated this as cancelling a single UDP send. Because a one-shot send completes in milliseconds, the "Stop" state was never observable and it "stopped automatically" — not what was asked. Reworked to a continuous refresh loop (below).

## Steps to reproduce

1. Open a variant's detail view and pick a display.
2. Tap **Send Now**.
3. While it shows "Sending…", observe the button is disabled — there's no way to stop it. If the display is unreachable, the user must wait for the connection timeout.

## Expected behavior

- The button remains tappable while a send is in progress.
- During a send it reads as a stop action (e.g. "Stop" / "Cancel", with a stop icon).
- Tapping it while sending cancels the in-flight send promptly and returns the UI to the idle "Send Now" state (no success/error flash for a user-initiated cancel, or a neutral "Send cancelled" message).

## Actual behavior

The button is `.disabled(isSending || selectedDisplay == nil)` (`UI/VariantDetailView.swift`), so it is inert for the whole send. The send runs in a detached `Task` in `handleSendToDisplay` whose handle isn't retained, so there is nothing to cancel.

## Notes

Implementation sketch:

- Retain the send `Task` in `@State` (e.g. `sendTask`) so it can be cancelled; toggle behavior in `handleSendToDisplay` — if a send is active, call `sendTask?.cancel()` instead of starting a new one.
- Keep the button enabled while `isSending` (still disable when `selectedDisplay == nil`); swap the label/icon on `isSending`.
- `FTDisplayClient.send` must honor cancellation: check `Task.isCancelled` / use cancellation-aware awaits around the `NWConnection` setup and send, and cancel the connection on cancel. Map a cancelled send to a distinct outcome (not `sendFailed`) so the UI can show a neutral "cancelled" state rather than an error.
- The layer stepper is currently disabled during send (#0047); decide whether it stays disabled while a cancellable send is in flight (likely yes).

## Relation

- Pairs with: [#0047](0047.md) — same Send section; layer stepper enable/disable interacts with this change.

## Behavior (awaiting user confirmation)

Tap **Send Now** → a refresh loop pushes the frame to the display every ~2s and the button becomes a red **Stop Sending**; a persistent "Sending to X… Tap Stop Sending to end." banner shows while it runs. Tap **Stop Sending** → the loop ends and a neutral "Stopped sending to X" banner appears. A genuine network error stops the loop and shows the error.

## Approach

- `FTDisplayClient` (from the first pass, retained): `FTDisplayError.cancelled` plus cancellation checkpoints and a `withTaskCancellationHandler` around connect/send, so a single `send` aborts promptly and reports `.cancelled` rather than a network failure. This is what lets the loop stop mid-packet.
- `VariantDetailView`: the send `Task` is retained in `@State var sendTask` and now runs `while !Task.isCancelled { try await client.send(...); try await Task.sleep(refreshInterval) }`. The button stays enabled while sending (disabled only when no display is selected) and toggles to "Stop Sending"; tapping it cancels the task, which ends both the in-flight packet and the sleep. Refresh interval is 2s (well inside the FT server's ~15s layer timeout so the image stays up).
- Neutral outcomes use the added `StatusBanner.Kind.info` + `flashInfo`; a persistent info banner (driven by `isSending`) communicates the ongoing state.

## Gotchas

- A one-shot UDP send is effectively instantaneous, so "stay active while sending / tap to stop" only makes sense as a *continuous* operation — hence the refresh loop. This is the correction from the first pass.
- UDP `NWConnection` has no handshake, so a send to a routable bogus address can go `.ready` and "succeed" immediately — the cancellation test targets an unresolvable `.invalid` host so the connection stays pending until the cancel lands.
