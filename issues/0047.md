## Description

When sending a variant to a Flaschen Taschen display via the Send button, there is no way to choose the layer (the FT z-offset). Every send is hard-coded to layer 0, so images can't be placed on a non-default layer of a display like `ft.noise` (10.21.1.201). Settings should hold a per-display **default layer** (alongside host and port) that defaults to **5**, and the Send button should use that default while letting the user adjust it. The layer must stay in **1…15 and never be 0**.

## Steps to reproduce

1. Open a variant's detail view.
2. Tap Send and pick a display (e.g. `ft.noise` / 10.21.1.201).
3. Observe there is no field or picker to choose which layer the image paints on.

## Expected behavior

- Settings for the default display shows a **Layer** control in addition to host and port, defaulting to **5**, constrained to **1…15** (never 0).
- The Send button seeds its layer from the selected display's default layer and lets the user increase it up to **15** and down to **1** before sending — never 0.
- The chosen layer is threaded into `FTDisplayClient.send`'s `offset.z` so the image lands on that layer.

## Actual behavior

The layer is always 0. `FTDisplayClient.send` accepts an `offset: (x, y, z)` (z is the layer) and encodes it into the FT footer, but the UI never supplies it, so it falls back to the `(0, 0, 0)` default on every send. Settings has no layer field.

## Notes

The wire plumbing already exists — this is a model + UI change, not a protocol change:

- `FTDisplayClient.send(...)` in `Networking/FTDisplayClient.swift` takes `offset: (x: Int, y: Int, z: Int) = (0, 0, 0)`; `makePacket` writes the z value into the FT footer (`<0x00><x>\n<y>\n<z>\n`), which the FT server reads as the paint layer.
- The Send call site in `UI/VariantDetailView.swift` omits `offset`, so it always uses layer 0.

Plan: add a persisted `layer` property to `FlaschenTaschenDisplay` (default 5, clamped to 1…15), surface it as a `Stepper` in `SettingsView`, seed the Send section's layer from the selected display and expose a 1…15 `Stepper` there, and pass `offset: (0, 0, layer)` to `send`. The 1…15 stepper range structurally enforces "never 0". The separate report that `ft.noise` isn't showing up at all is a discovery/reachability concern, not part of this ticket.

## Resolution notes

> 🟢 Resolved 2026-07-06 — Displays now carry a default paint layer (default 5, range 1…15, never 0). Settings edits it; the Send button seeds from it and lets the user adjust it per send; the chosen layer is written to the FT packet.

## Root cause

Feature gap, not a regression. The FT wire path already supported a z-offset (`FTDisplayClient.send(offset:)` → `makePacket` footer), but nothing in the model or UI ever set it, so every send used the `(0, 0, 0)` default — layer 0.

## Fix

- `FlaschenTaschenDisplay` gained a persisted `layer: Int` (inline default `5` so SwiftData lightweight-migrates existing stores), plus `layerRange` (1…15), `defaultLayer` (5), and a `clampedLayer(_:)` helper. The init clamps its `layer` argument so an out-of-range value (including 0) can never be stored.
- `SettingsView` adds a "Default Layer" `Stepper` bound to `FlaschenTaschenDisplay.layerRange`, loaded from and saved to the default display record.
- `VariantDetailView`'s Send section adds a "Layer" `Stepper` (1…15) that seeds from the selected display's default layer (and reseeds when the selection changes), and passes `offset: (x: 0, y: 0, z: layer)` — clamped once more defensively — into `send`.

The 1…15 `Stepper` ranges make layer 0 structurally unreachable in the UI; `clampedLayer` is the belt-and-suspenders backstop for stored/legacy values.

## Review

Self-reviewed (single session). Layer validation lives in pure static helpers on the model so it is unit-tested without SwiftData; the UI stepper ranges are the primary "never 0" guarantee, with `clampedLayer` backing the persistence and send paths.

## Gotchas

- Adding a non-optional SwiftData attribute requires an **inline** default on the property (`var layer: Int = …`) for lightweight migration of existing stores; a default only in `init` is not enough.
- The package sets `.defaultIsolation(MainActor.self)`, so tests that construct `@Model` instances or touch its static members must be `@MainActor` (the new suite is annotated accordingly).
