# 0047 — Send button has no way to set the display layer (FT z-offset)

| | |
|---|---|
| **Status** | open |
| **Module** | Networking / UI |
| **Platform** | All |
| **First seen** | 2026-07-06 |

## Description

When sending a variant to a Flaschen Taschen display via the Send button, there is no way to choose the layer (the FT z-offset). Every send is hard-coded to layer 0, so images can't be stacked or placed on a non-default layer of a display like `ft.noise` (10.21.1.201). We need a control to set the layer as part of the send flow.

## Steps to reproduce

1. Open a variant's detail view.
2. Tap Send and pick a display (e.g. `ft.noise` / 10.21.1.201).
3. Observe there is no field or picker to choose which layer the image paints on.

## Expected behavior

The send flow should let the user set the layer (z) before sending — e.g. a stepper or field defaulting to 0 — and pass it through to the FT packet so the image lands on the chosen layer.

## Actual behavior

The layer is always 0. `FTDisplayClient.send` accepts an `offset: (x, y, z)` (z is the layer) and encodes it into the FT footer, but the UI never supplies it, so it falls back to the `(0, 0, 0)` default on every send.

## Notes

The plumbing already exists — this is a UI + call-site gap, not a protocol change:

- `FTDisplayClient.send(...)` at `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Networking/FTDisplayClient.swift:90` takes `offset: (x: Int, y: Int, z: Int) = (0, 0, 0)`.
- `makePacket` (same file, ~line 150) writes the z value into the FT footer (`<0x00><x>\n<y>\n<z>\n`), which the FT server reads as the paint layer.
- The Send call site at `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/VariantDetailView.swift:219` omits `offset`, so it always uses layer 0.

Scope suggestion: expose a layer control (default 0) in the send UI, thread it to the `send(...)` call, and consider persisting a per-display default layer. The separate report that `ft.noise` isn't showing up at all may be a discovery issue rather than a layer issue — worth confirming reachability independently of this ticket.
