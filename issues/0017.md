## Description

After importing an image, it does not show at all on macOS or iOS — the gallery row and the item detail screen display a generic `photo.fill` SF Symbol instead of the imported picture. Import itself works (bytes are saved and dimensions are read), so the original image is on disk but never loaded into the UI.

## Steps to reproduce

1. Launch the app (Mac or iPhone Simulator).
2. Import an image.
3. Observe the gallery list row and open the item's detail view.

## Expected behavior

The gallery row shows a thumbnail of the imported image, and the detail view's "Original Image" section shows the actual image.

## Actual behavior

Both show a static `Image(systemName: "photo.fill")` placeholder. The real image is never rendered.

## Notes

- Import is confirmed working from the logs: `Created gallery item: Imported Image (4288×2848) at 1DE8D4CE-…​.dat` — bytes saved via `FileStorageManager`, dimensions correct.
- Root cause: the views never load the stored file. `GalleryListView.swift` (thumbnail, ~line 57) and `GalleryDetailView.swift` (original preview, ~line 22) both hardcode `Image(systemName: "photo.fill")` and ignore `item.originalImagePath`.
- Fix needs to load bytes from `FileStorageManager` (actor; `load(filename:)`) keyed on `originalImagePath` and render a cross-platform image (`NSImage`/`UIImage`). Thumbnails should downsample the (potentially 12MP) original rather than decoding it at full size for a 60×60 cell.

## Root cause

Import was never the problem — the original bytes were saved to Application Support and `originalImagePath` recorded correctly (the logs confirm it). The two views that should show the image instead rendered a fixed SF Symbol: `GalleryListView` used `Image(systemName: "photo.fill")` for the row thumbnail, and `GalleryDetailView` used the same symbol for the "Original Image" preview. Neither view ever read `item.originalImagePath` or loaded the stored file, so the imported picture was never displayed on any platform.

## Fix

Added `StoredImageView` (UI), a reusable async-loading image view that:
- loads the stored original's bytes through a new `GalleryCoordinator.loadOriginalImageData(path:)` (which reuses the coordinator's cached `FileStorageManager` actor and returns `nil` rather than throwing on a missing/unreadable file),
- decodes them on a detached task via `StoredImageDecoder.downsample(_:maxPixelSize:)`, an ImageIO `CGImageSourceCreateThumbnailAtIndex` pass that decodes the (potentially 12MP) original at a reduced size — cheap and memory-safe for a small cell. The decoded `CGImage` is returned across the actor boundary through a small `@unchecked Sendable` wrapper and rendered with `Image(decorative:scale:)` (no `NSImage`/`UIImage` branching needed),
- shows the `photo.fill` placeholder only while loading or when the file can't be read/decoded, and reloads via `.task(id: path)` when the path changes.

The gallery row now uses `StoredImageView(maxPixelSize: 180, contentMode: .fill)` clipped to 60×60, and the detail "Original Image" section uses `StoredImageView(maxPixelSize: 2048, contentMode: .fit)`.
