## Description

The PRD requires a manual host/port entry path for when mDNS discovery fails or isn't available. No such UI or flow exists today.

## Expected behavior

The user can add a display by entering host, port, a friendly name, and (optionally) native width/height. This creates a persisted `FlaschenTaschenDisplay` with `source = "manual"`, usable everywhere discovered displays are (registry, send, dimension matching).

## Notes

- New form view under `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/`.
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Models/FlaschenTaschenDisplay.swift` — has `host`, `port`, `displayName`, `displayWidth`, `displayHeight`, `source`, and an `endpoint` helper.
- PRD §3 "Manual Display Entry". Complements 0009; pairs with the registry UI in 0011.

## Root cause

This was a missing feature, not a bug: the PRD requires a manual host/port entry path for when mDNS discovery (0009) fails or is unavailable, but no UI or persistence flow existed. Discovered displays could be added via mDNS only.

## Fix

Added `ManualDisplayEntryView` under `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/`, styled after `VariantCreationView` (Form, toolbar Cancel/Add, `#if os(iOS)` guards for keyboard/autocapitalization modifiers). The form takes host, port, a friendly name, and native width/height. Validation is extracted into a pure, `nonisolated`, testable `ManualDisplayInput` struct whose `validate()` returns a `Result<Validated, ValidationError>`: host must be non-empty after trimming, port must parse to 1–65535, width/height must be positive integers, and an empty name falls back to the host. On confirm, the view calls a new `GalleryCoordinator.addManualDisplay(...)` which constructs a `FlaschenTaschenDisplay` with `source = "manual"` and inserts+saves it through the injected `ModelContext`, matching the coordinator's existing persistence pattern. Scope was kept to manual entry; the registry list screen remains 0011.

## Gotchas

- The package uses default `MainActor` isolation, so the validation struct had to be marked `nonisolated` to be callable from XCTest's nonisolated test methods.
- With a nested `Validated` and `ValidationError` in the same struct, the compiler couldn't infer `Result.failure`'s base — the error cases must be written fully qualified (`ValidationError.emptyHost`), and `ValidationError` must conform to `Error` for use as `Result`'s failure type.
