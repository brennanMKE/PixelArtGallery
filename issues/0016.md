## Description

The PRD specifies structured `os.log` logging with a fixed set of categories — `Gallery`, `ImageProcessor`, `Variant`, `FTDiscovery`, `Export`, `GridRenderer` — at `debug`/`info`/`warning`/`error` levels. Logging today is sparse and ad hoc: a couple of view models use category strings like `GalleryCoordinator`/`VariantDetailView`, and the image processing, discovery, and export paths have little or no logging.

## Expected behavior

Each subsystem logs through a consistent `Logger` with the PRD's category names and appropriate levels, so key events (import, pixelation, variant creation, discovery, export, send, render) are traceable in the unified logging system.

## Notes

- Existing loggers: `GalleryCoordinator.swift`, `VariantDetailView.swift`, `ExportPickerView.swift` (subsystem `com.pixelartgallery.ui`). Align categories and add coverage to `PixelationEngine`, the discovery service (0009), the FT client (0012), and the exporter (0007).
- PRD §"Logging". Lower priority than the functional issues; can be folded into each feature as it lands.

## Root cause

Logging accreted feature-by-feature with no shared definition. Three
view-model/view types each declared their own `Logger(subsystem:category:)`
under `com.pixelartgallery.ui` with type-named categories
(`GalleryCoordinator`, `VariantDetailView`, `ExportPickerView`); the
networking layer used a third subsystem (`com.pixelartgallery.networking`)
with `FTDiscovery` and `FTSend` categories. Neither subsystem nor categories
matched the PRD's fixed set, and the image-processing and export value types
had no logging at all, so key events (pixelation, export destination/result,
scan lifecycle) were untraceable.

## Fix

Added a single central definition, `AppLog` (`AppLog.swift`), with one
subsystem (`co.sstools.PixelArtGallery`, matching the app bundle id) and a
`Logger` per PRD category — `Gallery`, `ImageProcessor`, `Variant`,
`FTDiscovery`, `Export`, `GridRenderer` — plus a `Category` enum exposing the
canonical raw strings. `AppLog` is `nonisolated` (the package defaults to
`@MainActor` isolation) so the off-main-actor types can share the loggers;
`Logger` is `Sendable`. Every ad hoc logger was removed and each call site
routed to the matching central category. The former `FTSend` category was
folded into `FTDiscovery` so the set matches the PRD exactly (no extra
category added). Coverage was added where thin: pixelation start/source-
size/errors, scan start/results/browser failures, send target/encode/
success/validation-warnings, export format/destination/success/failure,
Photos save outcomes, gallery/variant import/create/duplicate/delete
lifecycle, and grid render/load-failure. Hot draw loops are deliberately
left unlogged. Levels follow debug=trace, info=event, warning/error=failure.

## Gotchas

The package sets default `@MainActor` isolation, so a plain `enum AppLog`
makes its static `Logger` properties main-actor-isolated and unusable from
`PixelationEngine`/`VariantExporter`/`PhotoLibrarySaver` (nonisolated) and
the `FTDiscoveryService`/`FTDisplayClient` actors. Marking the enum
`nonisolated` fixes it cleanly — `Logger` is `Sendable`.
