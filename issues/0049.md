## Description

All unit test automation should use Swift Testing (`import Testing`, `@Test`, `@Suite`, `#expect`, `#require`), not XCTest. The package currently has 14 test files still on XCTest. Migrate them. UI tests are the exception and must remain XCTest — but there is no UI-test target today, so this migration covers the whole `PixelArtGalleryKitTests` suite.

## Resolution notes

> 🟢 Resolved 2026-07-07 — All unit tests now use Swift Testing. `grep -rl "import XCTest" Tests/` returns nothing; the suite runs 106 tests across 17 Swift Testing suites with no XCTest harness cases.

## Root cause

The package was seeded with XCTest before the project settled on Swift Testing (see [[feedback_swift_testing]] / `issues/Issues.md`). 13 files still subclassed `XCTestCase` (two others — `PixelationEngineTests`, `FTDisplayClientTests` — were migrated earlier in #0048 and #0051).

## Fix

Rewrote all 13 remaining files against Swift Testing, preserving each test's intent, messages, and regression references:

- `XCTestCase` classes → `@Suite struct` (or `@Suite final class` where `deinit` is needed for temp-dir teardown: `FileStorageManagerTests`, `VariantExporterTests`, `GalleryCoordinatorTests`).
- `setUp`/`tearDown` → `init()` / `deinit`; this let per-test fixtures become non-optional `let`s, removing the `guard let … else { XCTFail }` boilerplate.
- `XCTAssert*` → `#expect(...)`; `XCTAssertEqual(_, _, accuracy:)` → `#expect(abs(a - b) <= tol)`; `XCTUnwrap` → `try #require`; `XCTFail` → `Issue.record`.
- Error assertions → `#expect(throws:)` (value or `.self`) and the `#expect { … } throws:` closure form for associated-value matching (`VariantExporterTests.invalidPixelData`).
- `@MainActor` preserved exactly where the originals had it (whole-suite or per-method, e.g. `DiscoveredFTDisplayTests`), since the test target has no default actor isolation.
- `throw XCTSkip` (HEIC-unavailable) → graceful early return.

## Review

Self-reviewed. Two Swift Testing gotchas surfaced and were fixed: `#expect`'s expression autoclosure is non-throwing, so throwing `context.fetch(...)` and `rethrows` `allSatisfy(...)` calls had to be hoisted into `let`s before the `#expect`. Test count is unchanged in coverage (same assertions), and each suite kept its per-test temp-dir isolation, so Swift Testing's default parallel execution is safe (no shared fixed paths).

## Gotchas

- `#expect(...)` evaluates its argument in a **non-throwing** autoclosure. Any `try` (e.g. `context.fetch`) or `rethrows` call (e.g. `allSatisfy`, `contains(where:)`) inside it must be hoisted to a `let` first — otherwise "call can throw, but it is not marked with 'try'". `#require` does not have this problem (its autoclosure is throwing).
- Temp-dir teardown needs a **class** suite (`deinit`); structs can't have `deinit`. Per-test unique UUIDs keep parallel runs from colliding, so no `.serialized` trait was needed.
- The SwiftData `@Model` types are constructed off the main actor in some suites exactly as the XCTest versions did; the test target's lack of `.defaultIsolation(MainActor.self)` is why per-method/suite `@MainActor` annotations had to be carried over verbatim.
