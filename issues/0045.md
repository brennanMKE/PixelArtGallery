## Description

Development builds from Xcode currently use the same bundle ID (`co.sstools.PixelArtGallery`) as the future notarized release, so a dev build and the shipped app share UserDefaults, the SwiftData store/Application Support directory, Sparkle state, and dock identity on the same machine. Give development builds a distinct identity so both coexist cleanly and dev data never contaminates the released app's state.

## Expected behavior

- Xcode/dev builds: bundle ID `co.sstools.PixelArtGallery.beta`, display name "Pixel Art Gallery Beta", a visually distinct app icon variant, and no production Sparkle feed (empty/absent `SUFeedURL` keeps Check for Updates disabled per #0038's `isConfigured` guard).
- Release builds (via `scripts/release.sh` / the release configuration): bundle ID `co.sstools.PixelArtGallery`, name "Pixel Art Gallery", production icon and feed.
- Switching requires no manual file edits — the build configuration/scheme drives it (xcconfig layering; e.g. an uncommitted `Active.xcconfig` including either a Beta or Prod xcconfig, or per-configuration settings).
- Both variants install side-by-side with separate Application Support/SwiftData stores.

## Notes

- The app's data directory is derived from the bundle's Application Support path in `PixelArtGalleryApp.getGalleryDatabaseURL()` — confirm the beta bundle ID actually yields a separate container path when unsandboxed (it uses a shared `~/Library/Application Support/PixelArtGallery` folder name today; the beta variant may need its own folder name driven by bundle ID or an Info.plist value).
- Beta icon: a badged variant of the existing macOS icon set (e.g. `AppIcon-macOS-Beta`) so dev and prod are distinguishable in the Dock.
- Optional but recommended before the first public release; the preflight (#0043) should assert release builds carry the production bundle ID.
- Watch the #0023 precedence rule when moving `PRODUCT_BUNDLE_IDENTIFIER` out of the pbxproj into xcconfigs, and keep the iOS build working (its bundle ID handling can stay as-is or follow the same split).

## Root cause

The target's Debug and Release pbxproj configurations both hard-coded `PRODUCT_BUNDLE_IDENTIFIER = co.sstools.PixelArtGallery`, `Info-MacOS.plist` baked in the production `SUFeedURL`, and both `PixelArtGalleryApp.getGalleryDatabaseURL()` and `FileStorageManager` hard-coded the `Application Support/PixelArtGallery` folder — so an Xcode dev build and the shipped app shared bundle identity, UserDefaults, the SwiftData store/image files, the Sparkle feed, and Dock appearance.

## Fix

The split is driven purely by the build configuration (Debug = beta, Release = production) — Xcode Run uses Debug by default and `scripts/release.sh` archives Release, so no manual file edits or scheme tricks are needed.

- **Per-configuration identity in the target's pbxproj buildSettings** (chosen over split xcconfig files: xcconfig lines are not per-configuration-aware, both configurations share `Build.xcconfig`, and per the #0023 precedence rule the target pbxproj level wins anyway — this also avoids the `//`-starts-a-comment problem an appcast URL would hit in an xcconfig):
  - Debug: `PRODUCT_BUNDLE_IDENTIFIER = co.sstools.PixelArtGallery.beta`, `INFOPLIST_KEY_CFBundleDisplayName = "Pixel Art Gallery Beta"`, `ASSETCATALOG_COMPILER_APPICON_NAME[sdk=macosx*] = AppIcon-macOS-Beta`, `PAG_SPARKLE_FEED_URL = ""`.
  - Release: `PRODUCT_BUNDLE_IDENTIFIER = co.sstools.PixelArtGallery`, `PAG_SPARKLE_FEED_URL = https://pixelartgallery.sstools.co/appcast.xml`; display name and icon fall through to the existing xcconfig values.
- **Sparkle feed**: `Info-macOS.plist`'s `SUFeedURL` is now `$(PAG_SPARKLE_FEED_URL)`. Debug builds get an empty string, which #0038's `isConfigured` (requires non-empty) treats as unconfigured — updater never starts, menu item stays disabled.
- **Beta icon**: `AppIcon-macOS-Beta.appiconset`, a copy of the macOS set with an orange BETA band drawn programmatically (AppKit script) on all 10 sizes. iOS keeps the shared `AppIcon-iOS` (single 1024px source; badging left out as non-trivial), but iOS Debug builds do carry the beta bundle ID and display name.
- **Storage separation**: new `StorageFolder` helper in PixelArtGalleryKit returns `PixelArtGallery-Beta` when the bundle ID has the `.beta` suffix, `PixelArtGallery` otherwise (nil/test runners included, so released-app data is never orphaned). Used by `FileStorageManager` and `getGalleryDatabaseURL()`.
- **Preflight**: new "Identity gates" (Release configuration must be the production bundle ID, Debug the beta one) and reworked Sparkle gates (Release settings must carry `PAG_SPARKLE_FEED_URL`; the built Debug app must NOT contain a production `SUFeedURL`).

- macOS Debug build: built app's Info.plist shows `CFBundleIdentifier = co.sstools.PixelArtGallery.beta`, `CFBundleDisplayName = Pixel Art Gallery Beta`, `CFBundleIconName = AppIcon-macOS-Beta`, `SUFeedURL` empty. Launched it: unified log shows `Sparkle updater not started: SUFeedURL is not configured (#0039)` (the #0038 guard), and the app created `~/Library/Application Support/PixelArtGallery-Beta/` with its own `gallery.db` while `~/Library/Application Support/PixelArtGallery` stayed untouched (mtime unchanged; beta window shows an empty gallery — see `issues/0045/beta-app-window.png`; badged icon: `issues/0045/beta-icon-256.png`). Beta test data removed afterwards.
- macOS Release build (`xcodebuild -configuration Release build`): built app shows the production bundle ID, name, `AppIcon-macOS`, and the real `SUFeedURL`.
- iOS Simulator build succeeds; its Debug app carries `co.sstools.PixelArtGallery.beta` / "Pixel Art Gallery Beta" with the shared iOS icon.
- `cd PixelArtGalleryKit && swift test` → 96 tests pass (92 existing + 4 new `StorageFolderTests`).
- `scripts/preflight.sh --skip-build --allow-dirty` runs; both new Identity gates and the reworked Sparkle gates pass (remaining failures are the pre-existing notary-profile/PAG_EC2_* environment gates).

## Gotchas

- An empty `SUFeedURL` is still a *present* plist key — #0038's `isConfigured` requires a non-empty string, so that's fine, but don't "simplify" it to a presence check.
- Per-config values must stay in the **target's** pbxproj configurations: moving them into `Build.xcconfig` can't express per-configuration differences, and an appcast URL in an xcconfig would be truncated at `//` (comment syntax).
- `StorageFolder` deliberately maps every non-`.beta` bundle ID (including nil) to the original `PixelArtGallery` folder — changing release builds to a bundle-ID-derived folder would orphan existing users' data.
- The preflight's old "built macOS app has SUFeedURL" gate would now fail on every Debug build; it was inverted on purpose (Debug must have an empty feed) with the production feed asserted via Release build settings instead.
- `AppLog`'s unified-logging subsystem stays `co.sstools.PixelArtGallery` in both variants; filter logs by process or `subsystem BEGINSWITH` when both apps run at once.
