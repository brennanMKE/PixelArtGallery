## Description

The popover itself ([#0065](0065.md)). Tapping a gallery image opens a **popover** (replacing the pushed item screen) containing exactly: an **FT display dropdown** defaulting to the **last-used** display, the **fitted + centered image** for the selected display, a **Send** button, and the **list of saved variants**. Selecting a display shows the transient fitted preview from [#0066](0066.md) (no distortion for any aspect); Send transmits that fitted grid centered on the display; saving promotes the transient preview into the saved-variants list. Shared for iOS and macOS.

## Long Description

Current navigation: `GalleryListView` (grid) → pushed `GalleryDetailView` → pushed `VariantDetailView`. This issue changes the grid cell tap to present a **popover** instead of pushing `GalleryDetailView`, and puts the whole send experience in that popover. Advanced per-frame layer/x/y controls are NOT in the popover (the offset is computed/centered automatically, layer uses the display's stored default); the existing `VariantDetailView` can remain as the editor reached from a saved-variant row.

## Plan

> API names from #0066 (`FittedPreview`, `fittedPreview(for:display:)`, `saveVariant(from:)`, the last-used-display resolver) are assumed below; track whatever #0066 actually lands.

### 1. Popover presentation — per-cell `.popover(item:)`, accept the iPhone sheet adaptation

In `GalleryListView.swift`, replace each cell's `NavigationLink(value: item)` with a `Button { selectedItem = item }` and attach the popover **to the cell**, not the grid. A single `.popover(item: $selectedItem)` cannot be attached to every cell (each visible cell would present when the binding goes non-nil), and attaching it once at the ScrollView level loses per-cell anchoring on macOS/iPad. So each cell gets a derived binding:

```swift
.popover(item: Binding(
    get: { selectedItem?.id == item.id ? selectedItem : nil },
    set: { if $0 == nil { selectedItem = nil } }
)) { item in
    GallerySendPopoverView(item: item, coordinator: coordinator)
}
```

- **iPhone (compact width): accept the default sheet adaptation.** Do NOT force `.presentationCompactAdaptation(.popover)` — this popover is a full send surface (dropdown + preview + Send + variants list); a true anchored popover on an iPhone is a cramped bubble that would clip the list and fight the keyboard. The automatic sheet is exactly what Apple's adaptation is for. Add `.presentationDragIndicator(.visible)`; no detents needed (the adapted sheet's default full presentation suits the list). Document the choice in a comment.
- **iPad / macOS**: real popover anchored on the tapped cell. On macOS give the content `.frame(minWidth: 360, idealWidth: 400, minHeight: 520, idealHeight: 620)` inside `#if os(macOS)` so it doesn't collapse to intrinsic size.
- **Dismissal**: outside-tap/ESC (macOS), swipe-down (iOS). The popover content's `.onDisappear` calls the send controller's `stop()` (see §3) so dismissing mid-send stops the loop and clears the display — the stop-clear already runs in a detached task (#0052/#0053), so it survives view teardown.
- Remove `GalleryListView`'s `.navigationDestination(for: GalleryItem.self)` — nothing routes to `GalleryDetailView` anymore (the file itself is retired in #0068, not here). Keep the root `NavigationStack` (toolbar/title still need it).

### 2. Popover content — new `UI/GallerySendPopoverView.swift`

One shared view for both platforms, wrapped in its **own `NavigationStack`** so a saved-variant row can push the existing `VariantDetailView` editor *inside* the popover/sheet (`VariantDetailView` already tears down its own send `.onDisappear`, and its `.navigationDestination(for: Variant.self)` registration moves here). Layout top-to-bottom:

1. **Header**: item name (`.navigationTitle` inline) — plus, on iOS, a Done/close button.
2. **FT display dropdown**: `@Query(sort: \FlaschenTaschenDisplay.displayName) displays`, a `Picker` (`.menu` style on macOS/iOS) bound to `@State selectedDisplayID: UUID?`. On appear (`onChange(of: displays…, initial: true)` like `VariantDetailView`), initialize the selection via #0066's last-used resolver (falling back per its rules); on user change, write the new id back through the #0066 last-used setter. Locked (`.disabled(isSending)`) during a send — same endpoint-captured-at-start rationale as #0057.
3. **Fitted preview**: `.task(id: selectedDisplayID)` calls `try await coordinator.fittedPreview(for: item, display:)` into `@State preview: FittedPreview?`, showing a `ProgressView` while nil/in-flight (the #0066 cache makes re-selection instant). Render inside a **display-canvas letterbox** (in — it's cheap and conveys placement): `Color.black.aspectRatio(displayW/displayH, contentMode: .fit)` with the fitted image centered in an `.overlay` — since the offset is always the centering offset, plain centering inside the display-aspect frame is visually exact.
4. **Send/Stop button**: borderedProminent, `Send Now` / red `Stop Sending`, disabled when `preview == nil`. Drives the shared send controller (§3) with the transient preview. Status banners (`StatusBanner`) as in `VariantDetailView`. **No layer/x/y steppers** — offset comes from the preview, layer from `display.layer`.
5. **Save button + saved-variants list**: a `Save as Variant` button calls `try coordinator.saveVariant(from: preview)` (disabled when nil; #0066 dedups repeats); the list shows `item.variants` (live via `@Model` observation) using the existing row style (`VariantThumbnailView` + dims), each row a `NavigationLink(value: variant)` → `VariantDetailView`. Nothing else ever persists the transient preview — mere selection/Send never accumulates variants.

Rendering the transient grid: extract `VariantThumbnailView`'s render path into a lightweight `PixelDataImageView(pixelGridData:width:height:)` (same `PixelGrid.fromRGBA8888(...).makeCGImage()` off-main + `Image(...).interpolation(.none).resizable().aspectRatio(contentMode: .fit)`), and make `VariantThumbnailView` a thin wrapper over it. Do **not** use `PixelGridView` here — its zoom/selection/readout chrome is wrong for a popover; no `PixelGridViewModel` change needed.

### 3. Send-loop reuse — extract `ViewModels/FTSendController.swift`

Move `VariantDetailView`'s continuous-send loop (`handleSendToDisplay`'s `sendTask` body, `stopSending`, `activeSend`, the 2s `sendRefreshInterval`, mid-send `requiresClear` clear, cancellation handling) into an `@Observable final class FTSendController` (MainActor via package default):

```swift
nonisolated struct FTSendPayload: Sendable {   // Networking/FTSendPayload.swift
    let host: String; let port: Int
    let width: Int; let height: Int
    let pixelGridData: Data; let scaleFactor: Double
}

@Observable final class FTSendController {
    private(set) var isSending = false
    private var sendTask: Task<Void, Never>?
    private var activeSend: FTPaintTarget?
    func start(payload: FTSendPayload,
               frameOffset: @escaping @MainActor () -> (layer: Int, offsetX: Int, offsetY: Int),
               onError: @escaping @MainActor (String) -> Void,
               onStopped: @escaping @MainActor (String) -> Void)
    func stop()   // cancel + detached sendClearFrame(to: activeSend) — #0053 semantics
}
```

`frameOffset` is read **each frame**, preserving #0057's live layer/x/y for `VariantDetailView` (its closure reads the `@State` steppers) while the popover passes a constant closure `{ (FlaschenTaschenDisplay.clampedLayer(display.layer), preview.offsetX, preview.offsetY) }`. Refactor `VariantDetailView` to delegate to the controller with identical behavior (banners stay in the view via the callbacks; `.onDisappear { controller.stop() }` unchanged). No duplication of the loop.

Pure, testable assembly helper (`Networking/FTSendPlan.swift`, `nonisolated`): `FTSendPlan.make(preview: FittedPreview, host:port:layer:) -> (payload: FTSendPayload, offset: (x: Int, y: Int, z: Int))` — clamps the layer via `FlaschenTaschenDisplay.clampedLayer`, offsets via `clampedOffset`, and carries the preview's fit dims/grid verbatim.

### 4. Empty state — no dead popover

When `displays` is empty, the popover body shows the Scan / Add Manually affordances instead of the picker+preview: extract `DisplayRegistryView`'s private `emptyState` + scan action into a small shared `UI/DisplayEmptyStateView` (taking the coordinator; presenting `DisplayEditorView(mode: .add)` in a sheet), used by both `DisplayRegistryView` and the popover. Note `GalleryListView` already seeds the default display (#0021), so this is a rare path — but it must work.

### 5. Platform notes

Single shared `GallerySendPopoverView`; `#if os(...)` only for: macOS popover `.frame`, iOS `.presentationDragIndicator`/Done button, and the existing navigation-bar conditionals. Both platforms must build **and be run** (see §6). iPad gets the true popover for free via regular width.

### 6. Testing + verification

**Swift Testing** (package, `cd PixelArtGalleryKit && swift test`):
- `FTSendPlanTests` — payload/offset assembly from a `FittedPreview` (fit dims carried, offsets = preview centering offsets, z = clamped `display.layer`, out-of-range layer clamped into 1…15).
- Controller-adjacent pure logic only; the loop itself is network-bound (existing #0057 behavior is regression-covered by `VariantDetailView` continuing to work).
- Default-display resolution is #0066's tests; here only assert the popover's wiring inputs via the plan helper if any glue logic is added.

**Builds (both platforms, mandatory):**
- `xcodebuild -project PixelArtGallery.xcodeproj -scheme PixelArtGallery -destination 'platform=macOS' build`
- `xcodebuild -project PixelArtGallery.xcodeproj -scheme PixelArtGallery -destination 'platform=iOS Simulator,name=iPhone 17 Pro' build`

**Run the app — mandatory, not optional** (#0064 shipped green-but-wrong because it was never run). On macOS and the iOS simulator, confirm by observation:
1. Tapping a gallery cell opens the popover (macOS: anchored popover; iPhone: sheet) — no push to `GalleryDetailView`.
2. The dropdown defaults to the last-used display and remembers a change across reopen.
3. Selecting a display with a non-matching aspect shows a visibly **un-stretched** fitted preview, letterboxed/centered in the display-aspect canvas.
4. Send toggles to Stop; dismissing the popover mid-send stops it (watch the log for the stop/clear lines).
5. Save adds exactly one row to the saved-variants list (and repeat-Save doesn't duplicate); merely selecting displays adds nothing.
6. A saved-variant row pushes the `VariantDetailView` editor inside the popover.

Design points (planner to finalize):
- **Popover presentation**: `.popover(item: $selectedItem)` on the grid. On iOS compact width a `.popover` adapts to a sheet by default — decide whether to force `.presentationCompactAdaptation(.popover)` or accept the sheet adaptation on iPhone (recommend accepting the adaptation so it's usable on a phone; a true tiny popover is bad on iPhone). Document the choice. Give the popover a sensible size on macOS (`.frame`/`.presentationDetents` as appropriate).
- **Dropdown**: a `Picker`/`Menu` over `@Query FlaschenTaschenDisplay`, default selection = last-used (from #0066), updating last-used on change. Empty state: if no displays, offer Scan/Add (reuse `DisplayRegistryView`'s affordances) instead of a dead popover.
- **Fitted preview**: on display selection, call `coordinator.fittedPreview(for:display:)` (async; show a brief spinner) and render the fitted grid via `PixelGridView`-style rendering at true aspect. Optionally show it within the display-canvas frame (letterboxed) to convey the centered placement — nice-to-have, planner's call.
- **Send**: a Send (and Stop) button that runs the continuous-send loop for the fitted grid at offset `(x: preview.offsetX, y: preview.offsetY, z: display.layer)`. Reuse/extract the loop from `VariantDetailView` so it can drive a transient preview (not only a persisted `Variant`) — factor the send-loop core into something callable from both. Keep the #0052/#0053 stop-clear + `.onDisappear` teardown semantics.
- **Saved variants list**: show `item.variants`; each row opens the existing editor/detail (`VariantDetailView`) or a save action promotes the current transient preview via `coordinator.saveVariant(from:)`. A **Save** affordance persists the current fitted preview into the list; the transient default must not silently accumulate saved variants.
- **Platform**: one shared view with `#if os` for popover sizing / presentation only.

Testing: extract any new pure logic (default-display resolution wiring, send-parameter assembly from a `FittedPreview`) into testable helpers and cover with Swift Testing. UI wiring itself isn't unit-testable — the required bar is both-platform builds AND **actually running the app** to confirm: popover opens on tap, dropdown defaults to last-used, the fitted preview is visibly un-stretched for a non-matching aspect, Send is wired, and Save adds to the variants list. The prior UI (#0064) passed builds/tests but was wrong in the running app — running it is mandatory here, not optional.
