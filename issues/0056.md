# 0056 — FT sends cannot set x/y offsets, and displays lack stored defaults for layer and offsets

## Description

Sending to an FT display uses a hostname and layer, but the FT protocol also supports an x and/or y offset which moves the image from the top or leading edge. When the image is smaller than the geometry of the display it may help to set these offsets, but the current UI does not make that possible. When a display is added it could store the height, width, layer and x/y values to use as defaults for sends.

## Plan

### Current state

The wire protocol work is already done: `FTDisplayClient.makePacket(width:height:pixelGridData:scaleFactor:offset:)` builds a P6 PPM datagram and injects `#FT: <x> <y> <z>\n` (x/y offset, z = layer) between the dimensions line and the `255` maxval — the exact form FT servers parse (`server/ppm-reader.cc`, fixed in #0051). `send(...)` already takes an `offset: (x:, y:, z:)` parameter per call, and `FTDisplayClientTests.swift` already asserts the `#FT:` comment for arbitrary x/y/z. What's missing is everything above the client:

- `FlaschenTaschenDisplay` persists `displayWidth`/`displayHeight`/`layer` but has **no `offsetX`/`offsetY`** default fields.
- `VariantDetailView` hardcodes `offset: (x: 0, y: 0, z: layer)` in both the continuous-send loop and the stop-clear black frame (#0053); only the layer is seeded from the display (`seedSendLayer()`).
- `ManualDisplayEntryView` / `ManualDisplayInput` collect host/port/name/width/height only — no layer or x/y — and `GalleryCoordinator.addManualDisplay(...)` doesn't accept them.
- `SettingsView`'s default-display editor edits width/height/layer but not x/y.

### Files and functions

- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Models/FlaschenTaschenDisplay.swift` — new persisted fields
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Networking/FTDisplayClient.swift` — no protocol change needed; API already offset-shaped (keep it per-call so per-frame values can vary, see #0057)
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/ManualDisplayEntryView.swift` — `ManualDisplayInput`, `Validated`, form fields
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/ViewModels/GalleryCoordinator.swift` — `addManualDisplay(...)` pass-through
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/SettingsView.swift` — default-display editor gains x/y
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/VariantDetailView.swift` — `sendSection`, `seedSendLayer()`, `startContinuousSend()`, `ActiveSend`, `clearSentLayer()`

### Approach

1. **Model fields with safe defaults.** Add `var offsetX: Int = 0` and `var offsetY: Int = 0` to `FlaschenTaschenDisplay` with inline defaults so SwiftData lightweight-migrates existing stores (same pattern as `layer` in #0047 — pre-existing records get `0, 0`, i.e. today's behavior). Add matching init parameters (defaulting to 0) and a small normalization helper (clamp negatives to 0; offsets must be non-negative and are bounded in practice by the display geometry). `displayWidth`/`displayHeight`/`layer` already exist — no change there.
2. **Entry/edit UI.** Extend `ManualDisplayInput` (+ `Validated` + `validate()`) with layer, x and y: layer via the existing `layerRange` stepper pattern, x/y as numeric fields validating to integers ≥ 0 (new `ValidationError` cases with messages). Add the fields to the `ManualDisplayEntryView` form and thread them through `DisplayRegistryView`'s `onAddDisplay` closure and `GalleryCoordinator.addManualDisplay(...)` into the model. Add x/y rows to `SettingsView`'s default-display editor (`editorSections`/`load`/`save`), mirroring how `layer` is handled there. #0054 will relocate/generalize display editing into Settings — keeping the validation on `ManualDisplayInput` (a plain testable struct) means that work reuses these rules unchanged.
3. **Wire defaults into the send UI.** In `VariantDetailView`, add `sendOffsetX`/`sendOffsetY` state alongside `sendLayer`; generalize `seedSendLayer()` to `seedSendDefaults()` seeding all three from the selected display (re-seeded on selection change, as today). Surface x/y controls next to the Layer stepper. Pass `(x: sendOffsetX, y: sendOffsetY, z: layer)` into `client.send(...)` in the continuous loop, and capture x/y in `ActiveSend` so the #0053 stop-clear black frame erases the frame at the *same* offset that was painted. Keep the offset a per-`send()` argument (it already is) — #0057 will make layer/x/y editable mid-send with the next frame picking up the new values, so nothing should bake the offset into the client at construction time.
4. **No packet change, but lock it in with tests.** `makePacket` already carries x/y — add/extend packet-level unit tests asserting a nonzero x/y offset from *display defaults* lands in the `#FT:` comment, plus tests for the new `ManualDisplayInput` validation cases (negative/non-numeric x/y) and `FlaschenTaschenDisplay` offset normalization/init defaults.

## Resolution notes

> 🟢 Resolved 2026-07-09 — Displays now persist migration-safe `offsetX`/`offsetY` defaults, validated x/y fields flow through the editor and coordinator into the send path, and `ActiveSend` captures the painted offset so the #0053 stop-clear black frame erases at the exact painted position.

## Root cause

`FlaschenTaschenDisplay` persisted `displayWidth`/`displayHeight`/`layer` but had no `offsetX`/`offsetY` fields, so there was nothing to seed into the send UI. `VariantDetailView` hardcoded `offset: (x: 0, y: 0, z: layer)` in both the continuous-send loop and the #0053 stop-clear black frame, and `ManualDisplayInput`/`DisplayEditorView` (the current add/edit form — `ManualDisplayEntryView` was renamed to `DisplayEditorView` in #0054, and `SettingsView` now just embeds `DisplayRegistryView`, so there was no separate default-display editor to touch) had no offset fields or validation. The wire protocol (`FTDisplayClient.makePacket`/`send`) already accepted a per-call `offset`, so this was purely a persistence/UI gap above an already-correct client.

## Fix

Followed the plan as written, with one scope note: the plan's step 2 said to extend `ManualDisplayInput` with "layer, x and y," but the current `DisplayEditorView` already handles layer as a separate bounded `Stepper` (1…15) outside `ManualDisplayInput`, passed to `onSave` as a second `Int` argument — that pattern predates this issue and works fine, so layer was left untouched. Only `offsetX`/`offsetY` (which need free-text integer validation, unlike the bounded layer stepper) were added to `ManualDisplayInput`/`Validated` with new `.invalidOffsetX`/`.invalidOffsetY` cases. The plan's `SettingsView` item is satisfied as-is per the task's context note: `SettingsView` only embeds `DisplayRegistryView`, which already opens `DisplayEditorView` for both add and edit, so extending `DisplayEditorView`'s "Defaults" section covers it.

Changes:
- `FlaschenTaschenDisplay`: added `offsetX`/`offsetY: Int = 0` (inline defaults for lightweight SwiftData migration, matching the `layer` precedent from #0047), matching init parameters, and `static func clampedOffset(_:)` (clamps negative to 0).
- `ManualDisplayInput`/`DisplayEditorView`: added `offsetX`/`offsetY` string fields with `0` defaults, two new `ValidationError` cases, and "Default X/Y Offset" text fields in the Defaults section. `input`/`save()` thread the new fields through unchanged otherwise.
- `DisplayRegistryView`: passes `validated.offsetX`/`validated.offsetY` into `coordinator.addManualDisplay(...)`; `updateDisplay(...)` already forwarded the whole `validated` struct, so no call-site change was needed there.
- `GalleryCoordinator`: `addManualDisplay(...)` gained `offsetX`/`offsetY: Int = 0` parameters (default-backward-compatible); `updateDisplay(...)` now also clamps and assigns `display.offsetX`/`display.offsetY`.
- `VariantDetailView`: added `sendOffsetX`/`sendOffsetY` state, `offsetXRange`/`offsetYRange` computed properties (bounded to the selected display's width/height so the stepper can't push a frame off-display), renamed `seedSendLayer()` to `seedSendDefaults()` (seeds layer + both offsets, clamped into range), added X/Y Offset steppers next to the Layer stepper, and added `offsetX`/`offsetY` to `ActiveSend` so `stopSending()`'s black-frame clear uses the exact offset that was painted rather than always `(0, 0)`.
- No change to `FTDisplayClient` — its `send`/`makePacket` API already took a per-call `offset`, per the plan's note that #0057 needs offset/layer to remain per-frame arguments rather than baked into the client at construction.

## Gotchas

- `offsetXRange`/`offsetYRange` in `VariantDetailView` bound the stepper to `0...(displayWidth - 1)` / `0...(displayHeight - 1)`; `seedSendDefaults()` additionally clamps a stored default into that range so a display whose stored offset exceeds its (possibly since-edited) geometry doesn't leave the stepper showing an out-of-range value.
- `DisplayEditorView`'s offset fields are plain, unbounded (≥ 0) text fields — they don't clamp to the display's own width/height at entry time, only in the send UI. This mirrors how width/height themselves aren't cross-validated against each other either, so it's consistent with the existing form's scope.

## Steps to reproduce

1. Open a variant smaller than the target display's geometry (e.g. an 8x8 variant for a 64x64 display).
2. Look for a way to position the image on the display before sending.

## Expected behavior

The send UI exposes x and y offset fields (defaulting from the display's stored values), and the FT packet carries them so the image is placed at that position on the display. Adding/editing a display lets the user set default width, height, layer, x and y.

## Actual behavior

Only hostname and layer are configurable; the image always renders at the display's origin, and displays store no default layer or offset values.

## Notes

The FT/FlaschenTaschen PPM header supports an offset footer (`x y z`, where z is the layer).
