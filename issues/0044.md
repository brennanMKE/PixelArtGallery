## Description

Document the end-to-end release process so cutting a release is a checklist, not tribal knowledge, and produce an explicit ship checklist that gates v1.0.0.

## Acceptance criteria

- `scripts/RELEASE.md` covers:
  - One-time setup: Developer ID certificate, notary keychain profile (`scripts/setup-keys.sh`), Sparkle EdDSA key generation/backup/restore (companion `scripts/SPARKLE.md` with the exact `generate_keys`/`sign_update` invocations and the `--account PixelArtGallery` convention), website deploy env vars.
  - Per-release sequence: preflight (#0043) → version bump (#0042, committed as `Bump version to <X.Y.Z>`) → build + full test pass → `release.sh` (#0037) → `verify-dmg.sh` → clean-Mac smoke test → appcast item via `appcast-item.sh` (#0041, never hand-typed) pasted newest-first + release notes → copy DMG to `website/downloads/` → re-run preflight (appcast↔DMG consistency) → `deploy-website.sh` (#0040) → tag `v<X.Y.Z>` → mirror notes in `changelog.html`.
  - Troubleshooting: common `spctl`/`notarytool`/Gatekeeper failure modes and the `notarytool log` incantation.
  - A note that notarization submissions and website deploys are user-run steps, not agent-run.
- `Ship-v1.md` at the repo root: an explicit checklist gating v1.0.0 — build/distribution gates (signed, notarized, stapled, Gatekeeper-clean on a never-seen Mac; Sparkle update from a previous build actually applies) plus a functional sanity matrix covering the app's core flows (import, name, gallery grid, pin/sort, variant create/edit/duplicate/delete, pixel preview zoom/pan/select, export per format, Photos save on iOS, display discovery/manual add, send to FT, settings/default display, rename/delete). Each row gets a "Verified" cell.

## Notes

## Root cause

Not a bug — a documentation gap. The release machinery (#0036–#0043) was complete but its operation lived only in eight resolved issue files: no single document said what to set up once, what to run per release and in what order, what the known failure modes are, or what gates v1.0.0. Cutting a release was tribal knowledge.

## Fix

- `scripts/RELEASE.md` — the end-to-end checklist. One-time setup: Developer ID Application cert (team `XV8BAAVZ6V`), `scripts/setup-keys.sh` for the `PixelArtGallery-notary` keychain profile, Sparkle EdDSA key **backup** (`generate_keys -x … --account PixelArtGallery`) and restore (`-f`) — the key already exists in this Mac's login keychain per #0039, so the doc warns against regenerating (fresh generation documented only for a new product line) — and the `PAG_EC2_HOST/PATH/KEY/PORT` deploy env vars. Per-release sequence (12 steps): `preflight.sh` → bump `MARKETING_VERSION` in `Configuration/Build.xcconfig` committed as `Bump version to <X.Y.Z>` → tests/builds → `release.sh` (user-run) → `verify-dmg.sh` → clean-Mac browser-download smoke test → `appcast-item.sh` item pasted newest-first with CDATA release notes → copy DMG to `website/downloads/` → re-run preflight (appcast↔DMG consistency gate) → `deploy-website.sh` (user-run) → `git tag v<X.Y.Z>` → mirror notes in `website/changelog.html`. Troubleshooting section covers the sibling issues' real gotchas: `notarytool log <id> --keychain-profile PixelArtGallery-notary`, missing notary profile, DMG-must-be-notarized-to-staple, sign_update keychain ACL prompt per binary path, pipefail+`grep -q` SIGPIPE, Xcode writing `MARKETING_VERSION` back into the pbxproj, and Gatekeeper's quarantine/assessment caching. Explicit note that notarization submissions and website deploys are user-run, not agent-run.
- `scripts/SPARKLE.md` — companion: how the integration is wired (macOS-only SPM product condition, `UpdaterController` starting only when `SUFeedURL` is configured, keys in `PixelArtGallery/Info-macOS.plist` incl. the real `SUPublicEDKey`), both helper-tool artifact paths (package `.build` and DerivedData `SourcePackages`), exact `generate_keys -p/-x/-f --account PixelArtGallery` and `sign_update --account PixelArtGallery` invocations, and per-release `appcast-item.sh` usage with the derived-not-hand-typed rationale.
- `Ship-v1.md` (repo root) — the v1.0.0 gate: 8 build/distribution rows (all-green preflight, xcconfig-only version bump, full `release.sh` run, six-check `verify-dmg.sh` pass, Gatekeeper-clean install on a never-seen Mac, Sparkle update actually applying — noting it needs two releases or a locally served appcast, website deployed with appcast reachable at the SUFeedURL, notes + tag) plus a 15-row functional sanity matrix (file-picker import + naming, grid + pin/sort, variant create/edit/duplicate/delete via dimension sheets, preview zoom/pan/select, PNG/HEIC/PPM/JSON export, FT send incl. default display + picker preselection, registry scan/manual add/rename/delete, Settings default-display edit/restore, item rename/delete, 640×480 min window, light/dark, Check for Updates enabled), each row with an empty Verified cell, and Known Limitations (iOS ships later, no Photos-library import on macOS, appcast empty until first release, English-only).
- `issues/Issues.md` — short Release pointer added to the build/verify section (no README exists at the repo root).
