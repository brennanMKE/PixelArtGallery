## Description

Document the end-to-end release process so cutting a release is a checklist, not tribal knowledge, and produce an explicit ship checklist that gates v1.0.0.

## Acceptance criteria

- `scripts/RELEASE.md` covers:
  - One-time setup: Developer ID certificate, notary keychain profile (`scripts/setup-keys.sh`), Sparkle EdDSA key generation/backup/restore (companion `scripts/SPARKLE.md` with the exact `generate_keys`/`sign_update` invocations and the `--account PixelArtGallery` convention), website deploy env vars.
  - Per-release sequence: preflight (#0043) → version bump (#0042, committed as `Bump version to <X.Y.Z>`) → build + full test pass → `release.sh` (#0037) → `verify-dmg.sh` → clean-Mac smoke test → appcast item via `appcast-item.sh` (#0041, never hand-typed) pasted newest-first + release notes → copy DMG to `website/downloads/` → re-run preflight (appcast↔DMG consistency) → `deploy-website.sh` (#0040) → tag `v<X.Y.Z>` → mirror notes in `changelog.html`.
  - Troubleshooting: common `spctl`/`notarytool`/Gatekeeper failure modes and the `notarytool log` incantation.
  - A note that notarization submissions and website deploys are user-run steps, not agent-run.
- `Ship-v1.md` at the repo root: an explicit checklist gating v1.0.0 — build/distribution gates (signed, notarized, stapled, Gatekeeper-clean on a never-seen Mac; Sparkle update from a previous build actually applies) plus a functional sanity matrix covering the app's core flows (import, name, gallery grid, pin/sort, variant create/edit/duplicate/delete, pixel preview zoom/pan/select, export per format, Photos save on iOS, display discovery/manual add, send to FT, settings/default display, rename/delete). Each row gets a "Verified" cell.
