## Description

`GalleryCoordinator` holds gallery items in a plain in-memory array (`var galleryItems: [GalleryItem] = []`) that nothing ever populates, and its create/delete methods don't touch a `ModelContext`. As a result nothing persists and the list is always empty. The coordinator needs access to the SwiftData context so items can be fetched, inserted, and deleted for real.

## Expected behavior

Gallery items are read from SwiftData (via `@Query` in the view feeding the coordinator, or a `ModelContext` injected into the coordinator) and reflect inserts/deletes immediately. `deleteGalleryItem`/`deleteVariant` remove objects from the context. Data survives app relaunch.

## Notes

- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/ViewModels/GalleryCoordinator.swift` ‚Äî the previous `@Query` property (invalid in an `@Observable` class) was replaced with a stub array during the build fix; this issue makes the data flow real.
- `@Query` belongs in a SwiftUI `View`; decide whether the view owns the query and assigns into the coordinator, or the coordinator fetches from an injected context. Keep `@Observable` (never `ObservableObject`) per PRD.
- Prerequisite for 0003, 0004, 0014, 0015.

## Root cause

`GalleryCoordinator` exposed a stub `var galleryItems: [GalleryItem] = []` that nothing ever populated (the original `@Query` had been removed because `@Query` is invalid in an `@Observable` class). The list always rendered the empty state, and `createGalleryItem` / `createVariant` only built model objects without a `ModelContext`, so nothing was inserted or saved. The delete methods mutated the local stub array instead of the context, so deletions did not persist either.

## Fix

Chose the **inject-the-context** approach: the SwiftUI view owns the live query, the coordinator owns mutations.

- `GalleryListView` now declares `@Query(sort: \GalleryItem.importedDate, order: .reverse) private var galleryItems` and reads `@Environment(\.modelContext)`. The list and empty-state bind directly to the query results (auto-updating), and the view injects the context into the coordinator via `coordinator.configure(modelContext:)` in `.onAppear`.
- `GalleryCoordinator` drops the stub array and instead holds `@ObservationIgnored private var modelContext: ModelContext?` (a mutation detail that must not invalidate views). `configure(modelContext:)` is idempotent (`!==` guard) so repeated `onAppear` calls don't churn. The type stays `@Observable` (never `ObservableObject`) and is `@MainActor` via the package's `.defaultIsolation(MainActor.self)`, per the PRD.
- `createGalleryItem` / `createVariant` now `insert` the new model and `save()` through the context (throwing a new `GalleryCoordinatorError.missingModelContext` if called before configuration).
- `deleteGalleryItem(_:)` / `deleteVariant(_:)` now take the model object, call `modelContext.delete(...)` + `save()`, clear selection, and surface failures via `currentError`.
- Wired swipe-to-delete (`ForEach` + `.onDelete`) in the gallery list and a destructive context-menu item for variants in `GalleryDetailView`, so the real delete paths are exercised.
- The import-save (0003) and variant-generation (0004) TODOs were left in place as instructed; only the fetch/insert/delete data flow was made real.
