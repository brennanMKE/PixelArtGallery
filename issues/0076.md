# 0076 — Minimal pixel editor: tap a pixel, pick a color, set it, and save the variant

| | |
|---|---|

## Description

Turn the variant **Preview** (today a read-only inspector — zoom/pan + tap-to-read a pixel's coordinate/RGBA) into a minimal editor: **tap a pixel → pick a color → set it → save**. This is v1 — no drag-paint, no undo/redo, no palette/eyedropper yet (those are follow-ons). It completes the "customize the pixels and save a variant" loop and pairs with the built-in sprites ([#0074](0074.md)): start from a sprite, tweak a few pixels, save your own variant.

## Long Description

The Preview section in `VariantDetailView` renders `PixelGridView` (backed by `PixelGridViewModel`), which decodes the variant's `pixelGridData`, supports zoom/pan, and lets the user tap a pixel to read its coordinate + RGBA. It has no write path. v1 adds: choose a color for the selected pixel and apply it, with an explicit **Save** that persists the edited grid back to the variant. The existing inspect (tap-to-read) behavior must be preserved — use a mode toggle so tapping doesn't accidentally paint.

## Notes

Relevant code:
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/PixelGridView.swift` + `ViewModels/PixelGridViewModel.swift` — the inspector: `init(variant:)` decodes `pixelGridData` into `[PixelColor]`; zoom/pan; `gridCoordinate(at:in:)` maps a tap to a pixel; the selected-pixel coordinate + RGBA readout. This is where the paint mode + write path go.
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/ImageProcessing/PixelGrid.swift` — `setColor(_:x:y:)` already exists; `toRGBA8888()` re-encodes. `PixelColor` (RGBA8888).
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/VariantDetailView.swift` — hosts the Preview section (and the send/export sections). The Save affordance lives here or in the grid view.
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/ViewModels/GalleryCoordinator.swift` — `updateVariantDimensions(...)` is the precedent for mutating + saving a variant; add an analogous `updateVariantPixels(_:pixelGridData:)` (or similar) that writes the new `pixelGridData` and `modelContext.save()`. Variants are the user's (editing a built-in sprite's variant is fine; built-in originals stay non-deletable).

Design points (planner to finalize, keep v1 minimal):
- **Mode toggle**: an Inspect ↔ Paint toggle (segmented control / button) so tapping reads vs. paints. Default Inspect (preserve current behavior). Only in Paint mode does a tap set the pixel.
- **Color source (v1)**: a single system `ColorPicker` bound to a "current paint color" (seed it from the currently selected pixel so "pick a color, set it" feels natural). Palette swatches + eyedropper are a follow-on, not v1.
- **Apply**: in Paint mode, tapping a pixel sets it to the current paint color via `PixelGrid.setColor` on an in-memory working copy; the preview updates live. (Single-tap per pixel; no drag in v1.)
- **Save**: an explicit **Save** button (enabled when there are unsaved edits) re-encodes the working grid to `pixelGridData` and persists via a new coordinator method + `modelContext.save()`. A subtle "unsaved changes" indicator; discard on leave (or prompt) — planner's call, keep it simple.
- **Working copy + dirty tracking**: edits mutate an in-memory `[PixelColor]`/`PixelGrid` in `PixelGridViewModel`, not the SwiftData model, until Save. Track a `hasUnsavedEdits` flag.
- **Zoom/pan while painting**: keep the existing zoom/pan; ensure a paint tap is distinguishable from a pan gesture (tap vs drag) so painting at high zoom is usable.
- **Out of scope (explicit follow-ons)**: drag-to-paint, undo/redo, palette + eyedropper, fill/line tools, resizing while editing. File these as follow-ons only if the user wants them.
- Shared iOS/macOS; **Swift Testing** for any pure logic.

Testing (Swift Testing): keep the paint mutation testable — e.g. a pure helper or a `PixelGridViewModel` method `setPixel(x:y:color:)` that updates the working grid + dirty flag, and an encode step producing the expected `pixelGridData`; assert a set-pixel changes exactly that pixel's bytes and marks dirty, and that the coordinator's `updateVariantPixels` persists the new bytes (in-memory container). UI wiring (mode toggle, tap-to-paint at zoom) isn't unit-testable — verify by **running the app**: enter Paint mode, tap a pixel, pick a color, confirm the pixel changes live, Save, reopen the variant and confirm the edit persisted; Inspect mode still reads without painting.
