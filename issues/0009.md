## Description

There is no Flaschen Taschen discovery at all. The `Networking` module referenced by `PixelArtGalleryKit.swift` does not exist, and nothing scans the local network. The `FlaschenTaschenDisplay` model exists but is never populated from discovery. This is a core MVP feature.

## Expected behavior

An on-demand mDNS/Bonjour scan finds FT displays advertising `_flaschen-taschen._tcp` (or the agreed service type), extracts endpoint (host/port) and — where available — native display width/height, and creates/updates `FlaschenTaschenDisplay` records (`source = "mdns"`) in SwiftData. Discovery fails gracefully so the manual-entry fallback (0010) still works.

## Notes

- Create a discovery service under `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Networking/`.
- Use `Network.framework` (`NWBrowser`) for macOS/iOS compatibility, in an `actor`/`nonisolated` context per the PRD concurrency rules.
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Models/FlaschenTaschenDisplay.swift` — target model.
- PRD §3 "Discovery" and §"mDNS Discovery". Log under category `FTDiscovery`.

## Root cause

The `Networking` module that `PixelArtGalleryKit.swift` referenced did not exist — nothing scanned the local network, so the `FlaschenTaschenDisplay` model could only ever be populated by manual entry. This was a missing feature, not a regression.

## Fix

Added the `Networking` module under `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Networking/`:

- **`FTDiscoveryService`** — an `actor` (so it runs off the main actor despite the package's `.defaultIsolation(MainActor.self)` setting) that browses the Bonjour service type `_flaschen-taschen._tcp` using `Network.framework`'s `NWBrowser`. It exposes `scan(duration:)`, which returns an `AsyncStream<DiscoveredFTDisplay>` and finishes automatically after a bounded scan window or when `stop()` is called. A failed browser state finishes the stream rather than throwing, so discovery degrades gracefully to manual entry (0010). Results are resolved to host/port and TXT metadata is read from each result. Logs under category `FTDiscovery`.
- **`DiscoveredFTDisplay`** — a `nonisolated`, `Sendable`, `Hashable` value type fully decoupled from SwiftData. `make(host:port:serviceName:txtRecord:)` parses display `width`/`height` from the TXT record (accepting `w`/`h` aliases, case-insensitive keys, whitespace-trimmed, positive integers only; otherwise `nil`). `makeDisplayModel(...)` (MainActor, since `FlaschenTaschenDisplay` is a SwiftData `@Model`) converts a discovered value into a `FlaschenTaschenDisplay` with `source = "mdns"`, falling back to default dimensions when none were advertised.

Per scope, SwiftData persistence/merge and the registry UI are deferred to issue 0011; this issue delivers the discovery service, the result model, and the conversion path only.

## Gotchas

- The package sets `.defaultIsolation(MainActor.self)`, so a plain `struct`/method is MainActor-isolated by default. `DiscoveredFTDisplay` had to be marked `nonisolated` so the browser's background callbacks can build it, while `makeDisplayModel()` stays `@MainActor` because it constructs a SwiftData `@Model`. Test methods that call it must also be `@MainActor`.
