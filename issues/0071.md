# 0071 — Bottom button bar (Settings, Sort, large centered +), removed from the top toolbar

## Description

Second half of the gallery chrome redesign ([#0069](0069.md)). Move the `+` (add image), **Settings**, and **Sort** actions out of the top toolbar into a **bottom button bar** — likely a custom bar — with the **`+` centered and larger** as the primary "add new image" affordance, flanked by Settings and Sort. This clears the top for the banner header ([#0070](0070.md)) and puts the most-used action (add image) in easy thumb reach.

## Long Description

Today (`UI/GalleryListView.swift`) the top `.toolbar` holds `+` (`.primaryAction`), a Sort `Menu` (`.secondaryAction`), and a Settings gear (`.secondaryAction`, iOS only). This issue relocates them to a bottom bar with a prominent centered `+`.

## Plan

### Current state (verified in code)

`UI/GalleryListView.swift` body is `NavigationStack → ZStack { BackgroundPixelsView().ignoresSafeArea(); Group { EmptyStateView | ScrollView+LazyVGrid } }`. The `.toolbar { … }` block (lines 105–135) holds exactly three items:

1. `ToolbarItem(placement: .primaryAction)` — `+` button → `coordinator.showImagePicker = true` (line 106–110).
2. `ToolbarItem(placement: .secondaryAction)` — Sort `Menu` wrapping `Picker("Sort By", selection: $sortOrderRawValue)` over `GallerySortOrder.allCases` / `order.displayName` (lines 113–123).
3. `ToolbarItem(placement: .secondaryAction)` inside `#if os(iOS)` — gear button → `showSettings = true` (lines 124–134). macOS has **no** Settings toolbar button; it uses the app `Settings` scene (⌘,, `PixelArtGallery/PixelArtGalleryApp.swift:97`).

State all lives on `GalleryListView`: `@AppStorage("gallerySortOrder") sortOrderRawValue` (line 46), `showSettings` (`#if os(iOS)`, line 34), and `coordinator.showImagePicker`. The presentation surfaces — image-picker sheet/`fileImporter` (`$coordinator.showImagePicker`, lines 209–244), naming sheet, and the iOS `SettingsView` sheet (`$showSettings`, lines 253–260) — hang off the view **outside** the `NavigationStack` and are untouched by this issue. `SettingsView` and the sort binding are directly reusable from any control; no coupling to `ToolbarItem`.

### Platform decision: bottom bar is iOS-only; macOS keeps its native top toolbar

Option (b). Justification: the redesign is iPhone-first (see `issues/0069/current-gallery-header.png` — the whole point is thumb-reach on a phone); macOS conventions put actions in the top toolbar, a raised center-action bar reads as foreign under a mouse pointer, and macOS already routes Settings through ⌘,. This also keeps the macOS diff at zero risk: on macOS nothing changes at all.

Concrete `#if os` split in `GalleryListView`:
- Wrap the entire existing `.toolbar { … }` block in `#if os(macOS)` and delete the iOS-only Settings `ToolbarItem` from inside it (it was iOS-only, so the macOS toolbar keeps exactly what it has today: `+` and Sort). No empty `.toolbar` remains on either platform.
- Add the bottom bar in `#if os(iOS)` via `.safeAreaInset(edge: .bottom)`.
- `showSettings`, the Settings sheet, and `.toolbarBackground(.hidden, for: .navigationBar)` stay iOS-only as they are today. Keep `.navigationTitle("Gallery")` untouched — #0070 owns the title/banner.

### New component: `UI/GalleryBottomBar.swift`

A dumb, platform-neutral view (compiles on both platforms, installed only on iOS) that owns zero state — it takes the sort selection binding plus two closures, so `GalleryListView` keeps ownership and the bar stays preview-/testable:

```swift
struct GalleryBottomBar: View {
    @Binding var sortOrderRawValue: String
    let onAddImage: () -> Void
    let onShowSettings: () -> Void

    var body: some View {
        HStack(spacing: 0) {
            // Left slot — Settings
            Button(action: onShowSettings) {
                Label("Settings", systemImage: "gearshape")
                    .labelStyle(.iconOnly)
                    .font(.title3)
            }
            .frame(maxWidth: .infinity)

            // Center — the primary add-image affordance
            Button(action: onAddImage) {
                Image(systemName: "plus")
                    .font(.title2.weight(.semibold))
                    .foregroundStyle(.white)
                    .frame(width: 60, height: 60)
                    .background(Circle().fill(Color.pixelAccent))
                    .shadow(radius: 3, y: 2)
            }
            .accessibilityLabel("Add Image")

            // Right slot — Sort (same Menu/Picker as today's toolbar)
            Menu {
                Picker("Sort By", selection: $sortOrderRawValue) {
                    ForEach(GallerySortOrder.allCases, id: \.rawValue) { order in
                        Text(order.displayName).tag(order.rawValue)
                    }
                }
            } label: {
                Label("Sort", systemImage: "arrow.up.arrow.down")
                    .labelStyle(.iconOnly)
                    .font(.title3)
            }
            .frame(maxWidth: .infinity)
        }
        .padding(.vertical, Theme.Spacing.s)
        .padding(.horizontal, Theme.Spacing.l)
        .background(.bar, ignoresSafeAreaEdges: .bottom)
        .overlay(alignment: .top) { Divider() }
    }
}
```

Notes on the shape:
- The two side slots use `.frame(maxWidth: .infinity)` so the `+` stays geometrically centered regardless of side-glyph widths; side buttons keep their standard ≥44pt targets, the `+` is a 60pt circle (≥ the ~56pt minimum asked for), gold `Color.pixelAccent` fill (`Palette.swift`), white glyph.
- "Raised" treatment: give the `+` a small `.offset(y: -10)` plus matching `.padding(.top, 10)` on the `HStack` so the circle pops above the bar line tab-bar-center-action style without being clipped. If it fights the divider visually, dropping the offset (flat centered circle) is an acceptable fallback — implementer's call on the exact raise, but the size/color/centering are firm.
- `.background(.bar, ignoresSafeAreaEdges: .bottom)` lets the material extend under the home indicator while the buttons stay inside the safe area. This coexists with #0070's matte background whichever lands first.
- Add an `#Preview` for the bar with a `@Previewable @State` raw value.

### Wiring in `GalleryListView` (iOS)

Attach to the `ZStack` (inside the `NavigationStack`, after the existing modifiers):

```swift
#if os(iOS)
.safeAreaInset(edge: .bottom) {
    GalleryBottomBar(
        sortOrderRawValue: $sortOrderRawValue,
        onAddImage: { coordinator.showImagePicker = true },
        onShowSettings: { showSettings = true }
    )
}
#endif
```

`safeAreaInset(edge: .bottom)` shrinks the content's bottom safe area, so:
- the grid `ScrollView` automatically insets — the last row scrolls clear of the bar (its existing `padding(Theme.Spacing.l)` provides the breathing room above the bar edge);
- the `EmptyStateView` path centers within the reduced area and its own "Import Image" button stays reachable (keep it — it's not a duplicate of the bar, it's the empty-state CTA);
- `BackgroundPixelsView().ignoresSafeArea()` still extends behind the bar's material.

No sheet/picker/state moves: `showImagePicker`, `pendingImport`, `showSettings`, and both sheets stay exactly where they are on `GalleryListView`.

## Notes

Relevant code:
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/GalleryListView.swift` — the `.toolbar { … }` block (~105–135): `+` opens `coordinator.showImagePicker`; Sort is a `Menu` over `@AppStorage("gallerySortOrder")` / `GallerySortOrder.allCases`; Settings sets `showSettings = true` (iOS) — macOS uses the `Settings` scene (⌘,), so there is no Settings button on macOS today.
- The image picker sheet, sort binding, and settings sheet already exist and stay; only the *controls* move.
- `UI/Theme.swift` / `Palette.swift` for spacing, the accent (`pixelAccent` = gold `#F7B900`) to make the `+` pop.

Design points (planner to finalize):
- **Bottom bar layout**: a custom bar pinned to the bottom safe area with three slots — Settings (left), a large centered circular `+` (raised/prominent, accent-filled), Sort (right). Decide exact styling (a floating pill vs a full-width bar; a raised center button à la a tab bar's center action). Keep it usable one-handed on iPhone.
- **`+` prominence**: larger tap target, accent color, clearly the primary action; opens the image picker (`coordinator.showImagePicker = true`).
- **Sort**: keep it a `Menu`/picker (same `GallerySortOrder` binding) triggered from the bar's Sort button.
- **Settings**: the bar's Settings button presents the same `SettingsView` sheet on iOS. On macOS, Settings stays in the app `Settings` scene (⌘,) — decide whether the bottom bar shows a Settings button on macOS at all (recommend: iOS shows Settings in the bar; macOS omits it and keeps ⌘,), or whether macOS uses a bottom bar at all (a top toolbar may remain more native on macOS — planner to weigh; the screenshot/intent is iPhone-first).
- **Remove from top toolbar**: delete the corresponding `ToolbarItem`s so the actions aren't duplicated; keep the root `NavigationStack` (the banner/title from #0070 lives there).
- **Layout interaction**: ensure the bottom bar doesn't overlap the last grid row (content inset / safe-area padding) and coexists with the banner from #0070.
- **Platform**: shared view with `#if os` where bottom-bar conventions differ.

Testing: mostly visual/non-unit-testable; keep any new pure logic (e.g. the sort-selection wiring) testable. Required bar: both-platform builds AND **running the app** (iOS) to confirm the bottom bar appears, the centered `+` adds an image, Sort and Settings work from the bar, and nothing is duplicated in the top toolbar or clipped by the bar.
