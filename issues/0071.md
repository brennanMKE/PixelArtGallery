## Description

Second half of the gallery chrome redesign ([#0069](0069.md)). Move the `+` (add image), **Settings**, and **Sort** actions out of the top toolbar into a **bottom button bar** ‚Äî likely a custom bar ‚Äî with the **`+` centered and larger** as the primary "add new image" affordance, flanked by Settings and Sort. This clears the top for the banner header ([#0070](0070.md)) and puts the most-used action (add image) in easy thumb reach.

## Long Description

Today (`UI/GalleryListView.swift`) the top `.toolbar` holds `+` (`.primaryAction`), a Sort `Menu` (`.secondaryAction`), and a Settings gear (`.secondaryAction`, iOS only). This issue relocates them to a bottom bar with a prominent centered `+`.

Option (b). Justification: the redesign is iPhone-first (see `issues/0069/current-gallery-header.png` ‚Äî the whole point is thumb-reach on a phone); macOS conventions put actions in the top toolbar, a raised center-action bar reads as foreign under a mouse pointer, and macOS already routes Settings through ‚åò,. Concrete `#if os` split in `GalleryListView`:
- Wrap the entire existing `.toolbar { ‚Ä¶ }` block in `#if os(macOS)` and delete the iOS-only Settings `ToolbarItem` from inside it (it was iOS-only, so the macOS toolbar keeps exactly what it has today: `+` and Sort). No empty `.toolbar` remains on either platform.
- Add the bottom bar in `#if os(iOS)` via `.safeAreaInset(edge: .bottom)`.
- `showSettings`, the Settings sheet, and `.toolbarBackground(.hidden, for: .navigationBar)` stay iOS-only as they are today. Keep `.navigationTitle("Gallery")` untouched ‚Äî #0070 owns the title/banner.

### New component: `UI/GalleryBottomBar.swift`

A dumb, platform-neutral view (compiles on both platforms, installed only on iOS) that owns zero state ‚Äî it takes the sort selection binding plus two closures, so `GalleryListView` keeps ownership and the bar stays preview-/testable:

```swift
struct GalleryBottomBar: View {
    @Binding var sortOrderRawValue: String
    let onAddImage: () -> Void
    let onShowSettings: () -> Void

    var body: some View {
        HStack(spacing: 0) {
            // Left slot ‚Äî Settings
            Button(action: onShowSettings) {
                Label("Settings", systemImage: "gearshape")
                    .labelStyle(.iconOnly)
                    .font(.title3)
            }
            .frame(maxWidth: .infinity)

            // Center ‚Äî the primary add-image affordance
            Button(action: onAddImage) {
                Image(systemName: "plus")
                    .font(.title2.weight(.semibold))
                    .foregroundStyle(.white)
                    .frame(width: 60, height: 60)
                    .background(Circle().fill(Color.pixelAccent))
                    .shadow(radius: 3, y: 2)
            }
            .accessibilityLabel("Add Image")

            // Right slot ‚Äî Sort (same Menu/Picker as today's toolbar)
            Menu {
                Picker("Sort By", selection: $sortOrderRawValue) {
                    ForEach(GallerySortOrder.allCases, id: \.rawValue) { order in
                        Text(order.displayName).tag(order.rawValue)
                    }
                }
            } label: {
                Label("Sort", systemImage: "arrow.up.arrow.down")
                    .labelStyle(.iconOnly)
                    .font(.title3)
            }
            .frame(maxWidth: .infinity)
        }
        .padding(.vertical, Theme.Spacing.s)
        .padding(.horizontal, Theme.Spacing.l)
        .background(.bar, ignoresSafeAreaEdges: .bottom)
        .overlay(alignment: .top) { Divider() }
    }
}
```

Notes on the shape:
- The two side slots use `.frame(maxWidth: .infinity)` so the `+` stays geometrically centered regardless of side-glyph widths; side buttons keep their standard ‚â•44pt targets, the `+` is a 60pt circle (‚â• the ~56pt minimum asked for), gold `Color.pixelAccent` fill (`Palette.swift`), white glyph.
- "Raised" treatment: give the `+` a small `.offset(y: -10)` plus matching `.padding(.top, 10)` on the `HStack` so the circle pops above the bar line tab-bar-center-action style without being clipped. If it fights the divider visually, dropping the offset (flat centered circle) is an acceptable fallback ‚Äî implementer's call on the exact raise, but the size/color/centering are firm.
- `.background(.bar, ignoresSafeAreaEdges: .bottom)` lets the material extend under the home indicator while the buttons stay inside the safe area. This coexists with #0070's matte background whichever lands first.
- Add an `#Preview` for the bar with a `@Previewable @State` raw value.

### Wiring in `GalleryListView` (iOS)

Attach to the `ZStack` (inside the `NavigationStack`, after the existing modifiers):

```swift
#if os(iOS)
.safeAreaInset(edge: .bottom) {
    GalleryBottomBar(
        sortOrderRawValue: $sortOrderRawValue,
        onAddImage: { coordinator.showImagePicker = true },
        onShowSettings: { showSettings = true }
    )
}
#endif
```

`safeAreaInset(edge: .bottom)` shrinks the content's bottom safe area, so:
- the grid `ScrollView` automatically insets ‚Äî the last row scrolls clear of the bar (its existing `padding(Theme.Spacing.l)` provides the breathing room above the bar edge);
- the `EmptyStateView` path centers within the reduced area and its own "Import Image" button stays reachable (keep it ‚Äî it's not a duplicate of the bar, it's the empty-state CTA);
- `BackgroundPixelsView().ignoresSafeArea()` still extends behind the bar's material.

No sheet/picker/state moves: `showImagePicker`, `pendingImport`, `showSettings`, and both sheets stay exactly where they are on `GalleryListView`.

## Root cause

Not a bug fix but a chrome-layout gap: the `+` (add image), Sort, and (iOS) Settings actions all lived in the top `.toolbar`, which is cramped, one-handed-unfriendly, and gives the most-used action (adding an image) no visual prominence. Companion issue #0070 cleared the top of the screen for a banner header, making room for the bottom-bar redesign planned here.

## Fix

Followed the plan as written, no deviations:

- Added `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/GalleryBottomBar.swift` ‚Äî a dumb, state-free `View` taking `@Binding var sortOrderRawValue: String` plus `onAddImage`/`onShowSettings` closures. An `HStack` with a Settings gearshape button and a Sort `Menu` (both `.frame(maxWidth: .infinity)` side slots) flanking a centered 60pt gold (`Color.pixelAccent`) circular `+` button with a white glyph, `.shadow(radius: 3, y: 2)`, and `accessibilityLabel("Add Image")`. The `+` is raised via `.offset(y: -10)` with a matching `.padding(.top, 10)` on the `HStack` so it pops above the bar line without being clipped by the top `Divider` overlay. Background is `.bar` material extending under the bottom safe area (`.background(.bar, ignoresSafeAreaEdges: .bottom)`). Includes a `#Preview` with `@Previewable @State`.
- In `GalleryListView.swift`: wrapped the entire existing `.toolbar { ‚Ä¶ }` block in `#if os(macOS)` and deleted the iOS-only Settings `ToolbarItem` from inside it ‚Äî macOS's toolbar is unchanged (`+` and Sort only; it never had a Settings button, using the app's `Settings` scene / ‚åò, instead). Added an `#if os(iOS)` `.safeAreaInset(edge: .bottom)` on the `ZStack` (attached alongside the other body modifiers, before `.navigationTitle`) that installs `GalleryBottomBar`, wired to the same `$sortOrderRawValue` binding and `coordinator.showImagePicker` / `showSettings` closures the old toolbar items used. No sheets, `@State`, or presentation surfaces moved ‚Äî only the trigger controls did.
- Updated the doc comments on `showSettings` and its `.sheet` to say the gear now lives in the bottom bar (iOS) rather than a toolbar button.
