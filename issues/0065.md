## Description

Umbrella for the corrected display-first design. Replace the pushed navigation (gallery item screen → "Send to Display" button → display-picker screen → send screen) with a single **popover** presented when a gallery image is tapped. The end state is just two surfaces: the **gallery grid** and a **popover**. The popover contains, and only contains: an **FT display dropdown** (defaulting to the last-used display), a **Send** button, the **fitted + centered image** for the selected display, and the **list of saved variants**. Selecting a display auto-builds a fitted, centered variant that is **cached transiently** (not persisted) unless the user edits and saves it. Aspect-mismatched images always **size to fit with a centering offset** — never stretched — everywhere in the app.

## Long Description

This supersedes the UI approach of [#0061](0061.md)/[#0064](0064.md), which added a step instead of removing one (a "Send to Display" button pushing a separate `DisplaySendPickerView`) and left the legacy/manual creation path stretching non-matching aspect ratios. The corrected flow, per the product owner:

1. Gallery keeps showing the grid of imported images.
2. Tapping an image presents a **popover** (not a pushed detail screen).
3. In the popover, an **FT display dropdown** defaults to the **last-used** display (users send to the same display repeatedly).
4. Selecting a display makes its dimensions known, so a **fitted, centered variant is created and cached** immediately — transient (in-memory) by default.
5. If the user doesn't customize it, it stays a transient cached preview (nothing added to the store). If it's **edited and saved**, it is persisted and appears in the popover's **saved-variants** section.
6. When the image's aspect ratio differs from the display's, it **sizes to fit** with the appropriate x/y **offset to center** — no distortion.

Reuse / rework of prior work:
- **Reused:** [#0062](0062.md)'s `AspectFit.fit(...)` / `AspectFit.centeringOffset(...)` (the fit + centering math is correct and stays).
- **Reworked:** [#0063](0063.md)'s `createFittedVariant` currently *always persists*. The new model makes a fitted variant **transient by default**, persisted only on save.
- **Retired:** [#0064](0064.md)'s `DisplaySendPickerView`, the "Send to Display" toolbar button, the pushed item-detail-as-primary flow, and the manual "Custom Size…" creation path — all replaced by the popover.

Phases (children):
- **[#0066](0066.md)** — Model/coordinator: a transient fitted-preview value type + in-memory cache, a `saveVariant(from:)` persistence path, and last-used-display persistence. Reuses #0062; reworks #0063 to transient-by-default.
- **[#0067](0067.md)** — The popover UI (iOS & macOS): tap image → popover with FT dropdown (default last-used), fitted+centered image, Send (sends the transient preview centered), and the saved-variants list with save/edit. Depends on #0066.
- **[#0068](0068.md)** — Retire the superseded navigation (Send-to-Display button, `DisplaySendPickerView`, manual Custom Size, any now-dead paths incl. the unused `processFitting`). Depends on #0067.

Acceptance criteria:
- Tapping a gallery image opens a popover; there is no separate pushed item screen or display-picker screen.
- The popover shows an FT dropdown defaulting to the last-used display, the fitted+centered image, a Send button, and the saved-variants list.
- Selecting a display builds a fitted, centered preview with no distortion for any aspect ratio; it is transient until saved.
- Editing + saving adds the variant to the saved-variants list; the transient default does not accumulate junk variants.
- The last-used display is remembered across launches.
- **Verified by actually running the app** (iOS and macOS) — not only unit tests and builds. The prior UI shipped "green" but wrong because it was never run; this umbrella is not done until the popover is exercised in the running app (fitted preview visibly un-stretched, dropdown defaulting correctly, transient-vs-saved behaving). Live send to a real FT remains a user/device step.

Out of scope (file as follow-ons if wanted): a pixel-level editor (only "save the fitted preview as a variant" is in scope for "customize"); batch-send to multiple displays.
