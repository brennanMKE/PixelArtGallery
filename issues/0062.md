## Description

Foundation for the display-first flow ([#0061](0061.md)): add a pure, tested helper that, given a source image size `srcW×srcH` and a display size `dispW×dispH`, computes the **aspect-fit** image dimensions (largest `fitW×fitH` with the source's aspect ratio that fits inside the display) and the **centering offset** `(offsetX, offsetY)` to center that image on the display canvas — with the invariant `offsetX + fitW ≤ dispW` and `offsetY + fitH ≤ dispH` (centering can never push the image off the display). Also add an aspect-fit pixelation path so a source is pixelated to the fit dimensions without the current stretching/distortion. No UI, no data-model changes.

## Long Description

Today `PixelationEngine.pixelate(_:targetWidth:targetHeight:)` draws the source into a rect covering the entire `targetWidth×targetHeight` bitmap (`context.draw(image, in: CGRect(0,0,targetW,targetH))`), which **stretches** a non-matching aspect ratio. The display-first flow needs aspect-preserving output. Because the FT wire protocol places a `W×H` image at an `(x,y)` offset on the larger display canvas, the chosen representation is a **fit-sized grid + centering offset** (not baked-in black bars): pixelate at the fit dimensions (which preserves aspect by construction), and let the send offset center it.

This issue is pure/logic only — the coordinator ([#0063](0063.md)) and UI ([#0064](0064.md)) consume it.

### 1. New pure helper — `AspectFit` (new file `PixelArtGalleryKit/Sources/PixelArtGalleryKit/ImageProcessing/AspectFit.swift`)

A caseless enum as a namespace, with a nested result struct (a struct rather than a tuple so #0063 can pass placements around, and so tests get `Equatable` for free). Mark **both** the enum and the nested struct explicitly `nonisolated` (don't rely on the outer annotation propagating — mirror how every type in ImageProcessing carries its own):

```swift
nonisolated public enum AspectFit {
    /// Fit dimensions and centering offset of a source placed on a display canvas.
    nonisolated public struct Placement: Equatable, Sendable {
        public let width: Int      // fitW, 1...displayWidth
        public let height: Int     // fitH, 1...displayHeight
        public let offsetX: Int    // (displayWidth - width) / 2, ≥ 0
        public let offsetY: Int    // (displayHeight - height) / 2, ≥ 0
        public init(width: Int, height: Int, offsetX: Int, offsetY: Int)
    }

    public static func fit(
        sourceWidth: Int, sourceHeight: Int,
        displayWidth: Int, displayHeight: Int
    ) -> Placement
}
```

**Algorithm (all integer, no floating point):**

1. Clamp inputs: `srcW = max(1, sourceWidth)` etc. for all four. The helper is **total** — no throws, no preconditions. Rationale: it will be called from UI/coordinator paths with values ultimately from user/network data; a garbage input yielding a safe 1×1 placement beats a crash, and the engine path (below) still throws for invalid display dims per its existing contract.
2. Pick the bound axis by cross-multiplication (exact, no rounding): if `srcW * dispH >= srcH * dispW` the source is at least as wide as the display (width-bound), else height-bound. Equality means the aspects match exactly; either branch gives fit == display, so fold it into width-bound.
3. Width-bound: `fitW = dispW`, `fitH = max(1, (srcH * dispW) / srcW)`. Height-bound: `fitH = dispH`, `fitW = max(1, (srcW * dispH) / srcH)`. Swift `/` on nonnegative `Int` truncates = **floor**.
4. Offsets: `offsetX = (dispW - fitW) / 2`, `offsetY = (dispH - fitH) / 2` (floor of a nonnegative value).

**Why the invariant `offset + fit ≤ disp` holds structurally** (state this in a doc comment):

- Width-bound branch condition gives `srcH * dispW ≤ srcW * dispH`, so `(srcH * dispW) / srcW ≤ dispH` exactly, and flooring only shrinks it — `fitH ≤ dispH`. The `max(1, …)` clamp is safe because `dispH ≥ 1` after input clamping. Symmetric for the height-bound branch. So `fitW ≤ dispW` and `fitH ≤ dispH` always.
- Then `offsetX + fitW = floor((dispW - fitW)/2) + fitW ≤ (dispW - fitW) + fitW = dispW`, since `floor(n/2) ≤ n` for `n ≥ 0`. No per-case fixups needed; the invariant falls out of flooring both the fit dims and the offsets.
- Overflow: products are `dim × dim` with real-world image dims ≪ 2³²; `Int` is 64-bit on iOS 18/macOS 15. No concern.

**Small-source policy — standard "contain", scale up.** A source smaller than the display is enlarged until one axis fills the display (e.g. 10×5 into 45×35 → 45×22). Rationale: FT displays are tiny (32×32-ish) but sources can be small pixel art too; filling the display maximizes use of scarce LEDs, matches SwiftUI `.aspectRatio(contentMode: .fit)` semantics users already see in the app's previews, matches the umbrella's promise ("pixelated to that display's dimensions"), and keeps the function one rule with no size-comparison branch. Document this in the doc comment so #0063/#0064 don't second-guess it.

### 2. Aspect-fit pixelation path on `PixelationEngine` (edit `PixelationEngine.swift`)

**Chosen shape — return the grid plus the placement** (not grid-only with caller-derived offsets): the source dimensions needed to derive the offset only exist *after* decode + EXIF-orientation, inside the engine. Returning the `Placement` keeps the invariant guaranteed at exactly one code site and hands #0063 everything it needs in one call.

```swift
/// Pixelate to the largest aspect-preserving size that fits the display,
/// returning the fit-sized grid and its centering placement. The grid IS the
/// fit size — letterbox bars are NOT baked in; the caller centers via offset.
public func processFitting(
    imageData: Data, displayWidth: Int, displayHeight: Int
) async throws -> (grid: PixelGrid, placement: AspectFit.Placement)
```

Implementation:

1. Validate `displayWidth > 0, displayHeight > 0` else throw `PixelationError.invalidTargetDimensions(width:height:)` — same contract as `process`. (The pure helper clamps; the engine throws. Both behaviors already exist in this module and serve different callers.)
2. Extract the decode block of `process` (guard `CGImageSourceCreateWithData`, `Self.uprightImage(from:) ?? CGImageSourceCreateImageAtIndex`) into a private `static func decodeUpright(_ imageData: Data) throws -> CGImage`; have `process` call it too. **Do not** implement `processFitting` by calling `process(imageData:…)` — that would decode twice, and reading dims from `CGImageSourceCopyProperties` instead would use pre-orientation dims (a portrait EXIF photo would fit as landscape). Decoding once and reading `cgImage.width/height` off the upright image is both correct and cheaper.
3. `let placement = AspectFit.fit(sourceWidth: cgImage.width, sourceHeight: cgImage.height, displayWidth: displayWidth, displayHeight: displayHeight)`, then `let grid = try pixelate(cgImage, targetWidth: placement.width, targetHeight: placement.height)`. Pixelating at the fit dims is distortion-free by construction — fitW:fitH matches the source aspect (within one pixel of rounding), so the existing stretch-draw becomes a uniform scale. No change to `pixelate` itself.
4. Log via `AppLog.imageProcessor` mirroring `process` (source dims → fit dims @ offset).

No changes to `PixelGrid`/`PixelColor`. Existing `process` behavior is untouched (manual custom-size variants still stretch deliberately).

## Resolution notes

> 🟢 Resolved 2026-07-10 — Aspect-fit math is correct and the `offset + fit ≤ display` invariant holds structurally (verified by reasoning and an exhaustive 16⁴ sweep); `centeringOffset` is the single shared helper `fit` delegates to, both types are `nonisolated`, `processFitting` computes from upright dims with no distortion and leaves `process` unchanged, and all 150 tests plus both platform builds pass.

## Root cause

Not a bug but a gap: `PixelationEngine` had exactly one output path, `process`/`pixelate(_:targetWidth:targetHeight:)`, which draws the source into a rect covering the *entire* target-sized bitmap (`context.draw(image, in: CGRect(0,0,targetW,targetH))`). Any source whose aspect ratio didn't match the requested target dimensions was stretched to fill it. There was no aspect-fit or centering math anywhere in the package (confirmed greenfield by the planner's grep for `aspectRatio|AVMakeRect|contentMode|letterbox|scaleToFit`), so nothing downstream (the coordinator in #0063, the UI in #0064) had a distortion-free way to pixelate a source for display on an FT canvas of a different aspect ratio.

## Fix

Implemented per the `## Plan` with no deviations:

- Added `AspectFit` (new file), a caseless `nonisolated public enum` namespace with a nested `nonisolated public struct Placement: Equatable, Sendable` (`width`, `height`, `offsetX`, `offsetY`). `static func fit(sourceWidth:sourceHeight:displayWidth:displayHeight:) -> Placement` clamps all four inputs to `>= 1`, picks the bound axis by exact integer cross-multiplication (`srcW * dispH >= srcH * dispW` → width-bound, else height-bound), floors the other axis's fit dimension (clamped to `>= 1`), and computes centering offsets via a new shared `nonisolated static func centeringOffset(imageWidth:imageHeight:displayWidth:displayHeight:) -> (x: Int, y: Int)` (each component `= max(0, (display - image) / 2)`). `fit` calls `centeringOffset` rather than duplicating the arithmetic, so `centeringOffset` is the single source of truth #0063 can also call directly for manually-sized (non-fit) grids. Both the enum and `Placement` are marked `nonisolated` explicitly, matching every other type in `ImageProcessing` (`PixelGrid`, `PixelColor`, `PixelationEngine`), since `Package.swift`'s `.defaultIsolation(MainActor.self)` only applies within the main target and would otherwise leave these types actor-isolated for nonisolated test code and #0063's off-main-actor callers (the #0057/#0060 failure mode). Small sources are enlarged to fill the display ("contain", scale up) rather than left at native size, documented in the doc comments along with the invariant proof.
- Extracted the shared decode step out of `PixelationEngine.process` into a private `static func decodeUpright(_ imageData: Data) throws -> CGImage` (EXIF-upright decode via the existing `uprightImage(from:)` thumbnail path, falling back to a raw `CGImageSourceCreateImageAtIndex` decode); `process` now calls it too, with no behavior change.
- Added `public func processFitting(imageData:displayWidth:displayHeight:) async throws -> (grid: PixelGrid, placement: AspectFit.Placement)`: validates `displayWidth > 0, displayHeight > 0` (throwing `PixelationError.invalidTargetDimensions`, matching `process`'s contract), decodes once via `decodeUpright`, computes `AspectFit.fit` from the *upright* `cgImage.width/height` (so EXIF-rotated sources fit correctly), and pixelates at the fit dimensions via the existing `pixelate(_:targetWidth:targetHeight:)` — no changes to `pixelate` or to `process`'s existing (deliberately stretching) behavior for manual custom-size variants.
