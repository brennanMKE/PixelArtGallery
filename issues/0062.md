# 0062 — Aspect-fit + centering math and an aspect-fit pixelation path

| | |
|---|---|
| **Status** | open |
| **Module** | ImageProcessing |
| **Platform** | All |
| **First seen** | 2026-07-10 |

## Description

Foundation for the display-first flow ([#0061](0061.md)): add a pure, tested helper that, given a source image size `srcW×srcH` and a display size `dispW×dispH`, computes the **aspect-fit** image dimensions (largest `fitW×fitH` with the source's aspect ratio that fits inside the display) and the **centering offset** `(offsetX, offsetY)` to center that image on the display canvas — with the invariant `offsetX + fitW ≤ dispW` and `offsetY + fitH ≤ dispH` (centering can never push the image off the display). Also add an aspect-fit pixelation path so a source is pixelated to the fit dimensions without the current stretching/distortion. No UI, no data-model changes.

## Long Description

Today `PixelationEngine.pixelate(_:targetWidth:targetHeight:)` draws the source into a rect covering the entire `targetWidth×targetHeight` bitmap (`context.draw(image, in: CGRect(0,0,targetW,targetH))`), which **stretches** a non-matching aspect ratio. The display-first flow needs aspect-preserving output. Because the FT wire protocol places a `W×H` image at an `(x,y)` offset on the larger display canvas, the chosen representation is a **fit-sized grid + centering offset** (not baked-in black bars): pixelate at the fit dimensions (which preserves aspect by construction), and let the send offset center it.

This issue is pure/logic only — the coordinator ([#0063](0063.md)) and UI ([#0064](0064.md)) consume it.

## Notes

Relevant code:
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/ImageProcessing/PixelationEngine.swift` — `nonisolated public struct PixelationEngine`; `process(imageData:targetWidth:targetHeight:) async throws -> PixelGrid` and `pixelate(_:targetWidth:targetHeight:) throws -> PixelGrid` (the stretch is at the `context.draw(...)` line).
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/ImageProcessing/PixelGrid.swift` — output type, `width`/`height` fixed at construction.
- No existing aspect-fit / `AVMakeRect` / centering logic anywhere in the package (greenfield).

Likely shape (planner to finalize):
- A pure value type / static helper, e.g. `AspectFit.fit(sourceWidth:sourceHeight:displayWidth:displayHeight:) -> (width: Int, height: Int, offsetX: Int, offsetY: Int)`, `nonisolated` and `Sendable`-friendly, with integer results ≥ 1, aspect-preserving "contain" scaling, and centered offsets (`(disp - fit) / 2`). Guarantee the no-overflow invariant explicitly.
- A convenience on `PixelationEngine`, e.g. `processFitting(imageData:displayWidth:displayHeight:)` that computes the fit dims via the helper and pixelates at those dims (reusing the existing `process`), returning both the `PixelGrid` and the centering offset (or the fit dims, letting the caller derive the offset).

Edge cases the tests must cover: landscape source into square display; portrait source into wide display; source aspect exactly matching the display (offset 0,0; fit == display); source smaller than display (still scale up to fit — or decide/return a policy; planner to specify and test); 1×1 and very small displays (no zero/negative dims); rounding (verify `fit + offset ≤ display` holds after integer rounding in the asymmetric cases).

Testing: pure helper is fully unit-testable with Swift Testing (`import Testing`, `@Test`, `#expect`) — no network/CoreGraphics needed for the math. The pixelation path can be tested by feeding a known-aspect source and asserting the output `PixelGrid` dimensions equal the computed fit dims. Mirror existing test style in `PixelArtGalleryKitTests` (e.g. `PixelGridTests`, `FTPaintTargetTests`).
