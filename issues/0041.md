## Description

Hand-typing appcast entries is a proven footgun: a `sparkle:version` that doesn't match the DMG's real `CFBundleVersion`, or a stale marketing version, produces broken "You're up to date" behavior on clients while claiming a newer version exists. Add a script that generates the complete `<item>` XML from the DMG itself so the appcast can never drift from the artifact.

## Acceptance criteria

- `scripts/appcast-item.sh <path-to-dmg>` mounts the DMG, reads `CFBundleShortVersionString` and `CFBundleVersion` from the app bundle inside, computes the byte length, runs Sparkle's `sign_update --account PixelArtGallery` for the EdDSA signature, and prints a ready-to-paste `<item>` with `sparkle:version`, `sparkle:shortVersionString`, `length`, `type`, `sparkle:edSignature`, the download `url` (the site's `downloads/` path from #0040), and an empty `<description>` placeholder for release notes.
- Every attribute comes from the artifact ŌĆö nothing is typed by hand.
- `scripts/release.sh` (#0037) prints this block automatically at the end of a successful run.
- Fails with an actionable error when the Sparkle signing key for the `PixelArtGallery` account isn't in the keychain, or when `sign_update` can't be located in the SPM artifacts directory.

## Notes

- `sign_update` and `generate_keys` live in the Sparkle SPM binary artifacts directory after any build that resolves the package (both the package `.build/artifacts/...` and Xcode DerivedData `SourcePackages/artifacts/...` variants exist ŌĆö probe both).
- `--account PixelArtGallery` must match the account used by `generate_keys` in #0039; the default account name fails with "Signing key not found".
- Depends on #0037 (DMG pipeline), #0039 (keys), #0040 (download URL base).

## Root cause

Feature gap, not a bug: the appcast (#0040) was an empty skeleton and nothing produced `<item>` entries, so any entry would have been hand-typed ŌĆö inviting `sparkle:version` / length / signature drift from the shipped DMG.

## Fix

- New `scripts/appcast-item.sh <path-to-dmg>`:
  - `hdiutil attach -readonly -nobrowse -mountpoint <mktemp -d>` with a `trap detach EXIT`; finds the `.app` at the volume root.
  - Reads `CFBundleShortVersionString` and `CFBundleVersion` via `plutil -extract ŌĆ” raw` from the bundle's `Info.plist`.
  - Byte length via `stat -f%z`.
  - Locates `sign_update` by probing `PixelArtGalleryKit/.build/artifacts/sparkle/Sparkle/bin/sign_update` first, then the DerivedData glob `~/Library/Developer/Xcode/DerivedData/PixelArtGallery-*/SourcePackages/artifacts/sparkle/Sparkle/bin/sign_update`; if neither exists, fails naming both probed paths. Runs it with `--account PixelArtGallery`; on failure the error includes sign_update's output and a hint to run `generate_keys --account PixelArtGallery`.
  - Parses `sparkle:edSignature` and `length` from sign_update's output and hard-fails if sign_update's length disagrees with `stat`'s.
  - Prints a ready-to-paste `<item>` with title, `<link>` to `https://pixelartgallery.sstools.co/changelog.html#vX-Y[-Z]` (marketing version with dots ŌåÆ hyphens, matching changelog.html anchors), RFC 822 `<pubDate>` (UTC, `LC_ALL=C`), `sparkle:version` (CFBundleVersion), `sparkle:shortVersionString`, `sparkle:minimumSystemVersion` 15.0, empty `<description><![CDATA[ ]]></description>` placeholder, and an `<enclosure>` with `url` (`https://pixelartgallery.sstools.co/downloads/<dmg-basename>`), `length`, `type="application/octet-stream"`, `sparkle:edSignature`. Every value is derived from the artifact.
- `scripts/release.sh`: after the "Release complete" summary of a full run, a new "Appcast item (website/appcast.xml)" banner calls `appcast-item.sh "$DMG_PATH"`; the `--dry-run` exit path now prints a hint that no DMG (and hence no appcast item) was produced.

## Gotchas

- The first `sign_update` run from a given binary path triggers a macOS keychain authorization dialog (the EdDSA key item's ACL is per-binary, and `sign_update` is a different executable from the `generate_keys` that created the item). The script appears to hang until Allow is clicked. Click "Always Allow" once per binary path (package `.build` vs DerivedData copies are separate binaries).
- `sign_update` prints `sparkle:edSignature="ŌĆ”" length="ŌĆ”"`; the script parses both and cross-checks the length against `stat -f%z` as a self-consistency guard.
- The changelog anchor derives from the real marketing version: the current `MARKETING_VERSION` is `1.0`, so the link is `#v1-0`, while `changelog.html` currently has `id="v1-0-0"`. Whoever ships 1.0 should align the marketing version (or the changelog anchor) ŌĆö e.g. set `MARKETING_VERSION = 1.0.0` (#0042 owns version sourcing).
- The generated `<title>` uses the marketing version; Sparkle only cares about `sparkle:version` (CFBundleVersion `20260706`, the UTC-date build number injected by release.sh) for update comparisons.
