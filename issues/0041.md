# 0041 — Appcast item generator: derive the Sparkle item from the DMG artifact

| | |
|---|---|
| **Status** | open |
| **Module** | Build / Website |
| **Platform** | macOS |
| **First seen** | 2026-07-06 |

## Description

Hand-typing appcast entries is a proven footgun: a `sparkle:version` that doesn't match the DMG's real `CFBundleVersion`, or a stale marketing version, produces broken "You're up to date" behavior on clients while claiming a newer version exists. Add a script that generates the complete `<item>` XML from the DMG itself so the appcast can never drift from the artifact.

## Acceptance criteria

- `scripts/appcast-item.sh <path-to-dmg>` mounts the DMG, reads `CFBundleShortVersionString` and `CFBundleVersion` from the app bundle inside, computes the byte length, runs Sparkle's `sign_update --account PixelArtGallery` for the EdDSA signature, and prints a ready-to-paste `<item>` with `sparkle:version`, `sparkle:shortVersionString`, `length`, `type`, `sparkle:edSignature`, the download `url` (the site's `downloads/` path from #0040), and an empty `<description>` placeholder for release notes.
- Every attribute comes from the artifact — nothing is typed by hand.
- `scripts/release.sh` (#0037) prints this block automatically at the end of a successful run.
- Fails with an actionable error when the Sparkle signing key for the `PixelArtGallery` account isn't in the keychain, or when `sign_update` can't be located in the SPM artifacts directory.

## Notes

- `sign_update` and `generate_keys` live in the Sparkle SPM binary artifacts directory after any build that resolves the package (both the package `.build/artifacts/...` and Xcode DerivedData `SourcePackages/artifacts/...` variants exist — probe both).
- `--account PixelArtGallery` must match the account used by `generate_keys` in #0039; the default account name fails with "Signing key not found".
- Depends on #0037 (DMG pipeline), #0039 (keys), #0040 (download URL base).
