## Description

Sending a variant to a display is faked. `VariantDetailView.handleSendToDisplay()` just waits 1.5s with `DispatchQueue.main.asyncAfter` and shows a fake success message. There is no FT client — the `FTDisplayClient`/`FTDisplayError` types named in `PixelArtGalleryKit.swift` don't exist, and the `ft-swift` library the PRD references is not a package dependency.

## Expected behavior

A real `FTDisplayClient` sends a variant's pixel data to a stored display's host/port (the FT/UDP image protocol, e.g. via `ft-swift`), reporting genuine progress and success/error. The "Send Now" button drives the real client against the selected persisted display.

## Notes

- Create `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Networking/FTDisplayClient.swift` (+ `FTDisplayError`).
- Add `ft-swift` as a dependency in `PixelArtGalleryKit/Package.swift` (currently no external dependencies), or implement the FT protocol directly.
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/VariantDetailView.swift` — replace simulated `handleSendToDisplay`.
- PRD §3 "Send to FT" and §"Export & Send". Depends on 0011 for real display selection. Log under `FTDiscovery`/a send category.

## Root cause

There was no FT send client at all. `VariantDetailView.handleSendToDisplay()` only
ran `DispatchQueue.main.asyncAfter` for 1.5s and then showed a hard-coded success,
never touching the network. The `FTDisplayClient`/`FTDisplayError` types referenced
in `PixelArtGalleryKit.swift` did not exist, and no FT image was ever transmitted.

## Fix

Added a public `FTDisplayClient` actor (plus public `FTDisplayError` and
`FTSendProgress`). It builds the Flaschen Taschen datagram from a variant's pixel
data — a P6 PPM payload produced by the existing `VariantExporter` (the same tested
`.ppm` path, so 0007 is untouched and there's one encoding source of truth) — and
appends an FT footer (`0x00` + ASCII `x\ny\nz\n`, default `0,0,0` for a full-frame
paint at the origin). It transmits the packet over UDP via `Network.framework`'s
`NWConnection`, fully async, with a bounded connection timeout and per-stage
progress callbacks. The client is an `actor`, so it runs off the package-default
`@MainActor` isolation per the PRD concurrency rules, and logs under category
`FTSend`. `VariantDetailView.handleSendToDisplay()` now reads the SwiftData
`@Model` fields on the main actor, then calls the client from a `Task` against the
selected `FlaschenTaschenDisplay`, surfacing real success/error through the existing
UI.

I implemented the FT protocol **directly on `Network.framework`** rather than adding
the external `ft-swift` SPM dependency, per the issue's guidance, so the build does
not depend on external package resolution in this sandbox. `Package.swift` was left
with no external dependencies.

  `testPacketByteCountMatchesHeaderPlusRGBPlusFooter`, `testPixelOrderingIsRowMajorRGB`,
  `testFooterEncodesOffset`, `testDefaultFooterIsOrigin`, `testMismatchedPixelDataThrows`,
  `testInvalidPortRejected`, `testEmptyHostRejected`). Confirmed they appear in the run output.
- `xcodebuild ... -destination 'platform=macOS' build` — BUILD SUCCEEDED (with code
  signing disabled; the default config requires a Mac Development cert unavailable here).
- `xcodebuild ... -destination 'platform=iOS Simulator,name=iPhone 17 Pro' build` — BUILD SUCCEEDED.

Limitation: the actual UDP transmission in `FTDisplayClient.send` requires a real
(or listening) FT display on the network and cannot be verified headlessly. The unit
tests cover everything that determines the on-the-wire bytes (header, byte count,
pixel ordering, footer) and input validation; the live send path is build-only. A
loopback `NWListener` test was considered but skipped to avoid a non-deterministic,
flaky network-bound test in the suite.
