## Description

The bundled built-in sprites ([#0074](0074.md)) are too small: `scripts/sprites.py` draws each sprite on its tiny native grid (8вЂ“11 px) and blits it **1:1, centered** onto the 45Г—35 canvas, so the art only fills ~25вЂ“30% of the display. Two changes: (1) update `scripts/sprites.py` to **integer-scale each sprite up** (uniform, nearest-neighbor, largest scale that fits 45Г—35) before centering, so the sprites are big and crisp for the original FT's 45Г—35 display; and (2) regenerate the PNGs **and** make the built-in reconcile **update existing built-ins when the bundled art changes** вЂ” because built-ins are non-deletable and matched by name, simply swapping the PNGs would otherwise never reach installs that already have the old sprites.

## Long Description

`render()` in `scripts/sprites.py` currently: `img = 45Г—35 filled with BG; blit native rows 1:1 centered`. A 9Г—9 star lands as a 9Г—9 blob in the middle of 45Г—35. Fix: scale the native grid by `scale = min(45 // w, 35 // h)` (uniform integer, `Image.NEAREST`), then center the `w*scale Г— h*scale` result on the 45Г—35 BG canvas. Keeps pixel edges crisp and dramatically larger (e.g. 11Г—8 invader в†’ Г—4 = 44Г—32; 9Г—9 star в†’ Г—3 = 27Г—27).

The "loaded automatically" requirement is the subtle part. [#0074](0074.md)'s `reconcileBuiltInSpritesIfNeeded()` matches built-ins **by name** and only **inserts missing** ones вЂ” so an install that already has the old sprites (by name) will never pick up the new bigger PNGs. Since built-ins are also **non-deletable**, the user can't force a refresh by deleting them. So the reconcile must additionally **update a built-in in place when the bundled PNG's content differs from the stored one** (compare `contentHash`).

### 1. `scripts/sprites.py` вЂ” integer-scale `render`

Replace `render` with: draw the sprite on its **native** `wГ—h` grid first (filled with `bg`, so transparent cells become BG exactly as today), then upscale by the largest uniform integer that fits, then center on the 45Г—35 canvas:

```python
def render(rows, bg=BG):
    h = len(rows)
    w = max(len(r) for r in rows)
    # Draw the sprite on its native grid (transparent cells -> bg).
    native = Image.new("RGB", (w, h), bg)
    px = native.load()
    for j, row in enumerate(rows):
        for i, ch in enumerate(row):
            col = LEGEND.get(ch, _)
            if col is not None:
                px[i, j] = col
    # Largest uniform integer scale that fits the canvas; nearest keeps edges crisp.
    scale = max(1, min(W // w, H // h))
    scaled = native.resize((w * scale, h * scale), Image.NEAREST)
    img = Image.new("RGB", (W, H), bg)
    img.paste(scaled, ((W - scaled.width) // 2, (H - scaled.height) // 2))
    return img
```

Uniform scale (single `scale` for both axes) в†’ no distortion; `Image.NEAREST` в†’ crisp pixel edges; `max(1, вЂ¦)` guards a hypothetical sprite larger than the canvas. Update the docstring line 4 from "then blitted 1:1 (no scaling) and centered onto a 45x35 canvas" to e.g. "then integer-scaled up (uniform, nearest-neighbor, largest multiple that fits) and centered onto a 45x35 canvas". Nothing else in the script changes (sprite grids, LEGEND, `main` stay).

**Per-manifest-name loop** (order preserved, per-sprite failure isolation preserved):

**The update helper** вЂ” `private func updateBuiltIn(_ item: GalleryItem, with data: Data, hash: String) async throws`:

```swift
let storage = try fileStorageManager()
let oldPath = item.originalImagePath
let newPath = try await storage.save(imageData: data)   // new UUID filename first вЂ” never overwrite in place
var width = 0, height = 0
if let cgImage = try? loadImage(from: data) {            // same ImageIO decode createGalleryItem uses
    width = cgImage.width
    height = cgImage.height
}
item.originalImagePath = newPath
item.originalWidth = width
item.originalHeight = height
item.contentHash = hash
do {
    try modelContext.save()
} catch {
    try? await storage.delete(filename: newPath)         // don't orphan the new file if the model save failed
    throw error
}
try? await storage.delete(filename: oldPath)             // best-effort; a failed delete only orphans ~250 B вЂ” log it
```

Key properties: the **same `GalleryItem` instance** is mutated, so `item.id` is preserved and its `variants` relationship is untouched (variants hold their own pixel data and never reference `originalImagePath`); ordering is save-new в†’ update-model в†’ persist в†’ delete-old, so a crash at any point leaves the store consistent (worst case: one orphaned file, never a dangling `originalImagePath`); the old file is deleted only after the context save succeeds. Holding fetched `GalleryItem`s across `await`s is fine вЂ” everything runs on the main actor.

**Docstring** on `reconcileBuiltInSpritesIfNeeded()`: extend to say it now also updates a built-in in place when the bundled PNG's content hash differs from the stored one (why: built-ins are non-deletable and matched by name, so shipping new art can never reach existing installs otherwise), and that it still never deletes items and never touches user items.

Nothing else changes: `createGalleryItem`, `builtInSpriteData(for:)`, `builtInSpriteNames`, the delete/rename backstops, and the `GalleryListView` trigger all stay as #0074 left them.

## Root cause

`render()` in `scripts/sprites.py` drew each sprite on its tiny native `wГ—h` grid (8вЂ“11 px per side) and blitted those pixels **1:1** onto the 45Г—35 canvas, centered but never scaled вЂ” so a 9Г—9 star occupied a 9Г—9 patch of a 45Г—35 frame (~6% of the area) instead of filling it. Separately, `GalleryCoordinator.reconcileBuiltInSpritesIfNeeded()` (#0074) matched built-ins **by name only** and inserted whatever was missing, but never compared content вЂ” so an install that already had the old small-sprite `GalleryItem`s (by name) would keep them forever, since built-ins are also non-deletable in the UI. Shipping bigger art alone could never reach an existing install.

## Fix

**`scripts/sprites.py`**: `render()` now draws the sprite onto its own native `wГ—h` canvas first (transparent cells still resolve to `bg`, same as before), computes `scale = max(1, min(W // w, H // h))` (uniform integer, so no distortion), upsamples with `Image.NEAREST` (keeps pixel edges crisp вЂ” no blurring/anti-aliasing), and pastes the result centered on the 45Г—35 canvas. Updated the module docstring's "blitted 1:1 (no scaling)" line to describe the new integer-scaling behavior. No other part of the script changed (sprite grids, `LEGEND`, `main` are untouched).

Regenerated all 11 PNGs into `sprites/` via `python3 scripts/sprites.py sprites`, then copied them over `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Resources/DefaultSprites/` вЂ” confirmed byte-identical via `shasum` (see Verification). Output stays 45Г—35; only the art within the canvas changed.

**`GalleryCoordinator.reconcileBuiltInSpritesIfNeeded()`**: return type changed from `-> Int` to `-> BuiltInReconcileResult` (a small `Equatable` struct with `inserted`/`updated` counts; `@discardableResult` kept, so the `GalleryListView.onAppear` call site вЂ” which already discards the return value вЂ” compiles unchanged). The existing-built-ins fetch now keys a `[String: GalleryItem]` dictionary by `originalName` (instead of a `Set<String>` of names), tolerant of a damaged store with duplicate names via `uniquingKeysWith`. For each manifest name: the bundled bytes are now loaded unconditionally (needed for the hash even when the item already exists) and hashed with the **same** shared `Self.contentHash(for:)` helper `createGalleryItem` already uses (no second hashing implementation). No existing item в†’ insert exactly as before. Existing item with a matching hash в†’ no-op (the idempotence path: zero file I/O beyond the bundle read). Existing item with a differing hash в†’ a new private `updateBuiltIn(_:with:hash:)` helper updates it in place: saves the new bytes under a fresh `FileStorageManager` filename (never overwriting in place), decodes the new dimensions via the existing private `loadImage(from:)`, sets `originalImagePath`/`originalWidth`/`originalHeight`/`contentHash` on the **same** `GalleryItem` instance (so `id` and `variants` are untouched), saves the model context, and only then best-effort-deletes the old file. If the model save throws, the newly-saved file is rolled back (deleted) before rethrowing, so nothing is orphaned on a failed save. The re-entrancy guard and per-sprite failure isolation from #0074 are unchanged; the update path's own failures are caught, logged, and skipped per-sprite without touching `currentError` (a background reconcile must not pop the launch-time error alert).

Implementation matches the plan with no deviations.
