# 0075 — Scale up the generated sprites to fill the 45×35 canvas, and make the reconcile update built-ins when their bundled art changes

## Description

The bundled built-in sprites ([#0074](0074.md)) are too small: `scripts/sprites.py` draws each sprite on its tiny native grid (8–11 px) and blits it **1:1, centered** onto the 45×35 canvas, so the art only fills ~25–30% of the display. Two changes: (1) update `scripts/sprites.py` to **integer-scale each sprite up** (uniform, nearest-neighbor, largest scale that fits 45×35) before centering, so the sprites are big and crisp for the original FT's 45×35 display; and (2) regenerate the PNGs **and** make the built-in reconcile **update existing built-ins when the bundled art changes** — because built-ins are non-deletable and matched by name, simply swapping the PNGs would otherwise never reach installs that already have the old sprites.

## Long Description

`render()` in `scripts/sprites.py` currently: `img = 45×35 filled with BG; blit native rows 1:1 centered`. A 9×9 star lands as a 9×9 blob in the middle of 45×35. Fix: scale the native grid by `scale = min(45 // w, 35 // h)` (uniform integer, `Image.NEAREST`), then center the `w*scale × h*scale` result on the 45×35 BG canvas. Keeps pixel edges crisp and dramatically larger (e.g. 11×8 invader → ×4 = 44×32; 9×9 star → ×3 = 27×27).

The "loaded automatically" requirement is the subtle part. [#0074](0074.md)'s `reconcileBuiltInSpritesIfNeeded()` matches built-ins **by name** and only **inserts missing** ones — so an install that already has the old sprites (by name) will never pick up the new bigger PNGs. Since built-ins are also **non-deletable**, the user can't force a refresh by deleting them. So the reconcile must additionally **update a built-in in place when the bundled PNG's content differs from the stored one** (compare `contentHash`).

### 1. `scripts/sprites.py` — integer-scale `render`

Replace `render` with: draw the sprite on its **native** `w×h` grid first (filled with `bg`, so transparent cells become BG exactly as today), then upscale by the largest uniform integer that fits, then center on the 45×35 canvas:

```python
def render(rows, bg=BG):
    h = len(rows)
    w = max(len(r) for r in rows)
    # Draw the sprite on its native grid (transparent cells -> bg).
    native = Image.new("RGB", (w, h), bg)
    px = native.load()
    for j, row in enumerate(rows):
        for i, ch in enumerate(row):
            col = LEGEND.get(ch, _)
            if col is not None:
                px[i, j] = col
    # Largest uniform integer scale that fits the canvas; nearest keeps edges crisp.
    scale = max(1, min(W // w, H // h))
    scaled = native.resize((w * scale, h * scale), Image.NEAREST)
    img = Image.new("RGB", (W, H), bg)
    img.paste(scaled, ((W - scaled.width) // 2, (H - scaled.height) // 2))
    return img
```

Uniform scale (single `scale` for both axes) → no distortion; `Image.NEAREST` → crisp pixel edges; `max(1, …)` guards a hypothetical sprite larger than the canvas. Update the docstring line 4 from "then blitted 1:1 (no scaling) and centered onto a 45x35 canvas" to e.g. "then integer-scaled up (uniform, nearest-neighbor, largest multiple that fits) and centered onto a 45x35 canvas". Nothing else in the script changes (sprite grids, LEGEND, `main` stay).

### 3. Reconcile: update-on-content-change (`GalleryCoordinator.swift`)

**Return type.** Change `reconcileBuiltInSpritesIfNeeded()` from `-> Int` to a small result struct so tests can distinguish inserts from updates (keep `@discardableResult`; the `GalleryListView.onAppear` call site compiles unchanged):

```swift
struct BuiltInReconcileResult: Equatable {
    var inserted = 0
    var updated = 0
}
```

**Fetch items, not names.** Replace the `Set<String>` of names with a dictionary of the fetched built-in **items** keyed by `originalName` (`Dictionary(existing.map { ($0.originalName, $0) }, uniquingKeysWith: { first, _ in first })` — tolerant of a damaged store with duplicate names). Keep the single fetch, the `modelContext` guard, and the `isReconcilingBuiltIns` guard exactly as-is.

**Per-manifest-name loop** (order preserved, per-sprite failure isolation preserved):

1. `displayName = name.capitalized`; load `data = Self.builtInSpriteData(for: name)` **unconditionally** (needed for the hash even when the item exists; 11 × ~250 B is nothing). Missing resource → log, `continue` (as today).
2. `let bundledHash = Self.contentHash(for: data)` — **the same shared helper `createGalleryItem` uses** (GalleryCoordinator.swift:296); no second hashing implementation. It's already a `static func` on the coordinator; no extraction needed.
3. **No item for `displayName`** → insert exactly as today (`createGalleryItem(name:imageData:isBuiltIn: true)`, catch → log + `currentError = nil` + `continue`), `result.inserted += 1`.
4. **Item exists, `item.contentHash == bundledHash`** → no-op (`continue`). This is the idempotence path: matching content means zero file I/O beyond the bundle read, zero writes, zero churn.
5. **Item exists, hashes differ** → update in place via a new private helper, in its own `do/catch` (log + `continue` on failure, never touching `currentError` — nothing in this path sets it), `result.updated += 1`.

**The update helper** — `private func updateBuiltIn(_ item: GalleryItem, with data: Data, hash: String) async throws`:

```swift
let storage = try fileStorageManager()
let oldPath = item.originalImagePath
let newPath = try await storage.save(imageData: data)   // new UUID filename first — never overwrite in place
var width = 0, height = 0
if let cgImage = try? loadImage(from: data) {            // same ImageIO decode createGalleryItem uses
    width = cgImage.width
    height = cgImage.height
}
item.originalImagePath = newPath
item.originalWidth = width
item.originalHeight = height
item.contentHash = hash
do {
    try modelContext.save()
} catch {
    try? await storage.delete(filename: newPath)         // don't orphan the new file if the model save failed
    throw error
}
try? await storage.delete(filename: oldPath)             // best-effort; a failed delete only orphans ~250 B — log it
```

Key properties: the **same `GalleryItem` instance** is mutated, so `item.id` is preserved and its `variants` relationship is untouched (variants hold their own pixel data and never reference `originalImagePath`); ordering is save-new → update-model → persist → delete-old, so a crash at any point leaves the store consistent (worst case: one orphaned file, never a dangling `originalImagePath`); the old file is deleted only after the context save succeeds. Holding fetched `GalleryItem`s across `await`s is fine — everything runs on the main actor.

**Docstring** on `reconcileBuiltInSpritesIfNeeded()`: extend to say it now also updates a built-in in place when the bundled PNG's content hash differs from the stored one (why: built-ins are non-deletable and matched by name, so shipping new art can never reach existing installs otherwise), and that it still never deletes items and never touches user items.

Nothing else changes: `createGalleryItem`, `builtInSpriteData(for:)`, `builtInSpriteNames`, the delete/rename backstops, and the `GalleryListView` trigger all stay as #0074 left them.
