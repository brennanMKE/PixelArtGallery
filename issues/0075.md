# 0075 — Scale up the generated sprites to fill the 45×35 canvas, and make the reconcile update built-ins when their bundled art changes

|---|---|

## Description

The bundled built-in sprites ([#0074](0074.md)) are too small: `scripts/sprites.py` draws each sprite on its tiny native grid (8–11 px) and blits it **1:1, centered** onto the 45×35 canvas, so the art only fills ~25–30% of the display. Two changes: (1) update `scripts/sprites.py` to **integer-scale each sprite up** (uniform, nearest-neighbor, largest scale that fits 45×35) before centering, so the sprites are big and crisp for the original FT's 45×35 display; and (2) regenerate the PNGs **and** make the built-in reconcile **update existing built-ins when the bundled art changes** — because built-ins are non-deletable and matched by name, simply swapping the PNGs would otherwise never reach installs that already have the old sprites.

## Long Description

`render()` in `scripts/sprites.py` currently: `img = 45×35 filled with BG; blit native rows 1:1 centered`. A 9×9 star lands as a 9×9 blob in the middle of 45×35. Fix: scale the native grid by `scale = min(45 // w, 35 // h)` (uniform integer, `Image.NEAREST`), then center the `w*scale × h*scale` result on the 45×35 BG canvas. Keeps pixel edges crisp and dramatically larger (e.g. 11×8 invader → ×4 = 44×32; 9×9 star → ×3 = 27×27).

The "loaded automatically" requirement is the subtle part. [#0074](0074.md)'s `reconcileBuiltInSpritesIfNeeded()` matches built-ins **by name** and only **inserts missing** ones — so an install that already has the old sprites (by name) will never pick up the new bigger PNGs. Since built-ins are also **non-deletable**, the user can't force a refresh by deleting them. So the reconcile must additionally **update a built-in in place when the bundled PNG's content differs from the stored one** (compare `contentHash`).

## Notes

Relevant code:
- `scripts/sprites.py` — `render(rows, bg=BG)` (lines ~44–56): builds the 45×35 canvas and blits native rows 1:1 centered. `W, H = 45, 35`. Uses `PIL`. The sprite dict + `LEGEND` define native grids. Change only `render` (plus a one-line docstring update); the sprite definitions stay.
- `sprites/` (repo root) — generator output, 11 PNGs. Regenerate with `python3 scripts/sprites.py sprites`.
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Resources/DefaultSprites/` — the shipped resource copies (from #0074). These MUST be regenerated too (copy the new `sprites/*.png` here) — this is what the app bundles.
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/ViewModels/GalleryCoordinator.swift` — `reconcileBuiltInSpritesIfNeeded()` (from #0074): fetches existing built-ins by `#Predicate { $0.isBuiltIn == true }`, inserts missing by `capitalized` name via `createGalleryItem(...isBuiltIn: true)`. `createGalleryItem` computes a SHA-256 `contentHash`. `GalleryItem` has `contentHash`, `originalImagePath`, `originalWidth/Height`. `FileStorageManager` has `save`/`load`/`delete`.

Design points (planner to finalize):
- **`sprites.py` scaling**: in `render`, build the native `w×h` sprite image, compute `scale = max(1, min(W // w, H // h))`, `resize((w*scale, h*scale), Image.NEAREST)`, and `paste` centered onto the `W×H` BG canvas. Uniform integer scale (no distortion), nearest-neighbor (crisp). Keep the BG fill and the transparent-cell → BG behavior. Update the module docstring's "blitted 1:1 (no scaling)" line.
- **Regenerate**: run the script into `sprites/`, then copy the 11 PNGs into `Resources/DefaultSprites/`. Both locations must match (the resource copy is what ships). Note the PNG bytes change → their `contentHash` changes → drives the update path below.
- **Reconcile update-on-content-change** in `reconcileBuiltInSpritesIfNeeded()`: for each manifest name, compute the bundled PNG's hash (reuse the same SHA-256 the import path uses). If no built-in with that name exists → insert (as today). If one exists but its stored `contentHash` differs from the bundled hash → **update it in place**: re-save the new bytes via `FileStorageManager` (new filename), delete the old original file (`item.originalImagePath`), and set `originalImagePath`/`originalWidth`/`originalHeight`/`contentHash` to the new values on the SAME `GalleryItem` (preserve its `id`; leave any user-made variants alone — they hold their own pixel data). Still never delete built-ins; still never touch user items. Keep it idempotent (unchanged content → no-op). Factor the hashing so the reconcile and `createGalleryItem` share one implementation. Return counts (inserted/updated) for tests.
- Keep the reconcile's re-entrancy guard and per-sprite failure isolation from #0074.

Testing (Swift Testing, in-memory container + temp storage, mirroring `GalleryCoordinatorTests`/`BuiltInSpritesTests`): reconcile updates an existing built-in whose stored bytes differ from the bundle (assert `contentHash`/dims/`originalImagePath` change, `id` preserved, old file deleted, count reported); reconcile is a no-op when content matches (no churn); still inserts missing; user items untouched; a built-in's user-made variant survives an update. For `sprites.py`, at minimum verify by regeneration + eyeballing dimensions (a tiny Python assertion that output is 45×35 and the sprite bounding box is larger than before is optional). **Verify by running the app**: (a) fresh install → sprites visibly fill the 45×35 much better; (b) simulate an existing install by seeding the OLD sprites, launch the new build, and confirm the reconcile replaces them with the bigger ones automatically.
