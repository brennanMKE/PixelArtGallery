## Description

After #0058 fixed the local-network authorization error, the iOS scan reaches "Browser ready" but still discovers 0 displays. A network scan (`scripts/scan-ft-displays.py --all`) revealed the reason: the FT displays advertise the Bonjour service type **`_flaschen-taschen._udp`**, but the app browses **`_flaschen-taschen._tcp`**. FlaschenTaschen's wire protocol is UDP (port 1337), so `_udp` is the correct type — the app is simply listening on the wrong protocol. Confirmed against a live Mac FT server (192.168.4.26:1337) and an ESP32 FT server (`ft-esp32.local`, 192.168.4.31:1337), both of which resolve immediately when browsing `_flaschen-taschen._udp` and are invisible when browsing `_flaschen-taschen._tcp`.

## Plan

### Confirmed root cause

The app browses a Bonjour service type nothing on the network advertises. Verified against the actual files:

- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Networking/FTDiscoveryService.swift` line 43 defines `static let defaultServiceType = "_flaschen-taschen._tcp"`. This is the **single source of the string in app code** — a repo-wide grep finds no other Swift definition or hardcoded duplicate. It flows into the browser via the `init(serviceType:)` default parameter, and `GalleryCoordinator.scanForDisplays()` (line 577) constructs `FTDiscoveryService()` with that default. The browser uses `NWBrowser.Descriptor.bonjourWithTXTRecord(type: serviceType, domain: nil)` with plain `NWParameters()` (+ `includePeerToPeer`) — the descriptor takes the type as an opaque string, and browsing never opens a transport connection, so switching `_tcp` → `_udp` is **purely a string change**; no descriptor/parameters changes are needed. (The class doc comment on line 23 also names `_flaschen-taschen._tcp` and should be updated to match.)
- `Configuration/Info-iOS.plist` (added in #0058) declares `NSBonjourServices` = `[_flaschen-taschen._tcp]`. iOS validates every browse against this list, so the Swift constant and this plist entry must change **together** — changing only the constant would reintroduce the exact `NoAuth(-65555)` failure #0058 fixed, this time as a type-not-declared denial.
- Meanwhile the live servers (Mac FT at 192.168.4.26:1337, ESP32 `ft-esp32.local` at 192.168.4.31:1337) advertise only `_flaschen-taschen._udp`, per `scripts/scan-ft-displays.py --all` — consistent with FT's wire protocol being UDP on port 1337.

**Platforms:** discovery is cross-platform. The scan is triggered from shared SwiftUI (`UI/DisplayRegistryView.swift` line 192 → `GalleryCoordinator.scanForDisplays()`) with no `#if os(...)` gating, so macOS browses too. However, `Configuration/Info-macOS.plist` declares **no** `NSBonjourServices` (and no `NSLocalNetworkUsageDescription`) today, and the macOS app is not sandboxed (`ENABLE_APP_SANDBOX[sdk=macos*] = NO` in `Configuration/macOS.xcconfig`) — macOS does not gate `NWBrowser` on an `NSBonjourServices` declaration the way iOS does, which is why the macOS browse already reaches the network without it. So the Swift constant change alone fixes macOS; no macOS plist change is required. (Optionally, `NSLocalNetworkUsageDescription` could be added to `Info-macOS.plist` for a friendlier macOS 15 local-network prompt — the same optional companion #0058 deliberately skipped — but keep it out of this fix unless the user asks.)

### Replace `_tcp` with `_udp` — don't keep both

Recommendation: **replace**, in both files, rather than adding `_udp` alongside `_tcp`:

- The user controls both the client and every server (Mac app, ESP32, Apple TV), and all of them advertise `_udp`; FT is fundamentally a UDP protocol, so `_udp` is the semantically correct type, not a workaround.
- The app browses exactly **one** type (one constant, one `NWBrowser`). A `_tcp` entry left in `NSBonjourServices` would be dead weight — the plist only authorizes types the code actually browses; it doesn't make the app discover anything extra. Keeping it invites the next reader to believe the app browses both.
- Keeping the plist as an exact mirror of `defaultServiceType` preserves a simple invariant: *the declared list is precisely what the code browses.* If a `_tcp` browse is ever introduced (it wasn't in the FT ecosystem here), both files change together at that point, guarded by the test below.

### Changes

1. `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Networking/FTDiscoveryService.swift` — line 43: `defaultServiceType = "_flaschen-taschen._tcp"` → `"_flaschen-taschen._udp"`; update the doc comment (line 23) that names the old type.
2. `Configuration/Info-iOS.plist` — the single `NSBonjourServices` array entry `_flaschen-taschen._tcp` → `_flaschen-taschen._udp`. `NSLocalNetworkUsageDescription` stays as-is.
3. **Test:** no existing test references the service-type string (`DiscoveredFTDisplayTests` / `DisplayMergeTests` only mention `FTDiscoveryService` in comments — real-network behavior is untestable in CI). Add a small Swift Testing test (e.g. in `PixelArtGalleryKit/Tests/PixelArtGalleryKitTests/Networking/`) asserting `FTDiscoveryService.defaultServiceType == "_flaschen-taschen._udp"` — it pins the browsed type to the value declared in `NSBonjourServices` so neither can silently drift without a failing test naming the contract. (A test that parses `Configuration/Info-iOS.plist` for true parity would reach outside the package via `#filePath`; optional, not required.)
4. `scripts/scan-ft-displays.py` — **no functional change needed** (diagnostic tool, and `--type` overrides its default), but note it defaults to `FT_SERVICE_TYPE = "_flaschen-taschen._tcp"` (line 42) and its docstring says the app browses `_tcp`; the implementer may flip that default/docstring to `_udp` in the same commit so the tool's no-args run mirrors the fixed app. Cosmetic either way.

## Root cause

`FTDiscoveryService.defaultServiceType` (the single source of the Bonjour type string in app code) was set to `_flaschen-taschen._tcp`, and `Configuration/Info-iOS.plist`'s `NSBonjourServices` array (added in #0058) declared the same `_tcp` type. FlaschenTaschen's wire protocol is UDP on port 1337, and every server on the network (Mac FT app, ESP32) advertises `_flaschen-taschen._udp` — a type distinct from what the app browsed. The browser was correctly authorized (post-#0058) and ran its full duration, but a type nothing advertises will always yield zero results.

## Fix

Followed the plan as written, replacing `_tcp` with `_udp` rather than browsing both:

1. `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Networking/FTDiscoveryService.swift` — `defaultServiceType` changed from `"_flaschen-taschen._tcp"` to `"_flaschen-taschen._udp"`; updated the class doc comment (previously named `_flaschen-taschen._tcp`) to match.
2. `Configuration/Info-iOS.plist` — the `NSBonjourServices` array entry changed from `_flaschen-taschen._tcp` to `_flaschen-taschen._udp`. `NSLocalNetworkUsageDescription` left unchanged. `Info-macOS.plist` was **not** touched, per the plan — macOS is unsandboxed and does not gate `NWBrowser` on `NSBonjourServices`, so the Swift constant change alone fixes macOS.
3. Added `PixelArtGalleryKit/Tests/PixelArtGalleryKitTests/Networking/FTDiscoveryServiceTests.swift` with a Swift Testing test (`defaultServiceTypeIsUDP`) pinning `FTDiscoveryService.defaultServiceType == "_flaschen-taschen._udp"`, matching the `@Suite`/`@Test`/`#expect` style used by `DiscoveredFTDisplayTests` in the same directory. This guards against the constant and the plist entry silently drifting apart again.
4. Optional/cosmetic item from the plan taken: flipped `scripts/scan-ft-displays.py`'s default `FT_SERVICE_TYPE` and docstring/example references from `_tcp` to `_udp`, so the diagnostic tool's no-args run mirrors the fixed app. Left the generic `_tcp`/`_udp` protocol-parsing code in the `--all` enumeration path untouched (unrelated to FT's default type).

No divergence from the plan.

## Notes

Two coordinated changes are needed so the browsed type and the declared Bonjour type match what the servers publish:

- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Networking/FTDiscoveryService.swift` — `defaultServiceType = "_flaschen-taschen._tcp"` must become `_flaschen-taschen._udp` (verify this is the only place the type string is defined).
- `Configuration/Info-iOS.plist` — the `NSBonjourServices` entry added in #0058 must change from `_flaschen-taschen._tcp` to `_flaschen-taschen._udp` (iOS validates browses against this list; browsing a type not declared here fails). If macOS also browses, mirror this in `Configuration/Info-macOS.plist`.

Diagnostic evidence came from `scripts/scan-ft-displays.py` (added this session): `--all` lists every advertised mDNS type and flagged `_flaschen-taschen._udp` as the only FT-ish type on the LAN; browsing that type resolved `ft-esp32` (192.168.4.31:1337) and `Brennan's Mac FT` (192.168.4.26:1337). Consider whether the app should browse both `_udp` and `_tcp` for resilience, or standardize on `_udp` (the correct FT protocol) — the planner should weigh this.
