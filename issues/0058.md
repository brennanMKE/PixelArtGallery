## Description

Scanning the local network for FlaschenTaschen displays from the iPhone app returns 0 results instantly — the browser fails before it can find anything. A Mac app (192.168.4.26:1337) and Apple TV app (192.168.4.99:1337) are both running and advertising `_flaschen-taschen._tcp`, so they should be discoverable. The scan log shows the `NWBrowser` failing with `DNSServiceBrowse failed: NoAuth(-65555)`, which is the iOS local-network-authorization / Bonjour-declaration failure, not an "empty network" result.

## Plan

### Suspected root cause

The iOS app target ships with **no local-network privacy declarations at all**, so iOS denies the Bonjour browse before it ever touches the network — exactly the immediate `DNSServiceBrowse failed: NoAuth(-65555)` in the log. Verified against the actual project files:

- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Networking/FTDiscoveryService.swift` browses `NWBrowser.Descriptor.bonjourWithTXTRecord(type:)` with `defaultServiceType = "_flaschen-taschen._tcp"` (TCP only; no UDP browse). The code is fine — a failed browser finishes the stream, which is why the scan "ends instantly with 0".
- The iOS target's Info.plist is **Xcode-generated**: `Configuration/iOS.xcconfig` sets `GENERATE_INFOPLIST_FILE[sdk=iphone*] = YES` with a handful of `INFOPLIST_KEY_*` entries (macOS instead uses the explicit `Configuration/Info-macOS.plist` via `INFOPLIST_FILE[sdk=macos*]`, per #0039). A repo-wide grep finds **zero** occurrences of `NSLocalNetworkUsageDescription` or `NSBonjourServices` in any xcconfig, plist, or the pbxproj — and the pbxproj's target-level settings don't set `INFOPLIST_FILE`/`GENERATE_INFOPLIST_FILE`, so xcconfig additions will take effect.
- There are no app-target entitlements files (none needed — the multicast entitlement is not required for `NWBrowser` Bonjour browsing once `NSBonjourServices` declares the type).
- The advertised service type couldn't be confirmed from code in this repo (the FT servers live elsewhere), but that's moot for this failure: `NoAuth(-65555)` is a pre-flight authorization denial that happens identically on an empty network. The issue report confirms the servers advertise `_flaschen-taschen._tcp`, which matches the browsed string exactly.

Since iOS 14, `NWBrowser` requires both `NSLocalNetworkUsageDescription` (the permission-prompt string) and the browsed type listed in `NSBonjourServices`; missing either yields precisely this error.

### Changes

`NSBonjourServices` is an **array**, and there is no `INFOPLIST_KEY_*` build setting for array-valued privacy keys — so the iOS target needs a partial Info.plist merged into the generated one (Xcode merges `INFOPLIST_FILE` content with generated keys when `GENERATE_INFOPLIST_FILE = YES`; explicit file keys win).

1. **New file `Configuration/Info-iOS.plist`** containing only:
   - `NSLocalNetworkUsageDescription` (string) — e.g. `Finds Flaschen Taschen displays on your local network so you can send pixel art to them.`
   - `NSBonjourServices` (array of strings) — exactly one entry: `_flaschen-taschen._tcp` (discovery does not browse UDP, so no `_udp` variant is needed; add `_flaschen-taschen._udp` only if a UDP browse is ever introduced).
2. **`Configuration/iOS.xcconfig`** — keep `GENERATE_INFOPLIST_FILE[sdk=iphone*] = YES` and add:
   ```
   INFOPLIST_FILE[sdk=iphone*] = Configuration/Info-iOS.plist
   ```
   (Mirrors the macOS arrangement in `macOS.xcconfig`; update the "iOS keeps the Xcode-generated Info.plist" comment to note the merged partial plist.)
3. **Optional, low-risk companion:** add `NSLocalNetworkUsageDescription` to `Configuration/Info-macOS.plist` too — macOS 15 also shows a local-network prompt and currently falls back to a generic message. Not required to fix this bug; skip if keeping the diff minimal.

No changes to `FTDiscoveryService.swift` or any Swift code.

## Root cause

The iOS app target shipped with no local-network privacy declarations at all. `Configuration/iOS.xcconfig` set `GENERATE_INFOPLIST_FILE[sdk=iphone*] = YES` with only a handful of `INFOPLIST_KEY_*` entries — no `NSLocalNetworkUsageDescription` and no way to express the array-valued `NSBonjourServices` via an `INFOPLIST_KEY_*` setting. `FTDiscoveryService.swift`'s `NWBrowser` browse of `_flaschen-taschen._tcp` therefore hit iOS's local-network authorization pre-flight check and failed immediately with `DNSServiceBrowse failed: NoAuth(-65555)` — identical behavior to an empty network, which is why the scan appeared to "finish instantly with 0 discovered" regardless of what was actually on the LAN. No Swift/networking code was at fault.

## Fix

Followed the plan as written, with no deviations:

1. Added `Configuration/Info-iOS.plist`, a partial Info.plist containing only the two keys that can't be expressed via `INFOPLIST_KEY_*` build settings:
   - `NSLocalNetworkUsageDescription` = "Finds Flaschen Taschen displays on your local network so you can send pixel art to them."
   - `NSBonjourServices` = `["_flaschen-taschen._tcp"]`
2. Updated `Configuration/iOS.xcconfig` to add `INFOPLIST_FILE[sdk=iphone*] = Configuration/Info-iOS.plist` while keeping `GENERATE_INFOPLIST_FILE[sdk=iphone*] = YES` unchanged, and updated the adjacent comment to explain the merge. Xcode merges the partial file's keys into the Xcode-generated plist (confirmed keys win; no conflict).
3. Verified before editing that `Info-macOS.plist` is *not* registered as a pbxproj file reference either (it's only referenced via the `INFOPLIST_FILE[sdk=macos*]` xcconfig path, `Configuration/Build.xcconfig` `#include`s both `macOS.xcconfig` and `iOS.xcconfig`), and that neither app target in `project.pbxproj` sets `INFOPLIST_FILE`/`GENERATE_INFOPLIST_FILE` at the target level — so the xcconfig-level addition takes effect cleanly. `Configuration/Info-iOS.plist` needed no pbxproj registration for the same reason.
4. Skipped the plan's optional companion step (adding `NSLocalNetworkUsageDescription` to `Info-macOS.plist` for the macOS local-network prompt) to keep the diff minimal and scoped to the reported iOS bug, per the plan's own note that it's optional.

No changes to `FTDiscoveryService.swift` or any other Swift code.

## Notes

`NoAuth(-65555)` on `DNSServiceBrowse` is the classic iOS 14+ symptom of a missing/mismatched Bonjour configuration in the app's `Info.plist`, not a networking-code bug. iOS refuses the browse before it starts when:

- `NSLocalNetworkUsageDescription` is absent (no permission string → no prompt → browse denied), and/or
- the browsed service type is not declared in `NSBonjourServices`. The app browses `_flaschen-taschen._tcp`, so `NSBonjourServices` must include `_flaschen-taschen._tcp` (both `_tcp` and `_udp` variants if the discovery also browses UDP).

Because the failure is immediate and identical regardless of what's on the network, this points at the app-target Info.plist / entitlements rather than `FTDisplayBrowser`/`NWBrowser` code. Worth checking whether these keys exist and are applied to the **iOS** app target specifically (they may be present for macOS but missing for iOS, or the service name may be misspelled vs. what the Mac/tvOS apps advertise).

Relevant code: the mDNS discovery in `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Networking` (the `NWBrowser` setup that logs "Browsing …" and "Browser failed"), plus the `PixelArtGallery` app-target `Info.plist` and entitlements.
