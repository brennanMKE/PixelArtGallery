## Description

On macOS, FT **display management** (scan / add / edit / delete displays, in `DisplayRegistryView`) is only reachable through **Settings** (Pixel Art Gallery ▸ Settings…, ⌘,). There is no visible affordance in the main window, so users won't discover it. Add a **gear button to the macOS toolbar** that opens Settings, making display management obvious. (On iOS this is already covered — the bottom bar has a Settings gear, #0071.)

## Plan

**Root cause (confirmed):** in `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/GalleryListView.swift`, the `#if os(macOS)` toolbar block (lines ~158–184) contains exactly two items — `+` at `.primaryAction` and the Sort menu at `.secondaryAction` — and its comment states macOS "never had a Settings button here since it reaches SettingsView through the app's Settings scene (⌘,)". ⌘, is the only path to `DisplayRegistryView` (display management) on macOS, which is undiscoverable.

**Preconditions verified:**
- `PixelArtGallery/PixelArtGalleryApp.swift` lines 95–100 declare the macOS `Settings { SettingsView() }` scene (with its own `.modelContainer`); `SettingsView` (in `PixelArtGalleryKit/.../UI/SettingsView.swift`) hosts `DisplayRegistryView`. `SettingsLink` opens exactly this scene natively — **no changes to the app file**.
- The package targets macOS 15 (`Package.swift`: `.macOS(.v15)`); `SettingsLink` is macOS 14+, so **no availability guard** is needed.
- `SettingsLink` conforms to `View` and is documented for use in toolbars/menus, so it can sit directly inside a `ToolbarItem` in the toolbar builder. Fallback only if it somehow fails to compile/behave there: a `Button` invoking the Settings window (e.g. via `openSettings` from `@Environment(\.openSettings)`, macOS 14+) — but **prefer `SettingsLink`**; do not use the legacy `NSApp.sendAction` selector hack.

### Change (one file)

`PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/GalleryListView.swift`, inside the existing `#if os(macOS)` `.toolbar { ... }` block — add a third item **after** the Sort `ToolbarItem`:

```swift
// Settings gear (#0081) — opens the app's Settings scene (⌘, still
// works), where FT display management lives (#0054). SettingsLink is
// the native macOS 14+ way to open the Settings scene from a view.
ToolbarItem(placement: .secondaryAction) {
    SettingsLink {
        Label("Settings", systemImage: "gearshape")
    }
}
```

- **Placement — `.secondaryAction`, declared after Sort:** on this macOS toolbar `.primaryAction` renders `+` at the trailing edge and `.secondaryAction` renders Sort just inboard of it (see #0080's screenshot). The gear joins Sort in the secondary group, reading as `Sort · gear · +` — the prominent `+` stays the primary action and the gear reads as ancillary chrome, which matches macOS conventions (settings affordances are never the primary toolbar action). Do not use `.primaryAction` (would compete with `+`) or a `.navigation`/leading slot (nothing else is leading; a lone gear at the far left over the banner would look stray).
- **Label — standard icon-only gear:** `Label("Settings", systemImage: "gearshape")`. macOS toolbars render the icon; the "Settings" text still serves the tooltip, the overflow (») menu, and accessibility, so no explicit `labelStyle` is needed. A "Displays"-titled item was considered for discoverability, but the button opens the whole Settings window (not a Displays-only panel), so titling it "Displays" would over-promise; the universally understood gear that lands the user one glance away from `DisplayRegistryView` is the least surprising v1. Keep the comment above the toolbar (lines ~159–163) accurate: update its "never had a Settings button here" sentence to note the gear was added by #0081.
- **Keep ⌘, working:** the `Settings` scene in `PixelArtGalleryApp.swift` is untouched; `SettingsLink` merely opens it. Nothing else in the app file changes.
- **iOS untouched:** the addition is entirely inside the existing `#if os(macOS)` block. iOS keeps its bottom-bar gear (`GalleryBottomBar`, #0071) and its `#if os(iOS)` Settings sheet — do not add a top-toolbar gear on iOS.
- **Coexists with #0080:** that paired issue owns toolbar legibility (banner bleeding under the title bar, transparent toolbar). This issue only adds the item; whichever of the two lands second inherits the other's toolbar state cleanly since both edit disjoint aspects (item list vs. background/safe-area treatment). Do not attempt any legibility styling here.

## Notes

Relevant code:
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/GalleryListView.swift` — the `#if os(macOS)` toolbar currently holds only `+` (`.primaryAction`) and Sort (`.secondaryAction`); a comment there notes macOS reaches Settings via ⌘, and "never had a Settings button here." That's the discoverability gap — add a gear item.
- `PixelArtGallery/PixelArtGalleryApp.swift` — the macOS `Settings { SettingsView() }` scene (⌘,). `SettingsView` hosts `DisplayRegistryView` (display management, #0054).
- Mechanism: **`SettingsLink`** (SwiftUI, macOS 14+; the app targets macOS 15) opens the Settings scene natively and can be used directly as a toolbar item — e.g. `SettingsLink { Label("Settings", systemImage: "gearshape") }`. Preferred over the older `NSApp.sendAction(Selector(("showSettingsWindow:")))` hack.

Design points (planner to finalize — macOS-only; do NOT change iOS, which already has the gear in its bottom bar):
- Add a **gear** toolbar item to the macOS `#if os(macOS)` toolbar in `GalleryListView` using `SettingsLink { Label("Settings", systemImage: "gearshape") }` (or a `Button` that triggers the Settings window if `SettingsLink` doesn't fit the placement). Keep ⌘, working (the Settings scene stays).
- Placement/legibility: it joins the macOS toolbar alongside `+` and Sort — coordinate with [#0080](0080.md) (which fixes that toolbar overlapping/illegible against the banner) so the gear is clearly visible too. Consider whether a short label ("Displays"/"Settings") helps discoverability vs an icon-only gear.
- Optional: since the primary reason to open Settings is display management, consider whether the button should read as "Displays" — but keeping it a standard Settings gear (that opens the Settings window where displays live) is the simplest, least surprising option; planner's call.

Verification: macOS `xcodebuild` build; and **launch the macOS app** to confirm the gear appears in the toolbar and clicking it opens Settings (→ display management). iOS unchanged (bottom-bar gear still works).
