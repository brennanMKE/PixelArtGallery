# 0077 — Grayscale send mode + an expandable "Advanced" send section (x/y offset + grayscale toggle)

| | |
|---|---|
| **Status** | open |
| **Module** | Networking / UI |

## Description

Add a **grayscale** send option: instead of sending the variant's colors, convert each pixel to a grayscale luminance value (`gray = 0.299·R + 0.587·G + 0.114·B`, sent as `R=G=B=gray`, alpha preserved) so the FT display renders it as grayscale — black stays black (transparent on FT layers 1–15) and gray shades drive the display's color-blend-over-time effect. Surface it in the send UI by adding an **"Advanced"** section **below the Send button** that expands to reveal the **X/Y offset** controls plus a **Grayscale** toggle.

## Long Description

Reference: ft-swift's `grayscale` demo (`~/Developer/FlaschenTaschen/ft-swift/Sources/grayscale/Grayscale.swift`) converts hex colors to gray via the standard luminance formula `0.299·R + 0.587·G + 0.114·B` and sends them over the **same** P6 PPM + `#FT: x y z` wire format the app's `FTDisplayClient` already uses. So grayscale is **not** a new protocol — it's a per-pixel transform applied to the grid before the existing send path.

**Do NOT add the `ft-swift` package as a dependency.** The other project you referenced does `.package(path: "../../ft-swift")`, but that's wrong for this app: (a) `grayscale` is an *executable* demo target, not an importable library; (b) a local relative path won't resolve for other machines/CI; (c) this app has its own `FTDisplayClient` that already emits identical PPM; (d) the reusable part is a ~5-line luminance transform. Reimplement the transform here (citing the ft-swift formula), no new dependency.

Design points (planner to finalize):
- **Grayscale transform (pure, testable)**: a `nonisolated` helper `grayscale(_ rgba: Data) -> Data` (or on `PixelGrid`/`PixelColor`) that maps each pixel `(r,g,b,a)` → `(y,y,y,a)` where `y = round(0.299·r + 0.587·g + 0.114·b)`. Black `(0,0,0)` → `(0,0,0)` (stays transparent). Preserve alpha. Operate on the RGBA8888 buffer directly for speed. Cite ft-swift's `Grayscale.swift` luminance formula in a comment.
- **Send wiring**: apply the transform to the grid when grayscale is enabled — cleanest is at payload-build time in `VariantDetailView`/`FTSendController` (`payload.pixelGridData = grayscale(variant.pixelGridData)` when the toggle is on), so the continuous loop sends the transformed frames unchanged. Decide whether to thread a `grayscale: Bool` through `FTSendPayload`/`send(...)` vs transforming the data before constructing the payload — recommend transforming at payload build (keeps `FTDisplayClient` unaware of grayscale). The stop-clear (all-black) path is unaffected (black is already black).
- **"Advanced" disclosure UI** in `VariantDetailView`: a `DisclosureGroup("Advanced")` placed **below** the Send button, collapsed by default, containing the **X Offset** and **Y Offset** controls (moved from their current position above Send) and a **Grayscale** `Toggle`. Decide Layer's placement — recommend keeping Layer as a primary control near Send (it's commonly adjusted) and moving only X/Y + grayscale into Advanced, OR grouping all fine-tuning under Advanced; pick one and justify against the user's explicit ask ("offset values for x and y as well as a checkbox for grayscale" in Advanced). Keep the live-adjust-during-send behavior (#0057) for the offsets.
- **Grayscale state**: a transient per-send `@State grayscale: Bool` (like the offset/layer transient state), not persisted, for v1. If the user later wants it remembered per-variant/display, that's a follow-on (no schema change now).
- **Preview (optional, planner's call)**: optionally reflect grayscale in the on-screen Preview so what-you-see matches what-you-send; if it complicates things, skip for v1 and note it — the transform on the wire is the requirement.
- **Popover**: this issue targets `VariantDetailView` (matches the referenced screen + where the offset controls live). Whether the minimal `GallerySendPopoverView` also gets a grayscale toggle is a follow-on decision, not v1.
- Shared iOS/macOS; **Swift Testing** for the transform.

Testing (Swift Testing): the grayscale transform — known RGB pixels map to the expected gray (e.g. pure red 255,0,0 → 76; white → 255; black → 0), alpha preserved, black stays black; buffer length unchanged; idempotent on already-gray input. UI wiring (the DisclosureGroup, toggle, moved offset controls) isn't unit-testable. **Verify by running the app**: the Advanced section expands to show X/Y offset + Grayscale; toggling Grayscale and sending transforms the frames (the actual blend-over-time effect is FT-hardware behavior, confirmed on a real display — user-run). Confirm normal (non-grayscale) send is unchanged when the toggle is off.
