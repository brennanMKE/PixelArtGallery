## Description

Add a **grayscale** send option: instead of sending the variant's colors, convert each pixel to a grayscale luminance value (`gray = 0.299·R + 0.587·G + 0.114·B`, sent as `R=G=B=gray`, alpha preserved) so the FT display renders it as grayscale — black stays black (transparent on FT layers 1–15) and gray shades drive the display's color-blend-over-time effect. Surface it in the send UI by adding an **"Advanced"** section **below the Send button** that expands to reveal the **X/Y offset** controls plus a **Grayscale** toggle.

## Long Description

Reference: ft-swift's `grayscale` demo (`~/Developer/FlaschenTaschen/ft-swift/Sources/grayscale/Grayscale.swift`) converts hex colors to gray via the standard luminance formula `0.299·R + 0.587·G + 0.114·B` and sends them over the **same** P6 PPM + `#FT: x y z` wire format the app's `FTDisplayClient` already uses. So grayscale is **not** a new protocol — it's a per-pixel transform applied to the grid before the existing send path.

**Do NOT add the `ft-swift` package as a dependency.** The other project you referenced does `.package(path: "../../ft-swift")`, but that's wrong for this app: (a) `grayscale` is an *executable* demo target, not an importable library; (b) a local relative path won't resolve for other machines/CI; (c) this app has its own `FTDisplayClient` that already emits identical PPM; (d) the reusable part is a ~5-line luminance transform. Reimplement the transform here (citing the ft-swift formula), no new dependency.

## Root cause

Gap, not a bug: the send path (`VariantDetailView.handleSendToDisplay()`) only ever pushed a variant's original RGBA8888 pixel data to the FT display. There was no per-pixel grayscale transform and no UI affordance to request one, so a user wanting the display's color-blend-over-time grayscale effect had no way to send gray frames — and the send section (Display / Layer / X Offset / Y Offset / Send) had no place to add a new toggle without further crowding an already-dense primary control group.

## Fix

Implemented per the plan with no material deviations:

1. **`PixelGrid.grayscale(rgba8888:)`** (`ImageProcessing/PixelGrid.swift`) — a `nonisolated public static func` that walks an RGBA8888 `Data` buffer in strides of 4, computes `y = UInt8((0.299·r + 0.587·g + 0.114·b).rounded())`, writes `y` to r/g/b, and leaves alpha untouched. Rounds (not truncates) per the plan's spec, citing ft-swift's `Grayscale.swift` Rec.601 formula and the round-vs-truncate rationale (idempotence; pure green -> 150 vs ft-swift's 149) directly in the doc comment. Empty `Data` returns empty `Data`; buffer length is unchanged; no new package dependency.
2. **Send wiring** (`VariantDetailView.handleSendToDisplay()`) — the `FTSendPayload`'s `pixelGridData` is now `sendGrayscale ? PixelGrid.grayscale(rgba8888: variant.pixelGridData) : variant.pixelGridData`, computed once when the payload is built (send start). `FTSendPayload`, `FTSendController`, and `FTDisplayClient` are unchanged — no new field, no signature change — matching the #0057 contract that pixel data is frozen at send start while only layer/x/y are read live per frame. The stop-clear path (`FTDisplayClient.sendClearFrame(to:)`) builds its own all-zero buffer and never passes through the transform, so it's unaffected.
3. **UI** (`VariantDetailView`) — added `@State private var sendGrayscale = false` and `@State private var isAdvancedExpanded = false`. Reorganized `sendSection`'s non-empty branch to: Display Picker -> Layer Stepper (kept as a primary control, unchanged binding/disabled rule) -> Send/Stop Button -> `DisclosureGroup("Advanced", isExpanded: $isAdvancedExpanded)` (collapsed by default) containing the X Offset Stepper and Y Offset Stepper (moved verbatim — same `$sendOffsetX`/`$sendOffsetY`, same `offsetXRange`/`offsetYRange`, same `.disabled(selectedDisplay == nil)`, so they remain live-editable during a send per #0057) plus a new `Toggle("Grayscale", isOn: $sendGrayscale)` with `.disabled(selectedDisplay == nil || isSending)` — disabled mid-send because the payload is frozen at start, so a toggle flip wouldn't affect the frames already in flight. `seedSendDefaults()`/`SendDefaultsSeed` untouched; grayscale has no stored per-display default in v1. Section order preserved: send section -> banners -> Preview -> Export.
4. **Preview reflection of grayscale is deferred** (per plan step 4) — the on-screen `PixelGridView`/`PixelGridViewModel` Preview does not reflect the toggle; this is a documented v1 gap, not a bug. A follow-on could thread the flag into the preview's rendering path, or accept a `.saturation(0)` approximation (rejected here since it uses different luminance weights than Rec.601 and wouldn't be true WYSIWYG). Same deferral applies to `GallerySendPopoverView`, which this issue does not touch.

No plan divergences.

## Gotchas

- Grayscale is frozen at send start like the rest of the payload (endpoint, dimensions, pixel data) — toggling it mid-send has no effect on the current send; the toggle is disabled while `isSending` to make that explicit instead of silently doing nothing.

## Plan

### 1. Pure grayscale transform — `PixelGrid.grayscale(rgba8888:)` (ImageProcessing)

Add a static helper to `PixelArtGalleryKit/Sources/PixelArtGalleryKit/ImageProcessing/PixelGrid.swift` (the file is already `nonisolated`, so the helper is callable from nonisolated Swift Testing tests and from `handleSendToDisplay` without hops):

```swift
/// Convert an RGBA8888 buffer to grayscale in place-of: each (r,g,b,a) pixel
/// becomes (y,y,y,a) where y = round(0.299·r + 0.587·g + 0.114·b) — the
/// Rec.601 luminance formula used by ft-swift's grayscale demo
/// (Sources/grayscale/Grayscale.swift). Alpha is preserved; black stays
/// black (transparent on FT layers 1–15). Buffer length is unchanged.
public static func grayscale(rgba8888 data: Data) -> Data
```

Implementation: copy to `[UInt8]`, walk in strides of 4, compute `let y = UInt8((0.299 * Double(bytes[i]) + 0.587 * Double(bytes[i+1]) + 0.114 * Double(bytes[i+2])).rounded())`, write `y` to r/g/b, leave `bytes[i+3]` untouched, return `Data(bytes)`. Any trailing remainder shorter than 4 bytes (shouldn't occur for valid grids) is copied through unchanged; empty `Data` returns empty `Data`. No `PixelGrid` round-trip — operate on the raw buffer for speed.

**Rounding, deliberately (differs from ft-swift by one code path):** ft-swift uses a truncating `UInt8(...)` cast (pure green → 149). This plan uses `.rounded()` per this issue's spec (pure green → **150**). Rounding is also what makes the transform **idempotent**: for an already-gray pixel, `0.299y + 0.587y + 0.114y` can land a hair under `y` in floating point, and truncation would drift it down by 1 on re-application; rounding maps it back to exactly `y`. Cite the ft-swift formula in the doc comment (as above), noting the round-vs-truncate choice.

The transform is data-only, so `UInt8(...)` overflow is impossible (max sum is exactly 255.0 for white — the coefficients sum to 1).

### 2. Send wiring — transform at payload-build time (`VariantDetailView.handleSendToDisplay`)

`FTSendPayload` (`Networking/FTSendPayload.swift`) captures `pixelGridData` once at send start, and `FTSendController` pushes that same buffer every frame. So the entire wiring is one expression in `handleSendToDisplay()`:

```swift
let payload = FTSendPayload(
    host: display.host,
    port: display.port,
    width: variant.targetWidth,
    height: variant.targetHeight,
    pixelGridData: sendGrayscale
        ? PixelGrid.grayscale(rgba8888: variant.pixelGridData)
        : variant.pixelGridData,
    scaleFactor: variant.scaleFactor
)
```

- `FTSendPayload`, `FTSendController`, and `FTDisplayClient` stay completely unaware of grayscale — no new field, no signature change.
- This matches the #0057 contract: endpoint + pixel data are frozen at send start (only layer/x/y are read live per frame). Grayscale is a static transform of the frozen data, so applying it once at start is correct.
- The stop-clear path is unaffected: `FTDisplayClient.sendClearFrame(to:)` builds its own all-zero buffer (`Data(count: w*h*4)`), which never passes through the transform — and grayscale(black) is black anyway.

### 3. UI — `DisclosureGroup("Advanced")` below the Send button (`VariantDetailView`)

New state (transient, like the existing layer/offset state — **no persistence, no schema change** for v1):

```swift
@State private var sendGrayscale = false
@State private var isAdvancedExpanded = false
```

Reorganize `sendSection`'s non-empty branch to this order:

1. Display `Picker` (unchanged, locked while sending)
2. **Layer** `Stepper` (stays where it is — see decision below)
3. Send/Stop `Button` (unchanged)
4. **`DisclosureGroup("Advanced", isExpanded: $isAdvancedExpanded)`** — collapsed by default — containing:
   - **X Offset** `Stepper` (moved here verbatim: same `$sendOffsetX`, `offsetXRange`, `.disabled(selectedDisplay == nil)`)
   - **Y Offset** `Stepper` (same, `$sendOffsetY` / `offsetYRange`)
   - **Grayscale** `Toggle("Grayscale", isOn: $sendGrayscale)` with `.disabled(selectedDisplay == nil || isSending)`

**Layer decision: keep Layer as a primary control next to Send; move only X/Y + Grayscale into Advanced.** This matches the user's literal ask — "offset values for x and y as well as a checkbox for grayscale" in the Advanced section, with no mention of Layer — and Layer is the routinely adjusted control (which FT z-plane you're painting), while x/y nudges and grayscale are fine-tuning.

Behavior preserved/clarified:

- **X/Y stay live during a send** (#0057): they keep binding to the same `@State` the `frameOffset` closure reads each frame; moving the steppers inside the DisclosureGroup changes layout only. Do not disable them while sending.
- **Grayscale is disabled while sending**: the payload's pixel data is frozen at send start, so a mid-send toggle would silently do nothing until the next send. Disabling it (like the display picker) makes the frozen-at-start contract visible instead of lying. To change it, stop and re-send.
- `seedSendDefaults()` / `SendDefaultsSeed` are untouched — grayscale has no stored per-display default (v1), so it is not seeded and simply keeps its `@State` value (false on view init).
- Section order stays: send section → banners → Preview → Export.

Skip for v1; note it in `## Fix` as a follow-on candidate ("preview honors the grayscale toggle"). Same call for `GallerySendPopoverView` — per this issue's notes, the popover toggle is explicitly a follow-on.
