## Description

The PRD lists "Save to file or Photos app (iOS only)" as an export destination. On iOS there is no file-system save panel; `ExportPickerView.showSavePanel()` just sets `isShowingSavePanel = true` with a `// TODO` and goes nowhere. There is no path to save an exported image into the Photos library, even though the Info.plist permission key is already configured.

## Expected behavior

On iOS, the user can export a variant's rendered image directly into the Photos library (and/or via the Files document picker). The save flow requests authorization and reports success/failure. The `NSPhotoLibraryAddUsageDescription` string already present in `Configuration/iOS.xcconfig` is used.

## Notes

- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/ExportPickerView.swift` — the `#else` (iOS) branch of `showSavePanel()`.
- Use `PHPhotoLibrary` to add the exported PNG/HEIC; pair with the real exporter from 0007.
- `Configuration/iOS.xcconfig` — `INFOPLIST_KEY_NSPhotoLibraryAddUsageDescription` already set.
- Depends on 0007.

## Root cause

The iOS `#else` branch of `ExportPickerView.showSavePanel()` was a dead `// TODO`
that set `isShowingSavePanel = true` and was never observed by anything, so iOS
had no export destination at all. The macOS path relied on an `NSSavePanel`-chosen
`selectedFileURL` that iOS could never populate, and the only `Export` button was
gated on that URL — making it permanently disabled on iOS. The real exporter from
0007 was in place but unreachable from any iOS save flow.

## Fix

Gave iOS a real destination without touching the 0007 encoding or the macOS path.
Added `PhotoLibrarySaver` (Export), which requests add-only `PHPhotoLibrary`
authorization and adds raster exports (PNG/HEIC) via `PHAssetCreationRequest`; its
pure `canSaveToPhotos(_:)` eligibility check rejects non-raster formats (PPM/JSON)
before any authorization request. Added `DocumentExporterView` (UI), a
`UIDocumentPickerViewController` representable that exports the exporter's temp file
into the Files app for any format. Reworked `ExportPickerView`'s iOS branch to offer
"Save to Photos" (raster only) and "Save to Files": both encode the variant to a
temp file off the main actor via `VariantExporter`, then route to the chosen
destination and report genuine success/failure through the existing success/error UI.
All iOS-specific state and methods, and the macOS `selectedFileURL` / save-panel /
`performExport` path, are guarded with `#if os(...)` so each platform compiles only
its own flow.

## Gotchas

- `.buttonStyle(cond ? .bordered : .borderedProminent)` does not type-check —
  the two styles are different concrete types and the ternary can't unify them.
  Use an `if/else` returning the same `Button` with each style applied instead.
- The package builds with `MainActor` default isolation, so `PhotoLibrarySaver`
  is explicitly `nonisolated` and operates on value types (URL/format), keeping the
  encode work on a detached task off the main actor.
