# 0073 — Seed the gallery with bundled default sprites on first launch (empty gallery)

| | |

## Description

Ship the 11 generated pixel-art sprites (`sprites/*.png`, 45×35, from `scripts/sprites.py`) bundled with the app and **seed them into the gallery on first launch when there are 0 images**, so new users start with content instead of an empty gallery. The sprites are ordinary gallery items — the user can add their own images and **delete** any/all of the seeded sprites. Seeding must be **once-only** (guarded by a persisted flag), so deleting the sprites does not cause them to reappear on the next launch.

## Long Description

The sprites are 45×35 — exactly a common FT display size — so they're ideal starting points: pick a display and the fitted preview is ~1:1. There are 11: coin, frog, ghost, heart, invader, mushroom, pacman, robot, ship, skull, star. They need to be (a) bundled as app/package resources and (b) imported as `GalleryItem`s through the existing import pipeline on first run.

## Plan

Verified facts from the code (2026-07-11): `PixelArtGalleryKit/Package.swift` declares **no resources today** and nothing in the package uses `Bundle.module` yet (grep confirms zero hits) — this issue introduces both. The import pipeline is `GalleryCoordinator.createGalleryItem(name:imageData:)` (`ViewModels/GalleryCoordinator.swift:126`): SHA-256 `contentHash` → duplicate check → `FileStorageManager.save(imageData:) -> filename` → decode real pixel dimensions via ImageIO → insert `GalleryItem(originalImagePath:originalName:originalWidth:originalHeight:contentHash:)` → save. The once-only precedent is `seedDefaultDisplayIfNeeded()` (line 591), invoked from `GalleryListView.onAppear` (line 181) right after `coordinator.configure(modelContext:)`. The coordinator already takes an injectable `UserDefaults` (`defaults`, default `.standard`) added for #0066's `lastUsedFTDisplayID` key — the seed flag goes in the same store. All 11 PNGs in `sprites/` are confirmed 45×35.

### 1. Resource bundling (`Package.swift` + folder)

- Create `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Resources/DefaultSprites/` and copy the 11 PNGs from `sprites/` into it (the repo-root `sprites/` folder stays as the generator output; the shipped copy is the resource).
- In `Package.swift`, add to the `PixelArtGalleryKit` target:

  ```swift
  resources: [
      .copy("Resources/DefaultSprites"),
  ],
  ```

  **`.copy`, not `.process`** — `.process` may re-encode/optimize PNGs per platform; `.copy` ships the exact bytes, which matters for pixel art (byte-identical resources also keep `contentHash` stable across builds/platforms). Copying the *directory* preserves the `DefaultSprites/` folder name, so lookup is `Bundle.module.url(forResource: name, withExtension: "png", subdirectory: "DefaultSprites")`.
- SwiftPM synthesizes `Bundle.module` once the target has resources; it resolves inside the app (the resource bundle `PixelArtGalleryKit_PixelArtGalleryKit.bundle` is embedded by Xcode) **and** in `swift test` (SwiftPM stages the bundle next to the test runner; the seeding code itself lives in the kit and is exercised via `@testable import`, so tests hit the same `Bundle.module`). A dedicated test (below) asserts all 11 resources resolve, so a bundling regression fails fast.
- **Manifest, not directory enumeration**: declare a static ordered array on the coordinator (or a small `DefaultSprites` enum/struct next to it):

  ```swift
  static let defaultSpriteNames = [
      "coin", "frog", "ghost", "heart", "invader",
      "mushroom", "pacman", "robot", "ship", "skull", "star",
  ]
  ```

  Explicit and stable-ordered; directory enumeration would give filesystem-dependent order and silently absorb a missing file. Friendly name = `name.capitalized` (`coin` → "Coin"; matches the Description's example — no per-sprite display-name table needed).

### 2. `seedDefaultSpritesIfNeeded()` on `GalleryCoordinator`

New method mirroring `seedDefaultDisplayIfNeeded()`'s shape, but `async` (the import path awaits the `FileStorageManager` actor):

- Add `static let didSeedDefaultSpritesKey = "didSeedDefaultSprites"` alongside `lastUsedDisplayIDKey`; persist via the already-injected `defaults` (`UserDefaults`, not `@AppStorage` — same rationale documented at line 902: `@AppStorage` only works inside SwiftUI views, and the injected store keeps tests isolated).
- **Predicate**: seed only if `defaults.bool(forKey: didSeedDefaultSpritesKey) == false` **and** `modelContext.fetchCount(FetchDescriptor<GalleryItem>()) == 0`; then **always** set the flag to `true` (in both the seed and the skip branch). Rationale by case:
  - *New install*: flag false, gallery empty → seed 11, set flag.
  - *Existing user upgrading*: flag false (key never existed), gallery non-empty → skip (no sprites dumped into their gallery), set flag so it's settled forever.
  - *User deleted all sprites*: flag true → never re-seed, even with an empty gallery (deletion is respected; this is why "seed whenever empty" is wrong).
- **Re-entrancy**: guard with an `@ObservationIgnored private var isSeedingSprites` in-memory flag set synchronously before the first `await` — the batch has suspension points, so a second `onAppear` (window re-open on macOS, nav pop on iOS) could otherwise start a concurrent seed. Guard `modelContext != nil` first, like every other mutation (log via `AppLog.gallery` and return).
- Return the number of items seeded (`@discardableResult`) for testability, mirroring `seedDefaultDisplayIfNeeded() -> Bool`.

### 3. Import mechanics (reuse the existing pipeline)

For each manifest name, in order:

1. `Bundle.module.url(forResource: name, withExtension: "png", subdirectory: "DefaultSprites")` + `Data(contentsOf:)`. Missing/unreadable resource → `AppLog.gallery.error(...)`, skip, continue — a seed failure must never block launch.
2. `try await createGalleryItem(name: name.capitalized, imageData: data)` — the exact user-import path, so seeded items get real `FileStorageManager`-persisted originals (deletable, variant-able, sendable), a decoded 45×35 `originalWidth/Height`, and a `contentHash` (which also dedups a half-seeded retry for free).
3. Wrap each call in `do/catch`: on throw, log and `currentError = nil` — `createGalleryItem` sets `currentError` on failure, and a background seed must not pop the launch-time error alert `GalleryListView` binds to it. (No `importMessage` handling needed: duplicates return `.duplicate` without throwing, and the empty-gallery predicate makes duplicates impossible in practice.)

Items get `importedDate = Date()` in loop order (milliseconds apart), so "Oldest First" shows coin→star; the default newest-first shows the reverse. Fine either way — the manifest order guarantees determinism, which is what matters.

### 4. Trigger + threading

In `GalleryListView.onAppear` (line 176), after `configure(modelContext:)` and `seedDefaultDisplayIfNeeded()`:

```swift
Task { await coordinator.seedDefaultSpritesIfNeeded() }
```

The package default-isolates to `@MainActor` and the coordinator is `@MainActor`, so the `Task` inherits main-actor context; only the `FileStorageManager` actor hops leave the main thread (identical to a normal import — 11 tiny ~250-byte PNGs, imperceptible). Repeated `onAppear` is safe: persisted flag → permanent no-op; `isSeedingSprites` → no concurrent double-seed within a launch. The `@Query` updates the grid live as items insert.

### 5. Idempotency / interruption

Set the persisted flag **after** the batch loop completes (also on the skip path). If the app is killed mid-seed (e.g. 5 of 11 done): next launch has flag false but a non-empty gallery → the predicate skips and sets the flag, leaving a partial set. **Trade-off accepted**: a partial seed of free sample content is harmless, whereas flag-first risks the same partial outcome with no better recovery, and any retry path that ignored the empty-check would fight deletion. `contentHash` dedup is the backstop against duplicates in any retry scenario. Individual sprite failures (step 3.3) do *not* abort the batch or leave the flag unset — log, skip, finish, set flag.

### 6. Tests (Swift Testing, in `GalleryCoordinatorTests` or a sibling `SpriteSeedingTests` suite)

Reuse the existing patterns: per-test temp directory for `FileStorageManager`, in-memory `ModelContainer` via `makeContext()`, and the `makeCoordinatorWithIsolatedDefaults()` helper (isolated `UserDefaults` suite, cleaned up with `removePersistentDomain`) from #0066's tests at `Tests/PixelArtGalleryKitTests/ViewModels/GalleryCoordinatorTests.swift:482`.

1. **Resources resolve**: all 11 names in `defaultSpriteNames` yield a `Bundle.module` URL with loadable non-empty data (guards the `Package.swift` declaration itself).
2. **Fresh seed**: empty gallery + flag false → returns 11; 11 `GalleryItem`s exist with expected capitalized names, `originalWidth == 45`, `originalHeight == 35`, non-empty `contentHash`, originals loadable via `loadOriginalImageData`; flag now true.
3. **Second call no-op**: immediately call again → returns 0, count stays 11.
4. **Upgrade skip**: import one item first, flag false → seed call returns 0, count stays 1, **flag is now true**.
5. **Deleted-all**: seed, delete all 11 via `deleteGalleryItem`, call again → returns 0, gallery stays empty (flag was true).

Fallback if `Bundle.module` unexpectedly fails under `swift test` (not anticipated — SwiftPM stages resource bundles for test runs): inject a sprite provider (`(String) -> Data?`) into the coordinator, defaulting to the `Bundle.module` lookup, and have tests feed synthetic 45×35 PNGs via the existing `makePNGData` helper. Only reach for this if the direct approach genuinely fails.

- Both app builds: `xcodebuild -project PixelArtGallery.xcodeproj -scheme PixelArtGallery -destination 'platform=macOS' build` and `xcodebuild -project PixelArtGallery.xcodeproj -scheme PixelArtGallery -destination 'platform=iOS Simulator,name=iPhone 17 Pro' build`.

## Notes

Relevant code / facts:
- Sprites live at `sprites/*.png` (repo root), 11 files, each 45×35 RGBA PNG. `scripts/sprites.py` generates them (reference only — not shipped/run at runtime).
- Import pipeline: `GalleryCoordinator.createGalleryItem(...)` (or the import entry it exposes) loads image bytes, saves originals via `FileStorageManager.save(imageData:) -> filename`, records `originalWidth/originalHeight` + `contentHash` (SHA-256, for dedup), and inserts a `GalleryItem` with `variants: []`. Seeding should reuse this exact path so seeded items are indistinguishable from user imports (deletable, variant-able, sendable).
- Precedent for once-only seeding: `GalleryCoordinator.seedDefaultDisplayIfNeeded()` seeds the default FT display. Mirror that shape for sprites (`seedDefaultSpritesIfNeeded()`), and note where it's invoked (app launch / `GalleryListView.onAppear` / `ContentView`).
- Persistence precedent: lightweight flags use `@AppStorage`/`UserDefaults` (e.g. `@AppStorage("gallerySortOrder")`, the `lastUsedFTDisplayID` UserDefaults key from #0066) — no SwiftData schema change needed for the seed flag.
- `Package.swift` for `PixelArtGalleryKit` — resources are declared per target; bundling the PNGs in the Kit and reading them via `Bundle.module` keeps them next to the seeding code (vs an app-target asset catalog). Planner to choose and specify.

Design points (planner to finalize):
- **Bundling**: add the 11 PNGs as a resource of `PixelArtGalleryKit` (e.g. copy `sprites/` into `Sources/PixelArtGalleryKit/Resources/DefaultSprites/` and declare `.copy`/`.process` in `Package.swift`), loaded via `Bundle.module`. Decide `.copy` vs `.process` (`.process` may optimize PNGs; `.copy` preserves exact bytes — for pixel art `.copy` is safer to avoid any recompression). Keep the source `sprites/` folder as the generator output; the shipped copy is the resource.
- **Seed trigger + guard**: `seedDefaultSpritesIfNeeded()` runs on launch; seeds ONLY when a persisted `didSeedDefaultSprites` flag is false. On seed: import all 11 (in a stable, sensible order), then set the flag true — permanently. This means: first launch (flag false, gallery empty) → seed; user deletes all → flag stays true → no re-seed. Do NOT seed merely because the gallery is empty (that would fight deletion). Consider also skipping the seed if the gallery is already non-empty at flag-check time (an existing user upgrading shouldn't get sprites dumped in) — recommend: seed only if `flag == false && gallery is empty`, then always set the flag. Handle a missing/corrupt resource gracefully (skip that sprite, log, continue) — a seed failure must never block launch.

Testing (Swift Testing, in-memory `ModelContainer` + temp `FileStorageManager`, mirroring `GalleryCoordinatorTests`): seeding into an empty gallery with the flag false creates 11 `GalleryItem`s with the expected names/dimensions (45×35) and sets the flag; a second `seedDefaultSpritesIfNeeded()` call is a no-op (flag true → count stays 11); seeding is skipped when the gallery is already non-empty; a deleted-all state does not re-seed (flag true). The bundled resources must be resolvable via `Bundle.module` in the test target (confirm the resource declaration makes them available to tests). Also verify by **running the app** on a fresh install (empty store) that the 11 sprites appear in the gallery, and that after deleting them and relaunching they stay gone.
