## Description

Ship the 11 generated pixel-art sprites (`sprites/*.png`, 45├Ś35, from `scripts/sprites.py`) bundled with the app and **seed them into the gallery on first launch when there are 0 images**, so new users start with content instead of an empty gallery. The sprites are ordinary gallery items ÔÇö the user can add their own images and **delete** any/all of the seeded sprites. Seeding must be **once-only** (guarded by a persisted flag), so deleting the sprites does not cause them to reappear on the next launch.

## Long Description

The sprites are 45├Ś35 ÔÇö exactly a common FT display size ÔÇö so they're ideal starting points: pick a display and the fitted preview is ~1:1. There are 11: coin, frog, ghost, heart, invader, mushroom, pacman, robot, ship, skull, star. They need to be (a) bundled as app/package resources and (b) imported as `GalleryItem`s through the existing import pipeline on first run.

## Notes

Relevant code / facts:
- Sprites live at `sprites/*.png` (repo root), 11 files, each 45├Ś35 RGBA PNG. `scripts/sprites.py` generates them (reference only ÔÇö not shipped/run at runtime).
- Import pipeline: `GalleryCoordinator.createGalleryItem(...)` (or the import entry it exposes) loads image bytes, saves originals via `FileStorageManager.save(imageData:) -> filename`, records `originalWidth/originalHeight` + `contentHash` (SHA-256, for dedup), and inserts a `GalleryItem` with `variants: []`. Seeding should reuse this exact path so seeded items are indistinguishable from user imports (deletable, variant-able, sendable).
- Precedent for once-only seeding: `GalleryCoordinator.seedDefaultDisplayIfNeeded()` seeds the default FT display. Mirror that shape for sprites (`seedDefaultSpritesIfNeeded()`), and note where it's invoked (app launch / `GalleryListView.onAppear` / `ContentView`).
- Persistence precedent: lightweight flags use `@AppStorage`/`UserDefaults` (e.g. `@AppStorage("gallerySortOrder")`, the `lastUsedFTDisplayID` UserDefaults key from #0066) ÔÇö no SwiftData schema change needed for the seed flag.
- `Package.swift` for `PixelArtGalleryKit` ÔÇö resources are declared per target; bundling the PNGs in the Kit and reading them via `Bundle.module` keeps them next to the seeding code (vs an app-target asset catalog). Planner to choose and specify.

Design points (planner to finalize):
- **Bundling**: add the 11 PNGs as a resource of `PixelArtGalleryKit` (e.g. copy `sprites/` into `Sources/PixelArtGalleryKit/Resources/DefaultSprites/` and declare `.copy`/`.process` in `Package.swift`), loaded via `Bundle.module`. Decide `.copy` vs `.process` (`.process` may optimize PNGs; `.copy` preserves exact bytes ÔÇö for pixel art `.copy` is safer to avoid any recompression). Keep the source `sprites/` folder as the generator output; the shipped copy is the resource.
- **Seed trigger + guard**: `seedDefaultSpritesIfNeeded()` runs on launch; seeds ONLY when a persisted `didSeedDefaultSprites` flag is false. On seed: import all 11 (in a stable, sensible order), then set the flag true ÔÇö permanently. This means: first launch (flag false, gallery empty) Ôćĺ seed; user deletes all Ôćĺ flag stays true Ôćĺ no re-seed. Do NOT seed merely because the gallery is empty (that would fight deletion). Consider also skipping the seed if the gallery is already non-empty at flag-check time (an existing user upgrading shouldn't get sprites dumped in) ÔÇö recommend: seed only if `flag == false && gallery is empty`, then always set the flag. Planner to state the exact predicate and its rationale for both new-install and upgrade cases.
- **Import mechanics**: read each PNG's bytes from `Bundle.module`, run them through the same `FileStorageManager.save` + `GalleryItem` creation as a normal import (so originals persist to Application Support and are deletable), with friendly names derived from the filename (e.g. `coin.png` Ôćĺ "Coin"). Handle a missing/corrupt resource gracefully (skip that sprite, log, continue) ÔÇö a seed failure must never block launch.
- **Idempotency/safety**: seeding is guarded by the flag, but also make the operation safe if interrupted mid-way (e.g. app killed after seeding 5 of 11) ÔÇö decide whether to set the flag only after all succeed (risking a partial + re-seed) or per-item; recommend setting the flag after the batch completes and relying on the empty-gallery predicate + contentHash dedup to avoid duplicates on a retry.

Testing (Swift Testing, in-memory `ModelContainer` + temp `FileStorageManager`, mirroring `GalleryCoordinatorTests`): seeding into an empty gallery with the flag false creates 11 `GalleryItem`s with the expected names/dimensions (45├Ś35) and sets the flag; a second `seedDefaultSpritesIfNeeded()` call is a no-op (flag true Ôćĺ count stays 11); seeding is skipped when the gallery is already non-empty; a deleted-all state does not re-seed (flag true). The bundled resources must be resolvable via `Bundle.module` in the test target (confirm the resource declaration makes them available to tests). Also verify by **running the app** on a fresh install (empty store) that the 11 sprites appear in the gallery, and that after deleting them and relaunching they stay gone.
