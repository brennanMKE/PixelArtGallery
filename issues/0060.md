# 0060 ‚Äî Discovered FT displays are silently dropped because NWBrowser service results are never resolved to host/port

| | |
|---|---|
| **Status** | open |
| **Module** | Networking |
| **Platform** | All |
| **First seen** | 2026-07-10 |

## Description

Even after #0058 (local-network authorization) and #0059 (correct `_flaschen-taschen._udp` type), the app's scan reaches "Browser ready" and then discovers 0 displays, while `scripts/scan-ft-displays.py` finds the same displays instantly (Mac 192.168.4.26:1337, ESP32 192.168.4.31:1337). Root cause is in `FTDiscoveryService.discovered(from:)`: `NWBrowser` delivers browse results whose endpoint is `.service(name:type:domain:interface:)` ‚Äî a bare Bonjour reference, **not** a resolved `.hostPort`. The code only builds a `DiscoveredFTDisplay` for the `.hostPort` case and returns `nil` for everything else, so every real result is discarded. `NWBrowser` never delivers a resolved `.hostPort` follow-up (the code comment assuming it does is wrong); resolving a Bonjour service to an IP/port requires a separate step (e.g. an `NWConnection` to the service endpoint, then reading `currentPath.remoteEndpoint`). This affects **both** platforms ‚Äî the macOS app cannot discover displays either, despite needing no local-network permission.

## Plan

### Confirmed root cause

Verified against the actual code in `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Networking/FTDiscoveryService.swift`:

- `scan(duration:)` builds an `NWBrowser` with `NWBrowser.Descriptor.bonjourWithTXTRecord(type:domain:)` and, in `browseResultsChangedHandler` (lines 93‚Äì100), passes every `NWBrowser.Result` to the private `static func discovered(from:)` (lines 127‚Äì145).
- `discovered(from:)` switches on `result.endpoint` and only builds a `DiscoveredFTDisplay` in the `.hostPort(host, port)` case. The `default:` branch returns `nil` with a comment claiming "The browser will deliver a resolved result on a later change" ‚Äî **that assumption is wrong**. A Bonjour `NWBrowser` delivers results whose endpoint is `.service(name:type:domain:interface:)`, an unresolved service reference, and it never follows up with a `.hostPort` result. Resolution is a separate step the caller must perform (this is exactly what `dns-sd -L` / `scripts/scan-ft-displays.py` do and the app skips).
- Net effect: with #0058 (authorization) and #0059 (correct `_flaschen-taschen._udp` type) in place, results now *arrive* ‚Äî and every one is silently dropped, so `GalleryCoordinator.scanForDisplays()` (line 579, which collects the stream) always logs "0 discovered". Both platforms are affected identically.

Isolation model, confirmed: `Package.swift` sets `.defaultIsolation(MainActor.self)`, but `FTDiscoveryService` is an `actor` (its own isolation domain). The browse handlers are `@Sendable` closures running on the browser's dispatch queue (`.global(qos: .userInitiated)`); they already call the `static` (nonisolated) helpers and `continuation.yield(...)` directly, which is safe ‚Äî `AsyncStream.Continuation` is `Sendable` and thread-safe. `stop()` cancels the browser; `continuation.onTermination` hops back into the actor via `Task { await self.stop() }`. The fix must preserve this shape while adding per-result resolution.

### Resolution approach (Network.framework)

For each browse result whose endpoint is `.service`, resolve it with a short-lived UDP `NWConnection` to the service endpoint ‚Äî the same idiom `FTDisplayClient` already uses for connect-with-timeout (`withConnectionReady`, `ResumeOnce` latch):

1. Create `let parameters = NWParameters.udp` (fresh instance per call) and set `parameters.includePeerToPeer = true`, matching the browse parameters ‚Äî a service discovered over peer-to-peer (AWDL) must also resolve over it. UDP matches FT's wire protocol and never actually transmits anything during resolution.
2. `let connection = NWConnection(to: result.endpoint, using: parameters)` ‚Äî connecting to a `.service` endpoint makes Network.framework perform the Bonjour SRV/A resolution internally.
3. `connection.stateUpdateHandler`: on `.ready`, read `connection.currentPath?.remoteEndpoint`. That is a resolved `.hostPort(host, port)`; extract the host via the existing `hostString(_:)` (handles `.name`/`.ipv4`/`.ipv6`) and the port via `port.rawValue`. Build the value with the existing `DiscoveredFTDisplay.make(host:port:serviceName:txtRecord:)`, folding in `txtRecord(from: result.metadata)` (TXT is already on the browse result ‚Äî `bonjourWithTXTRecord` ‚Äî no separate TXT fetch needed) and the `.service` case's `name` as `serviceName` (existing `serviceName(for:)` already extracts it). Then `continuation.yield(...)`, cancel the connection, and unregister it. On `.failed`/`.cancelled`: cancel/unregister and skip gracefully (discovery already degrades to manual entry).
4. `connection.start(queue: .global(qos: .userInitiated))` ‚Äî same queue convention as the browser and `FTDisplayClient`.
5. **Per-resolve timeout**: a `Task { try? await Task.sleep(for: resolveTimeout); ‚Ä¶ cancel if still pending }` bound to a once-only latch (mirror `FTDisplayClient`'s `ResumeOnce` + timeout-task pattern at lines 273‚Äì306 of `FTDisplayClient.swift`). Recommend `static let resolveTimeout: Duration = .seconds(3)` ‚Äî shorter than the default 5 s scan window so a silent host can't pin a connection past the scan, long enough for real mDNS resolution (sub-second on this LAN).

Keep the `.hostPort` fast path in `discovered(from:)`-successor code (a result that's already resolved yields immediately); the `.service` case now triggers async resolution instead of returning `nil`. Delete the incorrect comment.

No plist changes: resolving/connecting to a service the app was authorized to browse is covered by the existing `NSBonjourServices` + local-network grant from #0058/#0059.

### Async lifecycle, teardown, dedup

- **Yield-as-resolved**: each connection yields to the existing `AsyncStream` continuation from its state handler the moment it resolves, within the scan window. Multiple results resolve concurrently (one connection each). `GalleryCoordinator.scanForDisplays()` keeps collecting until the stream finishes at `duration` ‚Äî no coordinator changes needed. The "Found ‚Ä¶ at host:port" debug log moves into the resolve completion.
- **Tracking + teardown**: add a small `nonisolated final class` resolver pool (`@unchecked Sendable`, `NSLock`-guarded ‚Äî same pattern as `ResumeOnce`) holding the in-flight `NWConnection`s and a `Set<String>` of service names already resolved or in flight. One pool per scan, created inside `scan(duration:)` alongside the browser and stored on the actor (`private var resolverPool: ...?`). `stop()` calls `pool.cancelAll()` before/with `browser?.cancel()`, and `continuation.onTermination` already routes to `stop()` ‚Äî so stream termination (duration elapsed, caller stops iterating, or explicit `stop()`) cancels every outstanding connection; none leak. Late state callbacks after teardown are harmless: `yield` on a finished continuation is a no-op, and each connection's completion path is guarded by its once-latch (ready/failed/cancelled/timeout all funnel through cancel + unregister exactly once).
- **Dedup**: `browseResultsChangedHandler` receives the *complete* result set on every change, and the same service can appear once per interface ‚Äî so without dedup the code would open several connections per display per callback. Key dedup on the Bonjour service instance name (the `.service` case's `name`): `pool.beginResolving(name:)` returns `false` if that name is already in flight or done, and the handler skips it. Resolved host:port duplicates need no second guard here ‚Äî `DisplayMergePlan.build` already collapses discoveries by normalized `host:port` key downstream (`Models/DisplayMergePlan.swift` lines 47‚Äì59).
- **IPv6 note**: `currentPath?.remoteEndpoint` may hand back an IPv6 (possibly link-local `fe80::‚Ä¶%en0`) host. `hostString(_:)` renders it verbatim and `FTDisplayClient` parses it back via `NWEndpoint.Host(_:)`, so it round-trips; don't add special-casing. If on-device testing shows link-local noise, preferring IPv4 can be a follow-up ‚Äî don't over-engineer now.

## Steps to reproduce

1. Run FT servers advertising `_flaschen-taschen._udp` (Mac app, ESP32).
2. Open the app (iOS or macOS) and trigger a display scan.
3. Observe "Browser ready" then "0 discovered", with no "Found ‚Ä¶ at host:port" log lines ‚Äî the browse results arrived but were dropped as unresolved `.service` endpoints.

Contrast: `scripts/scan-ft-displays.py --type _flaschen-taschen._udp` resolves and lists the same displays, because `dns-sd -L` performs the resolve step the app skips.

## Expected behavior

Each advertised FT service is resolved to host/port (with TXT metadata) and surfaced as a `DiscoveredFTDisplay`, so the scan discovers the running displays.

## Actual behavior

`browseResultsChangedHandler` fires with `.service` endpoints; `discovered(from:)` returns `nil` for all of them (only `.hostPort` is handled), so nothing is yielded and the scan reports 0 discovered.

## Notes

Relevant code: `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Networking/FTDiscoveryService.swift` ‚Äî `scan(duration:)`'s `browseResultsChangedHandler` (lines ~93‚Äì100) and the private `discovered(from:)` (lines ~127‚Äì145), whose `default` branch drops unresolved service endpoints on the incorrect assumption that a resolved result will arrive later.

Likely fix: for each `.service` browse result, resolve it before yielding ‚Äî open a short-lived `NWConnection` to `result.endpoint` (UDP parameters, matching FT), start it on the actor's queue, and on `.ready` read `connection.currentPath?.remoteEndpoint` (a `.hostPort`) for the host/port, then cancel the connection. Fold in the TXT record already available on `result.metadata`. Because resolution is async, results should be yielded as each connection resolves (within the scan's duration window), and failures to resolve should be skipped gracefully (discovery already degrades to manual entry). Watch actor isolation (the service is an `actor`; the package defaults to `@MainActor`) and make sure connections are torn down when the scan stops/`stop()` is called so none leak past the AsyncStream's termination. Consider a per-result resolve timeout so a silent host doesn't hold the scan open.

Testing is constrained (real mDNS resolution isn't available in CI), but the endpoint‚Üíhost/port extraction and TXT folding can be factored into a pure, testable helper (mirroring how `FTPaintTarget`/other value types are unit-tested), and a test can cover the `.hostPort` remoteEndpoint parsing and TXT merge independent of the live network.
