# 0055 — Variant view does not auto-select the display matching the variant's dimensions

| | |
|---|---|
| **Status** | open |
| **Module** | UI |
| **Platform** | All |
| **First seen** | 2026-07-09 |

## Description

When selecting a variant such as 8x8 while a display which is 8x8 exists, the app should automatically select that display in the send UI. Today the user has to change the display selection manually, often, which is cumbersome.

## Plan

### How selection works today, and why it ignores geometry

`VariantDetailView.sendSection` (`PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/VariantDetailView.swift`, the `.onChange(of: displays.map(\.id), initial: true)` block around lines 228–243) seeds and maintains `selectedDisplayID` by calling `FlaschenTaschenDisplay.preferredSelection(current:among:)` (`PixelArtGalleryKit/Sources/PixelArtGalleryKit/Models/FlaschenTaschenDisplay.swift`, added for #0032). That pure function only sees `(id, source)` pairs: it keeps a still-valid `current`, otherwise prefers the seeded default display (`source == "default"`), otherwise the first candidate. Display geometry (`displayWidth`/`displayHeight`) and the variant's `targetWidth`/`targetHeight` are never consulted, so on open the picker lands on the default/first display regardless of dimensions. There is no other persisted "last used display" store — the `@State selectedDisplayID` starts `nil` on every view open, so the seeding call is the single place to fix.

### Approach

1. **Extend the pure selection rule with geometry** in `FlaschenTaschenDisplay.swift`: change `preferredSelection` to also take the variant's dimensions and geometry per candidate, e.g. `preferredSelection(current: UUID?, variantWidth: Int, variantHeight: Int, among candidates: [(id: UUID, source: String, width: Int, height: Int)]) -> UUID?`. Keep it a static pure function on plain tuples (no SwiftData) so it stays unit-testable. New rule order:
   - **Match set:** candidates whose `width × height` equals the variant's `targetWidth × targetHeight` (exact equality, no rotation/aspect matching).
   - **Current already matches:** if `current` is in the match set, keep it — never stomp a selection that already fits.
   - **Tie-break among several matches:** otherwise pick the *first* match in candidate order (candidates arrive from the `@Query` sorted by `displayName`, so this is deterministic: the alphabetically first matching display). This follows the issue's Notes; no default-source preference inside the match set.
   - **Fallback when none match:** exactly today's behavior — keep a still-valid `current`, else the default-source display, else the first candidate, else `nil`. This preserves the #0032 guarantees for registries with no geometry match.
2. **Update the call site** in `VariantDetailView.sendSection`: pass `variant.targetWidth` / `variant.targetHeight` and map candidates to `(id:source:width:height:)` from each display's `displayWidth`/`displayHeight`. Also broaden the `.onChange` trigger key from `displays.map(\.id)` to a value that includes each display's geometry (e.g. `displays.map { "\($0.id)|\($0.displayWidth)x\($0.displayHeight)" }`) and add an `.onChange` on the variant's dimensions, so editing a display's geometry in Settings or resizing the variant via Edit Dimensions re-evaluates the selection. `seedSendLayer()` already reseeds via the existing `.onChange(of: selectedDisplayID)`, so the layer follows automatically.
3. **Accepted trade-off (note in `## Fix` if kept):** because a valid-but-non-matching `current` is replaced by a geometry match, a user's explicit non-matching pick can be overridden if the registry or variant dimensions change while the view is open. That is rare and is exactly the auto-select behavior this issue asks for; on plain view open `current` is always `nil`, so the common path is a clean auto-select.
4. **Update tests** in `PixelArtGalleryKit/Tests/PixelArtGalleryKitTests/UI/DisplayPreferredSelectionTests.swift` (Swift Testing): adapt the five existing #0032 cases to the new signature (they exercise the none-match fallback), and add geometry cases — auto-selects the sole matching display over the default; keeps a current selection that already matches when several match; picks the first (name-sorted) match when current doesn't match; replaces a non-matching current with a match; falls back to default/first/nil when nothing matches; empty candidate list returns `nil`.

## Steps to reproduce

1. Register two displays with different geometries (e.g. 8x8 and 64x64).
2. Open an 8x8 variant.
3. Look at the display picker in the send section.

## Expected behavior

The display whose geometry matches the variant's dimensions (8x8) is selected automatically when the variant view opens.

## Actual behavior

The previously selected (or default) display stays selected regardless of the variant's dimensions, and the user must switch it manually every time.

## Notes

Likely code: `VariantDetailView.swift` (display picker / send section) and `FlaschenTaschenDisplay` (width/height). If several displays match the variant's dimensions, keep the current selection when it already matches, otherwise pick the first match; fall back to existing behavior when nothing matches.
