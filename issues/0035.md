## Description

With a large gallery (say 50 images), the ones you care about drown in the grid. Make it possible to pin an image so it appears first in the gallery — a short list that always leads — and then support a way to sort the rest of the (unpinned) images.

## Expected behavior

- Each gallery item can be pinned/unpinned (e.g. from the cell's context menu). Pinned items always appear at the front of the grid, visually marked (e.g. a small pin badge).
- The remaining, unpinned items follow, ordered by a user-selectable sort — at minimum newest first (current behavior, the default), oldest first, and name A→Z / Z→A — chosen from a toolbar sort menu and remembered across launches.
- Pinned state persists (it's part of the item, not the session).

## Actual behavior

No pinning; the grid is always sorted by import date, newest first.

## Notes

- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Models/GalleryItem.swift` — add `isPinned: Bool` with a default of `false` (SwiftData lightweight migration handles the new attribute).
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/GalleryListView.swift` — the grid (`@Query(sort: \.importedDate, order: .reverse)`); with a user-selectable sort the ordering can move in-memory (query unsorted, sort in the view) with the choice persisted in `@AppStorage`. Context menu gains Pin/Unpin; toolbar gains a sort menu.
- The ordering rule (pinned first, then the chosen sort) should live in a unit-testable helper rather than inline view code.
- `GalleryCoordinator` is the mutation path for model changes (see `renameGalleryItem`) — pin toggling belongs there for consistency.

## Root cause

Not a bug — a feature gap. `GalleryItem` had no pinned attribute, the grid's ordering was hard-wired into `@Query(sort: \.importedDate, order: .reverse)`, and the view offered no way to change it, so favorites drowned in a large gallery.

## Fix

- **Model** — `GalleryItem.isPinned: Bool = false`. The property default (no new initializer parameter) keeps existing call sites compiling and lets SwiftData lightweight migration backfill `false` on existing stores.
- **Ordering rule** — new `nonisolated public enum GallerySortOrder: String, CaseIterable` (`newestFirst` / `oldestFirst` / `nameAscending` / `nameDescending`, each with a `displayName`) in `Models/GallerySortOrder.swift`, with `sortedForGallery(_:)`. The rule: partition into pinned/unpinned, sort each group by the chosen order, concatenate pinned + unpinned — so the pinned "short list" always leads and is itself ordered by the same user-chosen sort. Name comparisons use `localizedStandardCompare` (case-insensitive, Finder-style numeric). Ties are deterministic: equal dates fall back to name A→Z, equal names fall back to newest first (Swift's `sort` isn't guaranteed stable).
- **Coordinator** — `togglePin(_:)` flips `isPinned` and saves, mirroring `renameGalleryItem`'s guard/save/error pattern.
- **View** — `GalleryListView` keeps the stable newest-first `@Query` as a base and applies `sortedForGallery` in-memory; the choice persists via `@AppStorage("gallerySortOrder")` (raw value, default `newestFirst`, unknown values fall back safely). Each cell's context menu gains Pin/Unpin (`pin` / `pin.slash`) above Rename; pinned thumbnails get a `pin.fill` badge on an `.ultraThinMaterial` circle (top-trailing, subtle shadow) so it reads on any image. The toolbar gains a sort `Menu` (`arrow.up.arrow.down`, `.secondaryAction` like Displays) wrapping a `Picker` so the active sort shows a checkmark. Navigation, rename/delete, empty state, import, and background untouched.

## Gotchas

- `GalleryListView` builds its own `GalleryCoordinator()` (default `FileStorageManager` → real `~/Library/Application Support`), so a harness that seeds into an injected temp directory would render broken thumbnails. Redirecting `HOME` for the harness process makes the view's default path land in the temp dir too — full isolation without touching the view's API.
- First harness run produced identical screenshots for newest-first and name A→Z because the seeded dates happened to be in alphabetical order; the dates were scrambled so the two sorts are visibly distinct.
