# 0004 — Generate and persist variant pixel data from the original image

## Description

Creating a variant produces an empty grid. `GalleryCoordinator.createVariant(for:width:height:)` feeds `let imageData = Data()` into the `PixelationEngine` instead of loading the gallery item's original image, so every variant's `pixelGridData` is empty/black and nothing is persisted to SwiftData.

## Expected behavior

Variant creation loads the original image bytes from `FileStorageManager` using `item.originalImagePath`, runs the `PixelationEngine` at the requested target dimensions, stores the resulting RGBA8888 data in a new `Variant`, links it to the parent item, inserts it into the context, and shows it in the detail view's variants list.

## Root cause

`GalleryCoordinator.createVariant(for:width:height:)` had a placeholder
`let imageData = Data()` where it should have loaded the gallery item's stored
original. The empty buffer was passed to `PixelationEngine.process(...)`, which
drew nothing into the target bitmap, so every variant's `pixelGridData` was an
all-black (zeroed) RGBA8888 grid. The insert+save wiring was already correct —
only the image source was missing.

## Fix

Replaced the placeholder with a load of the original bytes from the cached
`FileStorageManager` via `item.originalImagePath`. A missing or unreadable file
surfaces through `currentError` and is rethrown using a new
`GalleryCoordinatorError.originalImageMissing(_:)` case (the enum now conforms to
`LocalizedError` for friendly messages). The loaded bytes feed
`PixelationEngine.process(...)` and the resulting `pixelGrid.toRGBA8888()` is
stored on the new `Variant`, linked to the parent item, then inserted and saved.
The whole body is wrapped in a do/catch mirroring `createGalleryItem`. The
downsampling algorithm itself was left untouched (bilinear is 0005); no variant
edit/duplicate UI was added (0015).
