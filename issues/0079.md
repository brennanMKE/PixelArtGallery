# 0079 — Display selection flips to the wrong FT when two displays share the same dimensions

| | |
|---|---|
| **Status** | open |
| **Module** | UI / Models |
| **Platform** | All |
| **First seen** | 2026-07-11 |

## Description

With two FT displays both set to 45×35, opening an image sometimes preselects the *other* 45×35 display instead of the one the user has been using. Root cause: `FlaschenTaschenDisplay.preferredSelection(...)` breaks ties among same-dimension displays by **list order** (`matches.first`), not by the **last-used** display. Fix the selection logic so that when several displays match the variant's dimensions, it prefers the last-used display (then the current selection), rather than arbitrarily switching to another same-dimension FT.

## Long Description

`VariantDetailView.updateSelectedDisplay()` calls `FlaschenTaschenDisplay.preferredSelection(current:variantWidth:variantHeight:among:)`:

```swift
let matches = candidates.filter { $0.width == variantWidth && $0.height == variantHeight }
if !matches.isEmpty {
    if let current, matches.contains(where: { $0.id == current }) { return current }
    return matches.first?.id      // <-- arbitrary among same-dimension displays; ignores last-used
}
```

When the variant is 45×35 and two displays are 45×35, `matches` has both. If `current` (the screen's `selectedDisplayID`) isn't already set to one of them — which is the case on a fresh open of the screen — it returns `matches.first?.id`, i.e. whichever display the `@Query` happens to order first. It never consults the persisted last-used display. So the selection "changes to the other one." The popover (`GallerySendPopoverView`) already defaults correctly via `GalleryCoordinator.resolveLastUsedDisplay(among:)` (which returns the stored last-used id when present), but `preferredSelection` — the path `VariantDetailView` uses — does not.

## Plan

Root cause is confirmed exactly as described: `FlaschenTaschenDisplay.preferredSelection` (`Models/FlaschenTaschenDisplay.swift:158-177`) filters `candidates` to dimension `matches`, keeps `current` if it's among them, and otherwise returns `matches.first?.id` — the name-sorted `@Query` order — never consulting the persisted last-used display (#0066). On a fresh open of `VariantDetailView`, `selectedDisplayID` is `nil`, so with two 45×35 displays the picker lands on whichever sorts first, flipping away from the display the user has been using. Separately, `VariantDetailView` never calls `coordinator.rememberLastUsedDisplay(_:)` — its `onChange(of: selectedDisplayID)` (line 318) only calls `seedSendDefaults()` — so picking a display on this screen doesn't update last-used the way the popover does (`GallerySendPopoverView.swift:125-128`).

### 1. Extend `preferredSelection` with a `lastUsed` tie-breaker

`PixelArtGalleryKit/Sources/PixelArtGalleryKit/Models/FlaschenTaschenDisplay.swift` — change the signature to:

```swift
static func preferredSelection(
    current: UUID?,
    lastUsed: UUID? = nil,
    variantWidth: Int,
    variantHeight: Int,
    among candidates: [(id: UUID, source: String, width: Int, height: Int)]
) -> UUID?
```

`lastUsed` is defaulted to `nil` so every existing call site (the one production caller in `VariantDetailView` and all twelve existing tests) compiles unchanged with today's behavior. New tie-break order **among `matches` only**:

1. `current`, if it is one of the matches — a deliberate, still-valid selection is never moved (unchanged from today);
2. `lastUsed`, if it is one of the matches — **new**: a fresh open (where `current` is `nil` or stale) now prefers the display the user last used over arbitrary list order;
3. `matches.first?.id` — unchanged final fallback (deterministic name-sorted order).

Concretely, insert one line between the existing `current` check and the `matches.first` return:

```swift
if let lastUsed, matches.contains(where: { $0.id == lastUsed }) {
    return lastUsed
}
```

The no-matches fallback ladder (`current` if still valid → `source == defaultSource` → `candidates.first` → `nil`) stays byte-for-byte unchanged — `lastUsed` deliberately does not participate there (that path already has the sensible #0032 semantics, and the popover's `resolveLastUsedDisplay` covers the last-used-without-geometry case). Keep it a pure static func operating on plain tuples (unit-testable without SwiftData). It is *not* marked `nonisolated` today — the package default-isolates `@MainActor` and the test suite is `@MainActor` — so leave isolation as-is; no change needed. Update the function's doc comment to describe the new tie-break, and fix the two DocC references to the old signature (`GalleryCoordinator.swift:1151` and the header comment of `DisplayPreferredSelectionTests.swift`).

### 2. Wire `VariantDetailView` — pass `lastUsed`, remember on change

`PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/VariantDetailView.swift`:

- **Pass the tie-breaker** in `updateSelectedDisplay()` (line 352): add `lastUsed: coordinator.resolveLastUsedDisplay(among: displays)?.id` to the `preferredSelection` call. Candidate-tuple building from the `@Query` is otherwise unchanged. Note: `resolveLastUsedDisplay` has its own fallback ladder (stored id → default-source → first), so when nothing is stored yet `lastUsed` resolves to the seeded default display — which means the tie among equal-dimension displays then breaks toward the built-in default rather than raw sort order. That is deterministic, matches the #0032 spirit, and is acceptable; no need for a raw-stored-id accessor.
- **Centered-seed precedence is preserved for free**: `updateSelectedDisplay()` returns early on `pendingCenteredDisplayID` (lines 353–356) *before* ever calling `preferredSelection`, so the #0064 display-first push still wins whenever it's pending. Confirm this ordering is untouched.
- **Add remember-on-change** (currently missing): extend the existing `onChange(of: selectedDisplayID)` (line 318) to also persist the selection, mirroring `GallerySendPopoverView.swift:125-128`:

  ```swift
  .onChange(of: selectedDisplayID) { _, newValue in
      seedSendDefaults()
      if let newValue, let display = displays.first(where: { $0.id == newValue }) {
          coordinator.rememberLastUsedDisplay(display)
      }
  }
  ```

  This fires for programmatic selection too (auto-match on open, the centered-seed assignment), exactly as the popover's does for its own programmatic init — that's consistent and idempotent: on a fresh open the auto-selection now *is* the last-used (or the sole dimension match, which is then a reasonable new last-used), and a #0064 push remembers the display the user explicitly chose in the display-first picker. No fight with `seedSendDefaults()`/`SendDefaultsSeed` — remembering only writes a UserDefaults string and reads no seed state.

### 3. Coordinator and popover — no changes

- `GalleryCoordinator.rememberLastUsedDisplay(_:)` / `resolveLastUsedDisplay(among:)` (`ViewModels/GalleryCoordinator.swift:1146-1168`) are reused as-is; only the DocC comment at line 1151 needs the updated `preferredSelection` signature.
- `GallerySendPopoverView` already defaults via `resolveLastUsedDisplay` (line 265) and remembers on change (line 127) — verified correct; **no change needed** (it never calls `preferredSelection`, so the bug cannot reproduce there).

### 4. Tests (Swift Testing — extend `DisplayPreferredSelectionTests`)

Extend `PixelArtGalleryKit/Tests/PixelArtGalleryKitTests/UI/DisplayPreferredSelectionTests.swift` (existing `@MainActor @Suite`; do not create a new file). Existing tests compile unchanged thanks to the defaulted parameter and pin the `lastUsed: nil` → old-behavior path (notably `picksFirstMatchInCandidateOrderWhenCurrentDoesNotMatch`). Add a `// MARK: - #0079 last-used tie-break` section:

- `lastUsedBreaksTieAmongEqualDimensionMatches` — two 45×35 candidates, `current: nil`, `lastUsed:` the **second** → returns the second, not `matches.first` (the core bug).
- `currentMatchingSelectionWinsOverLastUsed` — both candidates match, `current:` first, `lastUsed:` second → returns first (`current` still wins).
- `lastUsedNotAmongMatchesFallsBackToFirstMatch` — two matches, `lastUsed:` the id of a third, non-matching (or absent) display → `matches.first`.
- `lastUsedDoesNotAffectNoMatchFallback` — no candidate matches the variant dimensions, `lastUsed:` set to a candidate → still the default-source display (unchanged #0032 ladder).
- `singleMatchIgnoresLastUsed` — one matching display, `lastUsed:` a non-matching display's id → the sole match.
- `nilLastUsedKeepsFirstMatchOrder` — explicit `lastUsed: nil` with two matches → `matches.first` (locks the compatibility default).
