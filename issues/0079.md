# 0079 — Display selection flips to the wrong FT when two displays share the same dimensions

| | |
|---|---|
| **Status** | open |
| **Module** | UI / Models |
| **Platform** | All |
| **First seen** | 2026-07-11 |

## Description

With two FT displays both set to 45×35, opening an image sometimes preselects the *other* 45×35 display instead of the one the user has been using. Root cause: `FlaschenTaschenDisplay.preferredSelection(...)` breaks ties among same-dimension displays by **list order** (`matches.first`), not by the **last-used** display. Fix the selection logic so that when several displays match the variant's dimensions, it prefers the last-used display (then the current selection), rather than arbitrarily switching to another same-dimension FT.

## Long Description

`VariantDetailView.updateSelectedDisplay()` calls `FlaschenTaschenDisplay.preferredSelection(current:variantWidth:variantHeight:among:)`:

```swift
let matches = candidates.filter { $0.width == variantWidth && $0.height == variantHeight }
if !matches.isEmpty {
    if let current, matches.contains(where: { $0.id == current }) { return current }
    return matches.first?.id      // <-- arbitrary among same-dimension displays; ignores last-used
}
```

When the variant is 45×35 and two displays are 45×35, `matches` has both. If `current` (the screen's `selectedDisplayID`) isn't already set to one of them — which is the case on a fresh open of the screen — it returns `matches.first?.id`, i.e. whichever display the `@Query` happens to order first. It never consults the persisted last-used display. So the selection "changes to the other one." The popover (`GallerySendPopoverView`) already defaults correctly via `GalleryCoordinator.resolveLastUsedDisplay(among:)` (which returns the stored last-used id when present), but `preferredSelection` — the path `VariantDetailView` uses — does not.

Design points (planner to finalize):
- **Tie-break toward last-used** in `preferredSelection`: add a `lastUsed: UUID?` parameter and, among the dimension `matches`, prefer in order: (1) `current` if it's one of the matches; (2) `lastUsed` if it's one of the matches; (3) `matches.first`. This keeps the "don't move a deliberate current selection" behavior while stopping the arbitrary flip between equal-dimension displays. Keep `preferredSelection` pure/testable (it already takes a plain tuple list — extend the signature).
- **Wire the caller**: `VariantDetailView.updateSelectedDisplay()` passes the last-used id (e.g. `coordinator.resolveLastUsedDisplay(among: displays)?.id`, or a direct read of the stored id) as `lastUsed`. Confirm the display `@Query`/candidate tuple building is unchanged otherwise.
- **Remember from VariantDetailView too**: if the send screen doesn't already `rememberLastUsedDisplay` when the user changes the picker, add it (so choosing display A here makes A the last-used everywhere). Confirm this doesn't fight the `pendingCenteredDisplayID`/seed logic (#0064) — the centered-seed path sets `selectedDisplayID` explicitly and should still win when present.
- Consider whether the popover path needs any change — it uses `resolveLastUsedDisplay` and should already be correct; verify and note. If the bug also reproduces in the popover, extend the fix there.
- No persistence/schema change (last-used already persists via #0066).
