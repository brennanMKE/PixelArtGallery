## Description

Final phase of the display-first flow ([#0061](0061.md)): the user-facing UI. From a gallery item, present the known FT displays; tapping one auto-creates the aspect-fit, centered variant (via [#0063](0063.md)) and navigates to the send scene with the x/y offset seeded to the centered value, so the source appears pixelated to that display's dimensions ready to send — no dimension entry. The manual width/height form is demoted to a secondary "Custom size…" path (kept, not removed). Manual offset nudges during a send are clamped so the image can't be pushed off the display. Shared views for iOS and macOS with `#if os` tweaks.

## Long Description

Current navigation: `GalleryListView` (grid) → `GalleryDetailView` (item + variant list) → `VariantDetailView` (send, with the #0056/#0057 send section at top and live layer/x/y). Displays are only reachable via Settings → `DisplayRegistryView`, whose rows currently open an *edit* sheet on tap.

New flow entry: from `GalleryDetailView`, a primary "Send to Display" action lists the FT displays (reuse the `@Query`/`DisplayRow` pattern from `DisplayRegistryView`). Tapping a display calls `coordinator.createFittedVariant(for:display:)` ([#0063](0063.md)) and pushes `VariantDetailView` for the resulting variant, with the send controls seeded to the centered offset (via the #0063 helper) and the display preselected. The user can send immediately, adjust live (layer/x/y), edit the pixelation, or keep it — it's a normal saved variant tagged with the display.

## Plan

Prerequisite: [#0063](0063.md) must land first — this plan consumes `GalleryCoordinator.createFittedVariant(for:display:)` and its pure centering-offset helper (referred to below as `centeringOffset(imageWidth:imageHeight:displayWidth:displayHeight:)`). **Track the exact names #0063/#0062 actually land** and adjust call sites; the shape assumed here is `async throws -> Variant` with dedup built in.

### 1. New view: `DisplaySendPickerView` (new file `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/DisplaySendPickerView.swift`)

A pushed list of FT displays for a given gallery item.

- Properties: `let item: GalleryItem`, `let coordinator: GalleryCoordinator`; `@Query(sort: \FlaschenTaschenDisplay.discoveredDate, order: .reverse) private var displays` (same query as `DisplayRegistryView`).
- **Reuse `DisplayRow`**: it is currently `private struct DisplayRow` at the bottom of `DisplayRegistryView.swift`. Extract it to its own file `UI/DisplayRow.swift` (internal), keeping `DisplayRegistryView` using it unchanged. Optionally add a caption computed from the #0062 fit helper ("Fits 48×32, centered") — nice-to-have, skip if it adds friction.
- Each row is a `Button` (not `NavigationLink(value:)` — creation is async and must complete before the push):

  ```swift
  @State private var creatingDisplayID: UUID?   // row spinner + disables other rows
  @State private var pushTarget: FittedVariantPush?
  @State private var errorMessage: String?      // local alert; don't rely on the root view's currentError alert (it's covered by pushed views)

  struct FittedVariantPush: Hashable {
      let variant: Variant
      let displayID: UUID
  }
  ```

  Tap handler: guard `creatingDisplayID == nil`; set it; `Task { do { let variant = try await coordinator.createFittedVariant(for: item, display: display); pushTarget = FittedVariantPush(variant: variant, displayID: display.id) } catch { errorMessage = error.localizedDescription }; creatingDisplayID = nil }`. Show a `ProgressView` in the tapped row while creating.
- **Onward push** via the item-binding destination, attached to the picker's `List`:

  ```swift
  .navigationDestination(item: $pushTarget) { push in
      VariantDetailView(variant: push.variant, coordinator: coordinator,
                        centerOnDisplayID: push.displayID)
  }
  ```

  This is programmatic push without threading a `NavigationPath` up to `GalleryListView`'s stack, and it carries both the created variant and the chosen display id. (iOS 17+/macOS 14+; app targets iOS 18/macOS 15.)
- **Empty state**: mirror `DisplayRegistryView`'s — icon, "No Displays", a prominent **Scan** button (`coordinator.scanForDisplays()` with an `isScanning` spinner) and an **Add Manually** button presenting `DisplayEditorView(mode: .add)` wired to `coordinator.addManualDisplay(...)` exactly as `DisplayRegistryView` does (`DisplayEditorView` is internal, reusable as-is). No dead end.
- Platform: `#if os(iOS)` for `.navigationBarTitleDisplayMode(.inline)` / `.toolbarBackground(.hidden, for: .navigationBar)`, matching sibling views. Title: "Send to Display".
- Call `coordinator.configure(modelContext:)` in `.onAppear` only if scan/add are used from here (mirrors `DisplayRegistryView`); the coordinator instance arriving from `GalleryListView` is already configured, so this is belt-and-suspenders.

### 2. Entry point in `GalleryDetailView`: pushed destination, manual creation demoted

**Pushed destination, not a sheet** — tapping a display must push onward to `VariantDetailView` in the same stack; `VariantDetailView` is built as pushed content (nav title, toolbar, `.onDisappear` send-stop). A sheet would require dismiss-then-push gymnastics or a nested stack.

Mechanism: a Bool-driven destination on `GalleryDetailView`'s `List` (alongside the existing `navigationDestination(for: Variant.self)`):

```swift
@State private var isPickingDisplay = false
...
.navigationDestination(isPresented: $isPickingDisplay) {
    DisplaySendPickerView(item: item, coordinator: coordinator)
}
```

`isPresented:` is chosen over `for: some Hashable` because there is exactly one instance per detail view, `item`/`coordinator` are captured directly (no new Hashable route type), and both the toolbar button and the empty-state button can trigger it. (`GalleryItem.self` is already claimed by `GalleryListView`'s destination, so the raw model can't be the route value anyway.)

Affordance changes:
- Toolbar `primaryAction` becomes **"Send to Display"** (`Label("Send to Display", systemImage: "display")`, action `isPickingDisplay = true`).
- Add a **"Custom Size…"** button to the `secondaryAction` group (alongside Rename), calling the existing `startVariantCreation()`. `VariantCreationView` and the `GalleryListView` sheet stay untouched.
- Empty state (`item.variants.isEmpty`): change `actionLabel` to "Send to Display" with `action: { isPickingDisplay = true }`; adjust the message ("Pick a display to auto-create a fitted variant, or use Custom Size… from the toolbar."). `EmptyStateView` (UI/Components.swift) supports one action; custom size remains reachable via the toolbar.

### 3. `VariantDetailView`: one-shot centered seed without regressing the existing path

Add an explicit init with a defaulted parameter so every existing call site compiles unchanged and keeps today's stored-defaults behavior:

```swift
@State private var pendingCenteredDisplayID: UUID?

init(variant: Variant, coordinator: GalleryCoordinator, centerOnDisplayID: UUID? = nil) {
    self.variant = variant
    self.coordinator = coordinator
    _pendingCenteredDisplayID = State(initialValue: centerOnDisplayID)
}
```

An init parameter (vs a coordinator-held "pending offset") keeps the seed scoped to this one push — the coordinator is shared with the variant-list path, and coordinator state would need exactly-once clearing to avoid leaking a centered seed into an unrelated open. `@State(initialValue:)` makes it naturally one-shot.

Wire-in:
- `updateSelectedDisplay()`: if `pendingCenteredDisplayID` is non-nil and present in `displays`, set `selectedDisplayID` to it and return; otherwise fall through to `preferredSelection(...)`. (Necessary: a letterboxed fitted variant's dims don't equal the display's dims, so the #0055 geometry match would not pick the right display.)
- `seedSendDefaults()`: when `pendingCenteredDisplayID == selectedDisplayID` (non-nil), seed `sendLayer` from the display's stored default (clamped, as today) but seed `sendOffsetX/Y` from the #0063 centering helper — `centeringOffset(imageWidth: variant.targetWidth, imageHeight: variant.targetHeight, displayWidth: display.displayWidth, displayHeight: display.displayHeight)` — clamped into the (new, see §4) stepper ranges, then **consume**: `pendingCenteredDisplayID = nil`. All later reseeds (user switches display) fall back to stored defaults exactly as today. The existing `onChange` call order (`updateSelectedDisplay()` then `seedSendDefaults()`, `initial: true`) already fires both on first appearance; no new lifecycle hooks needed.

### 4. Offset overflow clamp (`offsetXRange`/`offsetYRange`)

Add a pure helper next to `clampedOffset(_:)` in `Models/FlaschenTaschenDisplay.swift`:

```swift
/// Stepper range keeping `offset + imageDimension ≤ displayDimension`.
/// Collapses to 0...0 when the image is as large as (or larger than) the display.
static func offsetRange(displayDimension: Int, imageDimension: Int) -> ClosedRange<Int> {
    0...max(0, displayDimension - imageDimension)
}
```

Rewrite the computed ranges in `VariantDetailView` (~lines 282–291):

```swift
private var offsetXRange: ClosedRange<Int> {
    guard let display = selectedDisplay else { return 0...0 }
    return FlaschenTaschenDisplay.offsetRange(
        displayDimension: display.displayWidth, imageDimension: variant.targetWidth)
}
```

(and the Y mirror). This upper-bounds nudges at `displayDim - imageDim`, and `max(0, …)` handles manual variants larger than the display (`0...0` — steppers effectively locked, image pinned at origin; fitted variants can never hit this by #0062's invariant). `seedSendDefaults()` already `min`s stored defaults into these ranges, so oversized stored defaults stay handled; reseed-on-display-change keeps a live value in range when switching displays. Update the two doc comments.

## Review notes

> 🔴 Bounced 2026-07-11 (claude-opus-4-8, review). Builds/tests/clamp/no-regression all pass, but the display-first centered seed is **overwritten on first appearance**, so the headline behavior (a centered send) does not actually happen — the send lands at the display's *stored* offset instead.

**Confirmed defect — centered seed is clobbered by the `onChange(of: selectedDisplayID)` reseed (`VariantDetailView.swift`).**

On the display-first path, `seedSendDefaults()` runs **twice** on first appearance, and the second run undoes the first:

1. The `onChange(of: displays.map { … }, initial: true)` handler (line 280) calls `updateSelectedDisplay()` — which sets `selectedDisplayID = pendingCenteredDisplayID` (a `nil → UUID` change) — then calls `seedSendDefaults()` directly. That direct call seeds `sendOffsetX/Y` to the **centered** value and clears `pendingCenteredDisplayID = nil`.
2. Because `selectedDisplayID` just changed from `nil` to the display's id, the separate `onChange(of: selectedDisplayID)` handler (line 292) fires on the next update and calls `seedSendDefaults()` **again**. `pendingCenteredDisplayID` is now `nil`, so it falls through to the stored-defaults branch and overwrites `sendOffsetX/Y` with `clampedOffset(display.offsetX/offsetY)`.

Net result: the centered offset is discarded; the send opens (and lands) at the display's stored offset (typically `0,0`, i.e. top-left), not centered. The `## Gotchas` note only considered the "switch away and back" re-trigger; it missed this first-appearance double-seed via the pre-existing `selectedDisplayID` `onChange`.

**Empirically reproduced.** A faithful SwiftUI `NSHostingView` probe mirroring this exact structure (`@State selectedDisplayID = nil`, `@State pending = displayID` set at init, `onChange(trigger, initial: true) { updateSelectedDisplay(); seedSendDefaults() }` + `onChange(selectedDisplayID) { seedSendDefaults() }`) settled to `FINAL_OFFSET=STORED` with `SEED_CALLS=2` — the centered value never survives. (The pre-existing code tolerated this double-seed only because both calls seeded identical stored defaults; #0064's centered branch makes the redundancy harmful.)

**What the next implementation pass must fix:** ensure the centered seed survives the `selectedDisplayID` reseed on first appearance. Options: consume `pendingCenteredDisplayID` from the `onChange(of: selectedDisplayID)` path rather than (or in addition to) the direct call so the last seed for the pending display is the centered one; or gate the stored-defaults reseed so it can't clobber a still-pending/just-applied centered seed; or drop the redundant direct `seedSendDefaults()` call and drive seeding solely off `selectedDisplayID`. Whatever the approach, verify with the same probe (or on device) that the offset ends at the centered value after the view settles, and that the ordinary variant-list path still ends at stored defaults.

## Root cause

Not a bug fix — a feature gap. Before this change there was no way to go from a gallery item straight to a display-sized, correctly-fitted send: the user had to (a) open Settings to note a display's native resolution, (b) return to the item and use the manual "Create Variant" width/height form to type those exact numbers (risking distortion if mistyped or if the display's aspect ratio wasn't square), and (c) manually figure out (or accept un-offset) x/y so the image would land centered rather than pinned to the origin. There was also no guard stopping a manual offset nudge from pushing the image partially or fully off the display's edge — `offsetXRange`/`offsetYRange` clamped only to `0...(displayDim - 1)`, ignoring the image's own size entirely.

## Gotchas

- `DisplaySendPickerView.FittedVariantPush` relies on `Hashable` synthesis for `.navigationDestination(item:)`; this works because SwiftData's `@Model` macro already gives `Variant` `Hashable`/`Equatable` conformance (proven elsewhere in the app by `GalleryDetailView`'s existing `navigationDestination(for: Variant.self)`, which requires the same) — no manual `==`/`hash(into:)` was needed or added.
- The centered-seed consumption is keyed on `pendingCenteredDisplayID == selectedDisplayID` inside `seedSendDefaults()`, not on "has this seed ever fired." If a user arrives via display-first, then switches to a *different* display and back to the originally-pushed one before the pending id was cleared, it would reseed centered again on that return — but since the pending id is cleared the very first time `seedSendDefaults()` runs with it selected (which is on first appearance, before any display switch is possible), this can't actually happen in practice; noted here only because it's the one place the "one-shot" behavior depends on state ordering rather than being structurally impossible.
