## Description

Second phase of the display-first flow ([#0061](0061.md)): add `GalleryCoordinator.createFittedVariant(for item:display:)` that uses the aspect-fit math from [#0062](0062.md) to pixelate a gallery item's source image to a display's dimensions (aspect-fit, no distortion) and persist a `Variant` sized to the fit dimensions with `associatedDisplayId` set — deduplicating against an existing fitted variant for the same display on that item. Also add a pure centering-offset helper the send path uses to seed the x/y offset that centers the variant on the display. No SwiftData schema change.

## Long Description

The existing `GalleryCoordinator.createVariant(for:width:height:associatedDisplayId:)` loads the item's original bytes via `FileStorageManager`, runs `PixelationEngine`, and stores a `Variant`. The new method wraps that with the fit computation: derive `fitW×fitH` from the source size and the display's `displayWidth×displayHeight` (via #0062), pixelate aspect-fit at those dims, and create the variant tagged with `display.id`. Because the fit dimensions become the variant's `targetWidth/targetHeight` and the centering offset is a pure function of (display dims, variant dims), **nothing new needs persisting** — the offset is computed when the send controls are seeded.

## Plan

Grounded in the current code: `GalleryCoordinator` (`PixelArtGalleryKit/Sources/PixelArtGalleryKit/ViewModels/GalleryCoordinator.swift`) is an `@Observable final class` on the package's default MainActor isolation, mutating through an injected `ModelContext` (`configure(modelContext:)`) and a lazily created / test-injected `FileStorageManager`. `createVariant(for:width:height:associatedDisplayId:)` (lines 217–265) is the proven pipeline: guard `modelContext` → load original bytes via `storage.load(filename: item.originalImagePath)` (throwing `GalleryCoordinatorError.originalImageMissing` on nil) → `PixelationEngine().process(imageData:targetWidth:targetHeight:)` → build `Variant`, set `variant.galleryItem` + `item.variants.append` → `insert` + `save`, with `currentError` + `AppLog.variant` on failure. `Variant` stores exactly `targetWidth`/`targetHeight`/`pixelGridData`/`associatedDisplayId`/`scaleFactor` — no `layer`/`offsetX`/`offsetY`, and this plan adds **no schema change**: the centering offset is derived at send time, never persisted (adding fields would force lightweight-migration-safe inline defaults for no benefit).

**Dependency (#0062).** This issue consumes #0062's pure aspect-fit helper (sketched there as `AspectFit.fit(sourceWidth:sourceHeight:displayWidth:displayHeight:) -> (width:height:offsetX:offsetY:)` in `ImageProcessing/`, `nonisolated`, results ≥ 1, invariant `offset + fit ≤ display`). #0062 has no landed code yet — **the implementer must use whatever names/shape #0062 actually lands** and adjust call sites accordingly; the steps below assume the sketched API.

### 1. `createFittedVariant(for:display:)` in `GalleryCoordinator`

```swift
@discardableResult
func createFittedVariant(
    for item: GalleryItem,
    display: FlaschenTaschenDisplay
) async throws -> Variant
```

Steps inside:

1. Guard `modelContext` (throw `.missingModelContext`, matching every other mutation).
2. **Determine source dimensions without decoding**: use `item.originalWidth`/`item.originalHeight`. Degenerate guard: import sets these to `0` when decode failed (see `createGalleryItem`), so if either is `≤ 0`, load the original bytes and decode dims via the existing private `loadImage(from:)` before computing the fit (the bytes are needed for pixelation anyway on the create path); if that also fails, throw `originalImageMissing`.
3. **Compute fit dims** via #0062's helper: `let fit = AspectFit.fit(sourceWidth: srcW, sourceHeight: srcH, displayWidth: display.displayWidth, displayHeight: display.displayHeight)` → `fitW = fit.width`, `fitH = fit.height`. Do not re-derive any arithmetic here.
4. **Dedup (before any pixelation I/O)**: if `item.variants` already contains a variant with `associatedDisplayId == display.id && targetWidth == fitW && targetHeight == fitH`, return it **as-is** (most recent by `createdDate` if several match, for determinism). Rationale for the composite key: the display-id alone is not enough — a display's geometry can change in place (mDNS re-scan updates `displayWidth/Height` via `mergeDiscoveredDisplays`, and `updateDisplay` edits it), and an older Option-A variant tagged with the same display at raw display dims must not masquerade as a fitted one when the aspect differs. Including the fit dims means a geometry change naturally produces a new fitted variant while a genuine re-selection is a no-op. Rationale for *not* refreshing pixel data: re-selecting a display shouldn't churn — it would silently discard user edits to the grid, re-run the engine, and dirty the store for zero visible benefit; explicit regeneration already exists (`updateVariantDimensions`).
5. **Create via the existing pipeline — do not duplicate it.** Refactor `createVariant(for:width:height:associatedDisplayId:)` to `@discardableResult ... -> Variant` (return the `Variant` it already builds; existing `try await coordinator.createVariant(...)` call sites compile unchanged), then have `createFittedVariant` call `try await createVariant(for: item, width: fitW, height: fitH, associatedDisplayId: display.id)` and return the result. Pixelating with the existing `process(...)` **at the fit dims** is aspect-preserving by construction, which is why the plain `process` path is correct here; if #0062 lands `processFitting(...)` returning grid + offset, it is *not* needed by this method (fit dims must be known up-front for dedup anyway) — note that in `## Fix` if the reviewer asks.
6. Log via `AppLog.variant` (e.g. "Created fitted variant … fitW×fitH for display …" / "Reusing fitted variant …"), errors through `currentError` + rethrow, matching house style.

### 2. Centering-offset helper — lives with #0062's math, not the coordinator

```swift
nonisolated static func centeringOffset(
    imageWidth: Int, imageHeight: Int,
    displayWidth: Int, displayHeight: Int
) -> (x: Int, y: Int)
```

Put it on the **same pure type as #0062's fit helper** (e.g. `AspectFit` in `PixelArtGalleryKit/Sources/PixelArtGalleryKit/ImageProcessing/AspectFit.swift`), not on the coordinator: the coordinator is MainActor-isolated and this must stay pure/`nonisolated`/`Sendable`-friendly so #0064 (and any send path) can call it anywhere. **Reuse, don't re-derive**: if #0062's `fit` already computes offsets internally, factor its centering arithmetic so `fit` calls `centeringOffset(imageWidth: fitW, imageHeight: fitH, …)` — one source of truth for `(disp − img) / 2`. Clamp each component to `≥ 0` (an image larger than the display — possible for manually sized variants — yields `0`, mirroring `FlaschenTaschenDisplay.clampedOffset`), which preserves the invariant `offset + dim ≤ display` whenever `dim ≤ display`.

### 3. Tests (Swift Testing — `import Testing` / `@Test` / `#expect`, NOT XCTest)

Extend `PixelArtGalleryKit/Tests/PixelArtGalleryKitTests/ViewModels/GalleryCoordinatorTests.swift`, which already has the exact harness needed: in-memory `ModelContainer` (`makeContext()`), coordinator with temp-dir `FileStorageManager` (`makeCoordinator()`), real imported item via `makeItem(in:coordinator:)` (16×16 PNG from `makePNGData`).

- **Fit dims, not display dims**: square 16×16 source, display 64×32 (aspect differs) → variant `targetWidth == 32 && targetHeight == 32` (not 64×32), `pixelGridData.count == 32 * 32 * 4`, `associatedDisplayId == display.id`, and the returned `Variant` is the one fetched from the context.
- **Dedup**: call `createFittedVariant` twice with the same item + display → second call returns the same `variant.id`, `item.variants.count` stays `1` (and total `Variant` fetch count stays 1). Optionally: a second, different display creates a second variant (`count == 2`).
- **Aspect-matching display**: 16×16 source into a square display (e.g. 32×32) → fit == display dims, offset `(0, 0)` via the helper.
- **Centering-offset helper** (pure, no container needed — place with #0062's `AspectFit` tests if that file exists by then, else alongside): `(32, 32)` on `(64, 32)` → `(16, 0)`; odd remainders, e.g. `(10, 10)` on `(45, 35)` → `(17, 12)` with `#expect(x + 10 <= 45 && y + 10 <= 35)` (the invariant assertion); oversized image `(100, 10)` on `(45, 35)` → `(0, 12)` (clamped, never negative).

## Root cause

Not a bug — a feature gap. The coordinator had no path that combined the aspect-fit math ([#0062](0062.md)) with variant creation: `createVariant(for:width:height:associatedDisplayId:)` required the caller to already know exact target dimensions, and nothing computed those dimensions from a display's geometry, deduplicated against an already-fitted variant, or exposed a pure centering-offset helper for the send flow. [#0064](0064.md) needs both to seed a display-first "pick display → centered preview → send" UI.

## Fix

- Refactored `createVariant(for:width:height:associatedDisplayId:)` to `@discardableResult func createVariant(...) async throws -> Variant`, returning the `Variant` it already builds and persists. Existing call sites (`GalleryListView.swift`, `GalleryCoordinatorTests.swift`) are untouched — none captured the previous `Void` result, so this is source-compatible.
- Added `@discardableResult func createFittedVariant(for item: GalleryItem, display: FlaschenTaschenDisplay) async throws -> Variant` to `GalleryCoordinator.swift`:
  1. Guards `modelContext`, throwing `.missingModelContext` like every other mutation.
  2. Uses `item.originalWidth/originalHeight` as the source size; if either is `<= 0` (import-time decode failure), loads the original bytes via `FileStorageManager` and decodes dimensions with the existing private `loadImage(from:)`, throwing `.originalImageMissing` if that also fails.
  3. Computes `AspectFit.fit(sourceWidth:sourceHeight:displayWidth:displayHeight:)` (landed by #0062) to get `fit.width`/`fit.height` — no arithmetic re-derived here.
  4. **Dedups before pixelation**: filters `item.variants` for `associatedDisplayId == display.id && targetWidth == fit.width && targetHeight == fit.height`, returning the most-recently-created match as-is (no pixel-data refresh) if one or more exist.
  5. Otherwise calls the refactored `createVariant(for:width:height:associatedDisplayId:)` with the fit dimensions and `display.id`, returning its result.
  6. Logs via `AppLog.variant` ("Creating fitted variant…" / "Reusing fitted variant…"), routes errors through `currentError` and rethrows, matching house style.
- **Centering-offset helper**: `AspectFit.centeringOffset(imageWidth:imageHeight:displayWidth:displayHeight:) -> (x: Int, y: Int)` already landed with #0062 in `ImageProcessing/AspectFit.swift`, including the clamp-to-non-negative behavior this issue's plan called for. No coordinator-side duplicate was added — [#0064](0064.md) calls `AspectFit.centeringOffset` directly.
- **`PixelationEngine.processFitting` not used**: `createFittedVariant` needs the fit dimensions up front (before pixelation) to run the dedup check, so it goes through the plain `createVariant` → `PixelationEngine.process(imageData:targetWidth:targetHeight:)` path, which is aspect-preserving by construction once given fit-sized targets. `processFitting` remains available for callers that don't already know the fit dims ahead of time.
- No SwiftData schema change: `Variant` gained no new fields; the centering offset is computed on demand, never persisted.

## Gotchas

- `createFittedVariant`'s `catch` wraps `createVariant`'s own `catch`: if the underlying `createVariant` call fails, `currentError` is set and `AppLog.variant.error` logged twice (once by each method) before the error is rethrown to the caller. This is harmless (same message, no double side effects beyond logging) and mirrors how other layered coordinator methods already behave; noting it in case the reviewer wants it deduplicated later.
- The dedup key intentionally includes `targetWidth`/`targetHeight`, not just `associatedDisplayId` — a display's geometry can change in place (mDNS re-scan, manual edit) so an older non-fitted or differently-fitted variant tagged with the same display never masquerades as a match.
