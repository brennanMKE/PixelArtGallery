## Description

The variant preview (`PixelGridView`) doesn't show the full pixelated image and doesn't match the variant's aspect ratio/geometry — it's cropped. A 32×32 variant should render square and a 45×35 variant should render at 45:35, both fully visible, but instead the preview shows a cropped horizontal strip.

## Steps to reproduce

1. Open a variant's detail screen.
2. Observe the Preview: the image is cut off and its shape doesn't match the Dimensions shown (e.g. "32×32 px" but the preview is a wide strip).

## Root cause

`PixelGridView` renders the grid at a fixed 20pt-per-pixel inside a scroll view, so at "100%" a 32×32 grid is a 640×640 canvas shown in a ~355×170 window — you only see the cropped top-left region, and neither the aspect ratio nor the full image is visible.

## Expected behavior

- **Always show the full image by default** ("size to fit"): the whole grid fits the preview area at the correct aspect ratio.
- **Zoom in and out** (buttons + pinch), where 100% = fit.
- **When zoomed in, pan** around the image.
- **Select individual pixels** by tapping (still works when zoomed).
- **Reset** returns to size-to-fit and recenters.

## Notes

- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/PixelGridView.swift` — the fixed-`pixelSize` scrollable canvas; replace with a fit-based layout computed from the container size (GeometryReader).
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/ViewModels/PixelGridViewModel.swift` — `zoomLevel`/`pixelSize`/`displaySize`/`gridCoordinate(at:)` need reworking so 1.0 == fit and coordinate mapping accounts for the fitted cell size and centered/panned origin.
- Update `PixelGridViewModelTests` to the new container-aware coordinate API.
- `VariantDetailView` embeds `PixelGridView(...).frame(height: 260)`.

## Fix

Made `PixelGridViewModel` container-aware. `zoomLevel == 1.0` now means "size to fit" (and is the minimum), and all layout is derived from the container size passed in by the view: `fitScale(in:)` = `min(container.w/gridW, container.h/gridH)` (preserves aspect ratio), with `cellSize`/`contentSize`/`origin`/`clampedPan`/`gridCoordinate(at:in:)` built on top. `PixelGridView` now uses a `GeometryReader` + `Canvas` that fits the whole grid by default, centered. Zoom is `MagnifyGesture` (pinch) plus −/+ buttons (100% = fit); a `DragGesture(minimumDistance: 10)` pans only when zoomed in, clamped to content bounds; a `SpatialTapGesture` selects the pixel under the tap; and a "Size to Fit" button resets zoom + pan. Preview height bumped 260 → 340. Off-screen cells are culled while drawing.

## Verification

- `cd PixelArtGalleryKit && swift test` — 72 tests, 0 failures, including the reworked `PixelGridViewModelTests`: a 32×32 grid fits square (cell = container/32, origin 0), a **45×35 grid fits at 45:35 with vertical letterboxing** (content 450×350 in a 450×450 container, origin.y = 50) — directly asserting the reported bug is fixed — plus zoom clamping (can't go below fit, clamps to max, reset zeroes pan) and tap selection.
- `xcodebuild ... 'platform=macOS' CODE_SIGNING_ALLOWED=NO build` and `... 'platform=iOS Simulator,name=iPhone 17 Pro' build` — both BUILD SUCCEEDED.
- Visual: rendered a synthetic 45×35 variant (four equal palette quadrants) via a temporary harness on the iOS Simulator and screenshotted it — the **full image** shows at the **correct aspect ratio** (wider than tall, side margins, four equal quadrants, no cropping), with grid lines and the "100% / Size to Fit" controls. The harness was removed afterward.

## Gotchas

`zoomLevel` semantics changed from "points-per-pixel × 20" to "multiple of size-to-fit", so 100% now means the whole image is visible (not 20pt/pixel). Coordinate mapping must use the same centered+panned `origin` the Canvas draws with, or taps select the wrong cell.
