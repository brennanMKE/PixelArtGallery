# 0031 — macOS window has no minimum size and can shrink to an unusable layout

## Description

The `WindowGroup` presents `ContentView` with no minimum frame, so the macOS window can be resized down to a sliver where the gallery list, detail views, and toolbar collapse into an unusable layout. Standard Mac practice is to set a sensible minimum content size.

## Steps to reproduce

1. Run the macOS app.
2. Drag the window's corner to shrink it — it collapses far below a usable size.

## Expected behavior

The window resists shrinking below a reasonable minimum (roughly 640×480 or similar) so all primary screens stay usable.

## Actual behavior

No minimum; content squashes arbitrarily.

## Notes

- Apply `#if os(macOS) .frame(minWidth:…, minHeight:…)` on `ContentView` in `PixelArtGallery/PixelArtGalleryApp.swift` (or on `GalleryListView`). Keep it out of the iOS build path.

## Root cause

`PixelArtGalleryApp.swift` presented `ContentView` in a bare `WindowGroup` with no frame constraints, so AppKit allowed the macOS window to be resized down to a sliver where the gallery list, detail views, and toolbar collapsed.

## Fix

In `PixelArtGallery/PixelArtGalleryApp.swift`, applied `.frame(minWidth: 640, minHeight: 480)` to `ContentView()` inside the `WindowGroup`, and added `.defaultSize(width: 1000, height: 700)` on the scene for a pleasant first-launch size. Both are wrapped in `#if os(macOS)` so the iOS build path is unchanged.

## Verification

- `cd PixelArtGalleryKit && swift test` — Executed 73 tests, with 0 failures.
- `xcodebuild -project PixelArtGallery.xcodeproj -scheme PixelArtGallery -destination 'platform=macOS' CODE_SIGNING_ALLOWED=NO build` — BUILD SUCCEEDED.
- `xcodebuild -project PixelArtGallery.xcodeproj -scheme PixelArtGallery -destination 'platform=iOS Simulator,name=iPhone 17 Pro' CODE_SIGNING_ALLOWED=NO build` — BUILD SUCCEEDED.
- Functional: built signed, launched the .app. CGWindowList reported the window at exactly 1000x700, confirming the new `.defaultSize` is active in the running binary. Window screenshot (via `screencapture -l`) shows the gallery rendering normally. Direct resize-to-below-minimum could not be exercised: `osascript` System Events was denied assistive access (-1719, same as prior issues), the app is not AppleScript-scriptable (bounds errors -1728/-10006), and synthetic CGEvent drags were silently ignored (even a legal 800x600 resize did not move the window), so the 640x480 clamp itself rests on the observed default size plus code review. App was quit afterward.

## Gotchas

- Synthetic CGEvent mouse drags are silently dropped without Input Monitoring/Accessibility permission — a no-op resize result is not evidence of a minimum size; verify event injection works with a legal resize first.
