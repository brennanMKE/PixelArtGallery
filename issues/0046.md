## Description

Running the app on the iOS Simulator from Xcode fails with:

```
Invalid argument
Domain: NSPOSIXErrorDomain
Code: 22
Failure Reason: Invalid argument
```

Reproduced outside Xcode: a **signed** Debug build installed via `simctl` is refused by SpringBoard (`FBSOpenApplicationServiceErrorDomain code=1, request denied by service delegate (SBMainWorkspace)`), while an unsigned (`CODE_SIGNING_ALLOWED=NO`) build of the same code launches fine — so the failure is in signature validation, not the app.

## Root cause (diagnosed)

#0039 created `PixelArtGallery/Info-macOS.plist` inside the app target's folder, which is an Xcode file-system-synchronized group — every file in it is automatically treated as a target resource. The macOS build consumes the file as its `INFOPLIST_FILE`, but the iOS build copies it into the bundle root as a plain resource, leaving the built iOS app with two root-level `Info*.plist` files. That extra plist corrupts the bundle's identity for code signing — `codesign -d` on the built iOS app reports `Executable=…/PixelArtGallery.app/Info-macos.plist` — so SpringBoard rejects the signed app at launch. Xcode always signs, hence the launch failure from Xcode; `CODE_SIGNING_ALLOWED=NO` command-line builds masked the bug.

## Expected behavior

The macOS Info.plist lives outside the synchronized folder (e.g. `Configuration/Info-macOS.plist`, next to the xcconfigs that reference it), the iOS bundle contains only its own generated `Info.plist`, and a signed Debug build installs and launches on the simulator from Xcode.

## Notes

- Fix: move the file to `Configuration/Info-macOS.plist` and update `INFOPLIST_FILE[sdk=macos*]` in `Configuration/macOS.xcconfig`; sweep docs (`scripts/RELEASE.md`, `scripts/SPARKLE.md`, resolved issue references stay historical) for the old path.
- Verify with a SIGNED iOS build: bundle contains no `Info-macOS.plist`, `codesign -d` reports the real executable, and `simctl install` + `launch` succeeds. Also confirm the macOS build still carries the Sparkle keys and check the macOS bundle for stray copies.

## Fix

Moved the file out of the synchronized folder: `git mv PixelArtGallery/Info-macOS.plist Configuration/Info-macOS.plist` (the `Configuration/` folder is a plain group holding the xcconfigs, not file-system-synchronized, so nothing in it becomes a bundle resource). Updated `INFOPLIST_FILE[sdk=macos*]` in `Configuration/macOS.xcconfig` to the new path and the two path references in `scripts/SPARKLE.md`. No other references existed (`scripts/preflight.sh` reads the built bundle's plist, not the source file). Also reset the stale Xcode launch state that the corrupted builds left behind (quit Xcode, cleared the project's DerivedData, shut down simulators, restarted CoreSimulatorService) — the stale state alone was not the cause, but fresh state was needed to verify cleanly.

## Gotchas

- `PixelArtGallery/` is an Xcode 16 file-system-synchronized group: **any file added to that folder silently becomes a target resource on every platform**. Config-style files (plists consumed by build settings, templates, etc.) must live outside it (`Configuration/` is the natural home) or get a membership exception.
- The failure only manifests on SIGNED builds — `CODE_SIGNING_ALLOWED=NO` command-line builds (the project's standard CI/verify invocation) launch fine with the corrupted bundle, which is exactly how #0039's verification missed it. When a change touches bundle structure, verify with a signed build and an actual `simctl launch`, not just BUILD SUCCEEDED.
- Xcode surfaces the SpringBoard refusal as an unhelpful `NSPOSIXErrorDomain Code=22 "Invalid argument"`; the real error (`FBSOpenApplicationServiceErrorDomain … denied by service delegate (SBMainWorkspace)`) only shows up via `simctl launch`.
