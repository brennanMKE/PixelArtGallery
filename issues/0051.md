# 0051 — FT layer is never applied: the footer's leading 0x00 byte makes the server ignore the offset, so images always land on layer 0

| | |
|---|---|
| **Status** | open |
| **Module** | Networking |
| **Platform** | All |
| **First seen** | 2026-07-07 |

## Description

Images sent to a Flaschen Taschen display always paint on layer 0 (the background), regardless of the layer chosen in Settings or the Send stepper (#0047). The app encodes the offset/layer as a trailing footer that begins with a `0x00` byte, and the reference FT server's parser bails on that byte before it ever reads the layer. On a display running demos on higher layers, the app's image is composited *behind* them and appears to do nothing.

## Steps to reproduce

1. Send any variant to an FT display that is also showing demo content on a non-background layer.
2. The app reports success, but nothing visibly changes on the panel.

## Expected behavior

The chosen layer (Settings default 5, or the Send stepper value, 1–15) is actually applied, so the image paints on that layer and appears in front of lower layers.

## Actual behavior

The image always lands on layer 0. The layer value never reaches the server.

## Long Description

`FTDisplayClient.makePacket` builds the datagram as `PPM` + `footerStart (0x00)` + `"<x>\n<y>\n<z>\n"` (`Networking/FTDisplayClient.swift`, ~lines 172, 202–204).

The reference server parses trailing offsets in `server/ppm-reader.cc`:

```c
info->offset_x = readNextNumber(&start, end, NULL);
if (start != NULL) info->offset_y = readNextNumber(&start, end, NULL);
if (start != NULL) info->layer    = readNextNumber(&start, end, NULL);
```

and `readNextNumber` does `skipWhitespace()` (which uses `isspace()`) then `strtol()`. `0x00` is **not** `isspace`, so it is not skipped; `strtol` on a leading `0x00` converts nothing, so `readNextNumber` returns 0 **and sets `start = NULL`**. With `start == NULL`, `offset_y` and `layer` are never read — the layer defaults to 0 (background).

Verified against upstream `hzeller/flaschen-taschen`:
- `server/ppm-reader.cc` — `parseOffsets`/`readNextNumber` as above; no NUL separator is expected.
- `api/lib/udp-flaschen-taschen.cc` — the reference client encodes offset/layer as a PPM **header comment** `#FT: <x> <y> <z>` (written into the header as `"P6\n%d %d\n#FT: %d %d %d\n255\n"`), not a trailing footer, and never emits a `0x00`.
- `doc/protocols.md` — trailing offsets are "decimal numbers, separated by space", x then y then z; z=0 is background, higher z overlays; unrefreshed layers fade out over time (which is why #0050's periodic resend is needed).

## Fix options

1. **Minimal:** drop the `0x00` prefix so the footer is `"<x>\n<y>\n<z>\n"`. This parses correctly (x, y, layer) since the server delimits the footer by byte count, not a marker byte.
2. **Canonical (preferred):** stop appending a trailing footer and instead emit the offset as a PPM header comment `#FT: <x> <y> <z>` between the `W H` line and the `255` maxval — matching every standard FT client. This is the most interoperable form.

Either way, update `FTDisplayClientTests` (the footer-bytes assertions) to lock in the corrected format, and add a case asserting the layer value is present and parseable in the emitted bytes.

## Notes

- Independent of this bug: to appear *in front* of demos, the app's layer must be higher than the demo layer (max 15). Once the layer is honored, a default of 5 will still be hidden if demos run on layers ≥ 5 — worth revisiting the default/guidance after this fix.
- The current code comment on `footerStart` ("allowing the receiver to distinguish it from PPM pixel data") is the root misconception — the server distinguishes the footer by length, so the marker byte is not only unnecessary but actively breaks parsing.
