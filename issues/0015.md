## Description

The PRD requires managing variants within a gallery item: view, edit dimensions, delete, or duplicate. Today `GalleryDetailView` only lists variants and navigates to them — there are no delete/duplicate/edit affordances. `GalleryCoordinator.deleteVariant(id:)` exists but mutates an in-memory relationship without a `ModelContext` and isn't surfaced in the UI.

## Expected behavior

From a gallery item the user can delete a variant (with confirmation), duplicate it, and edit its target dimensions (regenerating its pixel data). All operations persist through SwiftData and update the list immediately.

## Root cause

Variant management was only half-built. `GalleryDetailView` listed variants and
(from #0002) had a bare destructive context-menu "Delete Variant" with no
confirmation, but there was no way to duplicate a variant or change its target
dimensions. `GalleryCoordinator` had a context-backed `deleteVariant` but no
`duplicate` or `edit-dimensions` operations, so the PRD §2 "Variant Management"
requirement (view, edit dimensions, delete, duplicate) was incomplete.

## Fix

Added two coordinator methods, both persisting through the injected
`ModelContext`:

- `duplicateVariant(_:)` — creates a new `Variant` on the same parent item,
  copying `targetWidth`/`targetHeight`, `pixelGridData`, `scaleFactor`, and
  `associatedDisplayId`, then inserts + saves. A pure duplicate copies the
  existing pixel data verbatim rather than re-running the engine.
- `updateVariantDimensions(_:width:height:)` — reloads the parent item's
  original bytes via `FileStorageManager`, re-runs `PixelationEngine` at the new
  size, replaces `pixelGridData`, updates `targetWidth`/`targetHeight`, and
  saves (mirrors `createVariant`).

`GalleryDetailView` now wraps delete in a `.confirmationDialog`, and exposes
Edit Dimensions / Duplicate / Delete from the context menu and (under
`#if os(iOS)`) trailing swipe actions. Edit is presented via a new lightweight
`VariantEditDimensionsView` sheet that mirrors `VariantCreationView`.
