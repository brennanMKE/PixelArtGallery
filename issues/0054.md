## Description

It is difficult to manage FT displays manually — the user has to be in a specific context to access and edit them. Display management should just be a part of Settings on macOS and the equivalent settings view on iOS. It should be possible to edit a display after it is created; currently none of them appear editable, so the only way to change one is to delete it and add it again.

## Plan

### Current structure / root cause

There is no edit path for a display because the two surfaces that touch displays were each built for a narrower purpose:

- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/SettingsView.swift` — backs the macOS `Settings` scene (wired in `PixelArtGallery/PixelArtGalleryApp.swift`, `Settings { SettingsView() }`) and the iOS gear-button sheet in `GalleryListView.swift` (~line 294). It edits **only** the seeded `source == "default"` record (its `@Query` filters on `"default"`), so manually added or mDNS-discovered displays are invisible from Settings.
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/DisplayRegistryView.swift` — the full registry (all displays, scan, add, delete), but it is reachable only via the "Displays" toolbar `NavigationLink` (`GalleryRoute.displays`) inside the gallery list, and its only mutation beyond delete is a rename **alert** (`beginRename` → `coordinator.renameDisplay`). Host, port, and dimensions are not editable.
- `ManualDisplayEntryView.swift` is add-only: it collects text into the unit-tested `ManualDisplayInput` validator and calls `onAddDisplay`, but has no concept of prefilling from, or writing back to, an existing `FlaschenTaschenDisplay`.
- `GalleryCoordinator.swift` (ViewModels) has `addManualDisplay` / `renameDisplay` / `deleteDisplay` but no general `updateDisplay`.

So the fix is structural: make the registry the thing Settings shows, and turn the manual-entry form into a reusable add/edit editor.

### Approach

1. **Reusable editor** — refactor `ManualDisplayEntryView.swift` into a `DisplayEditorView` usable in two modes: *add* (current behavior, defaults prefilled) and *edit* (fields prefilled from an existing `FlaschenTaschenDisplay`, confirm button "Save"). Keep `ManualDisplayInput` as the single validation path. Include the existing `layer` stepper (bounded by `FlaschenTaschenDisplay.layerRange`, clamped via `clampedLayer`) so the form already has the "per-display defaults" section that #0056 will extend with x/y offset fields — group Name/Host/Port/Width/Height/Layer so x/y slot in without another restructure.
2. **Coordinator mutation** — add `GalleryCoordinator.updateDisplay(_:with:)` mirroring `addManualDisplay` (guard on `modelContext`, apply the `ManualDisplayInput.Validated` values plus layer, `save()`, log via `AppLog.ftDiscovery`, surface failures through `currentError`). The existing rename alert in `DisplayRegistryView` becomes redundant — replace it with an "Edit" action (context menu + row tap / swipe) that presents the editor sheet prefilled. `renameDisplay` can stay for now or be folded into `updateDisplay`.
3. **Settings hosts the registry (macOS)** — restructure `SettingsView` so display management is a first-class part of Settings: embed the full registry (list of all displays with scan / add / edit / delete) rather than only the default record. Since the `Settings` scene has no `GalleryCoordinator`, `SettingsView` owns one (`@State private var coordinator = GalleryCoordinator()` — the registry already calls `coordinator.configure(modelContext:)` in `onAppear`, and the scene already gets `.modelContainer(modelContainer)` in `PixelArtGalleryApp.swift`). The default display loses its special-cased editor: it appears as an ordinary editable row (source label "Default"); keep a "Restore Default Display" affordance for when it has been deleted (reuse `FlaschenTaschenDisplay.makeDefault()` / the seeding logic). Size the scene appropriately (`minWidth`/`minHeight`); a `TabView` (General / Displays) is acceptable if the single form gets crowded, but is not required.
4. **iOS equivalent** — the gear-button sheet in `GalleryListView.swift` presents the same reworked `SettingsView` inside its existing `NavigationStack`, so displays are managed from Settings on iOS too (editor presented as a push or nested sheet). Remove the now-duplicate "Displays" toolbar `NavigationLink` and the `GalleryRoute.displays` destination from `GalleryListView` so Settings is the single home for display management (per the issue's expected behavior); no extra macOS affordance is needed since the ⌘, Settings scene already exists.
5. **Docs/comments** — update the doc comments on `SettingsView` / `DisplayRegistryView` that currently describe the old split.

## Resolution notes

> 🟢 Resolved 2026-07-09 — Displays are now editable in place via a reusable add/edit `DisplayEditorView` backed by `GalleryCoordinator.updateDisplay`, and display management (scan/add/edit/delete, plus default restore) is hosted in the macOS Settings scene and the iOS gear sheet; verified with 112 passing tests (5 new) and green macOS + iOS builds.

## Root cause

Confirmed the plan's diagnosis: `SettingsView` only ever edited the single
seeded `source == "default"` record, `DisplayRegistryView` (the only view that
listed every display) was reachable only via a "Displays" toolbar
`NavigationLink` outside Settings, and its only mutation beyond delete was a
rename-only alert — host, port, dimensions, and layer had no edit path at all.
`ManualDisplayEntryView`/`ManualDisplayInput` handled add-only input, and
`GalleryCoordinator` had no `updateDisplay`. So editing any field other than
the name required deleting and re-adding the display.

## Fix

- **`DisplayEditorView.swift`** (renamed from `ManualDisplayEntryView.swift`) —
  a single reusable form with an `add`/`edit(FlaschenTaschenDisplay)` mode.
  Edit mode prefills host/port/name/width/height/layer from the target
  display and reads "Save" instead of "Add". `ManualDisplayInput` (validation)
  is unchanged and used by both modes. A new "Defaults" section adds the
  per-display layer `Stepper` (previously only in `SettingsView`) so the
  add/edit form already has the "defaults" grouping #0056 will extend with
  x/y offsets.
- **`GalleryCoordinator.swift`** — added `updateDisplay(_:with:layer:)`
  (applies validated fields + clamped layer, saves, logs, throws
  `.missingModelContext` if unconfigured) and `restoreDefaultDisplay()`
  (unconditionally re-inserts `FlaschenTaschenDisplay.makeDefault()`, unlike
  `seedDefaultDisplayIfNeeded()` which only fires on a fully empty registry).
  `addManualDisplay` gained an optional `layer` parameter (defaulted, so
  existing callers/tests are unaffected).
- **`DisplayRegistryView.swift`** — rows now open the editor (tap or
  context-menu "Edit") prefilled for that display instead of the old
  rename-only alert; the alert and its state (`renameTarget`/`renameText`)
  are gone. Added a "Restore Default Display" banner shown whenever no
  display has `source == defaultSource`. Retitled to "Settings" since it's
  now the content Settings shows, not a separate "Displays" screen.
  `renameDisplay` itself is left in place on the coordinator (still covered by
  its own test) rather than folded into `updateDisplay`, per the plan's "can
  stay for now" option.
- **`SettingsView.swift`** — rewritten to own its own `GalleryCoordinator`,
  wrap `DisplayRegistryView` in a `NavigationStack`, and add an iOS-only
  "Done" toolbar button (`@Environment(\.dismiss)`; a no-op on macOS's
  non-modal Settings scene, so it's `#if os(iOS)`-gated). The default display
  is no longer special-cased — it's an ordinary row labeled "Default".
- **`GalleryListView.swift`** — removed `GalleryRoute`/`.displays` and the
  "Displays" toolbar `NavigationLink`; the iOS gear-button sheet now presents
  `SettingsView()` directly (it supplies its own nav stack/title/Done button),
  replacing the wrapper `NavigationStack` that used to live in
  `GalleryListView`.
- **`PixelArtGalleryApp.swift` / `ImportNamingView.swift`** — doc-comment-only
  updates reflecting the new Settings scope and the `ManualDisplayEntryView` →
  `DisplayEditorView` rename.

**Deviation from plan:** the plan's test list named
`ManualDisplayInputTests.swift` "edit-mode prefill/validation cases" — but
`ManualDisplayInput.validate()` is mode-agnostic and already fully covered
there, and view-level prefill is `@State` inside a SwiftUI view body, not
independently unit-testable. Instead the edit path is covered where the real
logic lives: `GalleryCoordinator.updateDisplay` (persistence + layer
clamping) in `DisplayMergeTests.swift`, alongside new tests for
`restoreDefaultDisplay`.

## Steps to reproduce

1. Add a display manually (via the current display registry / manual entry flow).
2. Try to change any of its values (name, hostname, dimensions, …) afterwards.
3. Try to reach display management from Settings.

## Expected behavior

Displays are listed under Settings (macOS Settings scene, and the equivalent settings view on iOS), and any display can be selected and edited in place after creation.

## Actual behavior

Display management is only reachable from a specific context in the app, and existing displays cannot be edited — they must be deleted and re-added to change anything.
