## Description

The Send UI is hidden down at the bottom of the variant view and seems difficult to get to. Sending is the primary reason to move to a variant, so it should be presented near the top. From there it should be quick to tap the Send button without messing with the layer and offset values. And while a send is active it should be possible to adjust the layer, x and y values so the next frame sends the changed values ‚Äî the user should not have to stop a send to change them.

## Plan

**Depends on #0056** ‚Äî the x/y offset fields (and per-display stored defaults for layer/x/y) come from that issue, so implement this one *after* #0056 lands; the "live x/y" parts of this plan apply to whatever controls #0056 adds.

### Current state and why parameters are frozen

`PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/VariantDetailView.swift` lays out its `ScrollView` as `infoCard ‚Üí previewSection (340pt tall) ‚Üí exportSection ‚Üí sendSection ‚Üí status banners`, so the send controls sit below the fold on both platforms (the same view serves macOS and iOS). Parameters are frozen during a send for two reasons:

1. `handleSendToDisplay()` reads `sendLayer` **once** at start, clamps it into a local `let layer`, and captures it (with host/port/width/height/scaleFactor) into the `ActiveSend` struct (#0053) and into the send loop's closure ‚Äî the `while !Task.isCancelled` loop always sends `offset: (x: 0, y: 0, z: layer)` from that one-time capture.
2. The layer `Stepper` is explicitly `.disabled(isSending || selectedDisplay == nil)`, a leftover from #0047/#0050 when mid-send changes had no effect anyway.

`stopSending()` (#0053) clears by sending 3 all-black frames to the *captured* `ActiveSend` layer. That is correct today precisely because the loop can never change layer; once layer/x/y become live, a mid-send change would strand the last frame on the old layer (or old offset region) forever ‚Äî the FT server only drops it on its ~15s layer timeout, and black-is-transparent clearing only reaches the layer we last targeted.

### Files and functions

- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/VariantDetailView.swift` ‚Äî `body` (section order), `sendSection` (control layout + `disabled` modifiers), `handleSendToDisplay()` (loop reads), `ActiveSend` (must track last-*painted* layer/x/y, not start-of-send values), `stopSending()` (clear target), plus a new shared clear-frame helper.
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Networking/FTDisplayClient.swift` ‚Äî no changes expected; `send(...offset:)` already takes x/y/z per call (x/y wiring itself is #0056).
- `PixelArtGalleryKit/Tests/PixelArtGalleryKitTests/Networking/FTDisplayClientTests.swift` ‚Äî existing all-black clear-frame packet test from #0053 keeps covering the clear payload.

### Approach

1. **Move the send section to the top.** Reorder `body`'s `VStack` to `infoCard ‚Üí sendSection ‚Üí (send/status banners) ‚Üí previewSection ‚Üí exportSection` so the send controls and their banners are visible without scrolling on both macOS and iOS (one shared view ‚Äî no platform forks needed).
2. **One-tap send with defaults.** Within `sendSection`, put the display picker and the prominent **Send Now / Stop Sending** button first; layer (and, post-#0056, x/y) sit below as optional adjustments. The existing `seedSendLayer()` / `onChange(of: selectedDisplayID)` seeding (extended by #0056 to seed x/y from the display's stored defaults) already means a bare tap sends with the display's defaults ‚Äî verify nothing forces the user through the steppers/fields first.
3. **Live per-frame reads of layer/x/y.** Remove `isSending` from the controls' `.disabled(...)` (keep `selectedDisplay == nil`). In the send loop ‚Äî a `Task {}` created in the view, so it runs on the main actor and may read `@State` safely ‚Äî read and clamp the *current* `sendLayer` (and x/y) at the top of each iteration and pass them to `client.send(offset:)`, instead of using the start-of-send capture. Endpoint (host/port) and geometry stay captured at start; only layer/x/y go live. Display picker stays disabled (or its changes ignored) during a send, matching #0053's rationale.
4. **Clear a layer/region the user switches away from.** Factor #0053's clear (3√ó all-black frame, best-effort, detached-capable) into a helper, e.g. `sendClearFrame(host:port:width:height:scaleFactor:x:y:layer:)`. In the loop, when this frame's layer or x/y differs from the last-painted values, first send a clear frame to the **old** layer at the **old** x/y (same width/height, so the old region is erased), then paint the new values. After each successful frame, update `activeSend` so it always holds the last-*painted* layer/x/y ‚Äî `stopSending()` then keeps working unchanged, clearing exactly what is on the display, and the #0052 `.onDisappear` path inherits the same correctness. Black remains transparent only on layers 1‚Äì15; `clampedLayer` already guarantees we never paint or clear layer 0.
5. **Keep the stop-clear semantics from #0053 intact**: clear only on user-initiated stop (the loop's `defer` still nils `activeSend` on error exit), clears stay detached so they survive view teardown.

## Root cause

`VariantDetailView`'s `body` laid out `infoCard ‚Üí previewSection (340pt) ‚Üí exportSection ‚Üí sendSection`, pushing the send controls (and their status banners) below the fold on both platforms even though sending is the primary reason to open a variant. Separately, the layer/x/y `Stepper`s carried `.disabled(isSending || selectedDisplay == nil)` ‚Äî a leftover from before #0056 added the offset fields, when mid-send changes had no effect anyway ‚Äî and `handleSendToDisplay()`'s continuous-send loop closed over `layer`/`offsetX`/`offsetY` captured once at the top of the function, so the `while !Task.isCancelled` loop always resent the same start-of-send values regardless of later stepper edits. Because the loop could never change layer/offset, `ActiveSend` only ever needed to record the start-of-send values for `stopSending()`'s clear frame; once layer/x/y become live per-frame, a mid-send switch would otherwise strand the previously painted layer/region (the FT server only drops it after its ~15s layer timeout).

## Fix

- Reordered `body` to `infoCard ‚Üí sendSection ‚Üí status banners ‚Üí previewSection ‚Üí exportSection` (one shared view, no platform forks needed) so the send controls are visible without scrolling on both macOS and iOS.
- Removed `isSending` from the layer/x/y `Stepper`s' `.disabled(...)` (kept `selectedDisplay == nil`). Added `.disabled(isSending)` to the display **Picker** ‚Äî it had no disabled modifier at all previously, which was an existing gap now worth closing since the endpoint is still captured once at send start and must not desync from the picker mid-send.
- `handleSendToDisplay()`'s send loop now reads and clamps the *current* `sendLayer`/`sendOffsetX`/`sendOffsetY` at the top of every iteration (endpoint host/port and the variant's pixel/geometry data stay captured once at start, per the plan). A local `painted: FTPaintTarget?` tracks what the loop last actually painted; when the next frame's layer/offset differs from `painted`, the loop first awaits a clear of the *old* target, then paints the new frame, then updates both `painted` and the `@State activeSend` to the newly painted values.
- Replaced the private `ActiveSend` struct with a new `nonisolated public struct FTPaintTarget` (in `Networking/FTPaintTarget.swift`) carrying the endpoint/geometry/layer/offset, plus two pure helpers: `requiresClear(beforePaintingLayer:offsetX:offsetY:)` and `painted(layer:offsetX:offsetY:)`. Marked `nonisolated` to opt out of the package's `.defaultIsolation(MainActor.self)` build setting, matching the existing convention for other cross-actor Sendable value types (`PixelGrid`, `DiscoveredFTDisplay`, etc.) ‚Äî without it, the type is implicitly MainActor-isolated and can't be read from a nonisolated test function or a detached task.
- Factored the 3x best-effort all-black clear-frame retry (previously inlined in `stopSending()`) into `FTDisplayClient.sendClearFrame(to:)`, an actor method shared by both the mid-send clear (awaited in the loop, so ordering is guaranteed) and `stopSending()`'s detached final clear.
- `stopSending()`, the #0053 stop-clear and #0052 `.onDisappear` path, is otherwise unchanged in behavior: it still captures `activeSend` before cancelling and clears exactly that target ‚Äî which now reflects the *last painted* layer/offset rather than the start-of-send values, so it keeps clearing precisely what's on the display after any number of mid-send switches.

No plan divergences beyond the two call-outs above (the picker's missing `.disabled`, and the `nonisolated` fix required for the new pure type to compile under this package's default MainActor isolation).

## Gotchas

- The package target sets `.defaultIsolation(MainActor.self)` in `Package.swift`, so any new Sendable value type meant to cross actor boundaries (used from a detached task, an actor method, or a plain test function) must be explicitly marked `nonisolated` or it becomes implicitly MainActor-isolated and fails to compile outside MainActor context ‚Äî this bit the first pass of `FTPaintTarget` and its test file.
- The mid-send clear is `await`ed in-line in the loop (not detached), so a layer/offset switch adds ~360ms (3 retries √ó 120ms) of latency before the new frame paints. This is intentional per the plan (guarantees clear-before-paint ordering) and only affects the frame immediately after a user-initiated switch, not steady-state refreshes.

## Steps to reproduce

1. Open a variant.
2. Scroll to find the send controls.
3. Start a continuous send, then try to change the layer (or offset) without stopping.

## Expected behavior

The send controls sit near the top of the variant view; tapping Send works immediately with the display's defaults; layer, x and y remain editable during an active send and take effect on the next frame.

## Actual behavior

The send controls are at the bottom of the view, and send parameters cannot be changed while a send is running.

Likely code: `VariantDetailView.swift` ‚Äî layout of the send section, and the continuous-send loop (see #0050/#0053), which captures its parameters into `ActiveSend` at start; live adjustment means the loop must read the current layer/x/y each frame instead (while the stop-clear from #0053 must still erase the layer(s) actually painted, including a layer switched away from mid-send).
