# 0057 — Send UI is buried at the bottom of the variant view and cannot be adjusted while sending

| | |
|---|---|
| **Status** | open |
| **Module** | UI |
| **Platform** | All |
| **First seen** | 2026-07-09 |

## Description

The Send UI is hidden down at the bottom of the variant view and seems difficult to get to. Sending is the primary reason to move to a variant, so it should be presented near the top. From there it should be quick to tap the Send button without messing with the layer and offset values. And while a send is active it should be possible to adjust the layer, x and y values so the next frame sends the changed values — the user should not have to stop a send to change them.

## Plan

**Depends on #0056** — the x/y offset fields (and per-display stored defaults for layer/x/y) come from that issue, so implement this one *after* #0056 lands; the "live x/y" parts of this plan apply to whatever controls #0056 adds.

### Current state and why parameters are frozen

`PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/VariantDetailView.swift` lays out its `ScrollView` as `infoCard → previewSection (340pt tall) → exportSection → sendSection → status banners`, so the send controls sit below the fold on both platforms (the same view serves macOS and iOS). Parameters are frozen during a send for two reasons:

1. `handleSendToDisplay()` reads `sendLayer` **once** at start, clamps it into a local `let layer`, and captures it (with host/port/width/height/scaleFactor) into the `ActiveSend` struct (#0053) and into the send loop's closure — the `while !Task.isCancelled` loop always sends `offset: (x: 0, y: 0, z: layer)` from that one-time capture.
2. The layer `Stepper` is explicitly `.disabled(isSending || selectedDisplay == nil)`, a leftover from #0047/#0050 when mid-send changes had no effect anyway.

`stopSending()` (#0053) clears by sending 3 all-black frames to the *captured* `ActiveSend` layer. That is correct today precisely because the loop can never change layer; once layer/x/y become live, a mid-send change would strand the last frame on the old layer (or old offset region) forever — the FT server only drops it on its ~15s layer timeout, and black-is-transparent clearing only reaches the layer we last targeted.

### Files and functions

- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/VariantDetailView.swift` — `body` (section order), `sendSection` (control layout + `disabled` modifiers), `handleSendToDisplay()` (loop reads), `ActiveSend` (must track last-*painted* layer/x/y, not start-of-send values), `stopSending()` (clear target), plus a new shared clear-frame helper.
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Networking/FTDisplayClient.swift` — no changes expected; `send(...offset:)` already takes x/y/z per call (x/y wiring itself is #0056).
- `PixelArtGalleryKit/Tests/PixelArtGalleryKitTests/Networking/FTDisplayClientTests.swift` — existing all-black clear-frame packet test from #0053 keeps covering the clear payload.

### Approach

1. **Move the send section to the top.** Reorder `body`'s `VStack` to `infoCard → sendSection → (send/status banners) → previewSection → exportSection` so the send controls and their banners are visible without scrolling on both macOS and iOS (one shared view — no platform forks needed).
2. **One-tap send with defaults.** Within `sendSection`, put the display picker and the prominent **Send Now / Stop Sending** button first; layer (and, post-#0056, x/y) sit below as optional adjustments. The existing `seedSendLayer()` / `onChange(of: selectedDisplayID)` seeding (extended by #0056 to seed x/y from the display's stored defaults) already means a bare tap sends with the display's defaults — verify nothing forces the user through the steppers/fields first.
3. **Live per-frame reads of layer/x/y.** Remove `isSending` from the controls' `.disabled(...)` (keep `selectedDisplay == nil`). In the send loop — a `Task {}` created in the view, so it runs on the main actor and may read `@State` safely — read and clamp the *current* `sendLayer` (and x/y) at the top of each iteration and pass them to `client.send(offset:)`, instead of using the start-of-send capture. Endpoint (host/port) and geometry stay captured at start; only layer/x/y go live. Display picker stays disabled (or its changes ignored) during a send, matching #0053's rationale.
4. **Clear a layer/region the user switches away from.** Factor #0053's clear (3× all-black frame, best-effort, detached-capable) into a helper, e.g. `sendClearFrame(host:port:width:height:scaleFactor:x:y:layer:)`. In the loop, when this frame's layer or x/y differs from the last-painted values, first send a clear frame to the **old** layer at the **old** x/y (same width/height, so the old region is erased), then paint the new values. After each successful frame, update `activeSend` so it always holds the last-*painted* layer/x/y — `stopSending()` then keeps working unchanged, clearing exactly what is on the display, and the #0052 `.onDisappear` path inherits the same correctness. Black remains transparent only on layers 1–15; `clampedLayer` already guarantees we never paint or clear layer 0.
5. **Keep the stop-clear semantics from #0053 intact**: clear only on user-initiated stop (the loop's `defer` still nils `activeSend` on error exit), clears stay detached so they survive view teardown.

### Verification

- `cd PixelArtGalleryKit && swift test` — all existing tests pass, including #0053's `allBlackFrameProducesBlackPixelsOnChosenLayer`; if the clear helper or per-frame parameter selection is factored into testable (non-view) code, add a test that a layer/offset change produces a clear to the old target before the paint to the new one.
- `xcodebuild -project PixelArtGallery.xcodeproj -scheme PixelArtGallery -destination 'platform=macOS' build` and `-destination 'platform=iOS Simulator,name=iPhone 17 Pro' build` — both must succeed (the layout move touches the shared view used by both platforms).
- Manual on the FT network: open a variant — send controls visible near the top; tap Send with no other input — frame appears using the display's defaults; while sending, step the layer (and change x/y) — the image moves on the next frame and the old layer/region clears; tap Stop or navigate away — the display clears.

## Steps to reproduce

1. Open a variant.
2. Scroll to find the send controls.
3. Start a continuous send, then try to change the layer (or offset) without stopping.

## Expected behavior

The send controls sit near the top of the variant view; tapping Send works immediately with the display's defaults; layer, x and y remain editable during an active send and take effect on the next frame.

## Actual behavior

The send controls are at the bottom of the view, and send parameters cannot be changed while a send is running.

## Notes

Likely code: `VariantDetailView.swift` — layout of the send section, and the continuous-send loop (see #0050/#0053), which captures its parameters into `ActiveSend` at start; live adjustment means the loop must read the current layer/x/y each frame instead (while the stop-clear from #0053 must still erase the layer(s) actually painted, including a layer switched away from mid-send). Depends on #0056 for the x/y offset fields existing at all.
