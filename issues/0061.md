# 0061 — Display-first variant flow: pick an FT display to auto-generate an aspect-fit, centered pixelation

| | |
|---|---|
| **Status** | in-progress |
| **Module** | UI / ImageProcessing / ViewModels / Networking |
| **Platform** | All |
| **First seen** | 2026-07-10 |

## Description

Umbrella for reworking variant creation from a manual "enter width/height" step into a **display-first** flow. Today the user must open a gallery item, tap "Create Variant", and type target dimensions before they can send anything. Instead: from a gallery item, list the known FT displays; tapping one navigates to a scene showing the source image pixelated to that display's dimensions, **aspect-fit (never stretched/distorted) and auto-centered** via an x/y offset, ready to send. The variant is created automatically from the chosen display; the user can still edit the pixelation and keep it as a saved variant. This ships for iOS and macOS (shared views).

## Long Description

Why: picking a display already determines the target dimensions, so making the user re-enter them is a redundant step, and the current pixelation *stretches* the source to the exact target (distorting aspect). The display-first flow removes the manual step and fixes distortion at once.

Approach (three shippable phases, each its own child issue):

1. **[#0062](0062.md) — Aspect-fit + centering math + aspect-fit pixelation** (foundation, no UI, no schema change): a pure helper that, given source `W×H` and display `W×H`, returns the aspect-fit image dimensions and the centering `(x, y)` offset (guaranteed `offset + fitDim ≤ displayDim`, so it never overflows), plus an aspect-fit pixelation path so a source is pixelated to the fit dimensions without distortion.
2. **[#0063](0063.md) — Coordinator: create a display-fitted, centered variant**: `GalleryCoordinator.createFittedVariant(for:display:)` using #0062 to produce a `Variant` sized to the fit dimensions with `associatedDisplayId` set (dedup against an existing fitted variant for that display), plus a pure centering-offset helper used to seed the send. No SwiftData schema change — fit dims are the variant's `targetWidth/targetHeight`; the centering offset is derived at send time.
3. **[#0064](0064.md) — Display-first UI (iOS & macOS)**: from a gallery item, list FT displays; tapping one auto-creates the fitted variant and opens the send scene with the offset seeded to the centered value; demote the manual dimension form to a secondary "Custom size…" path; clamp manual offset nudges so they can't push the image off the display.

Design decisions (grounded in the current code — see each child for detail):
- **Fit-sized grid + centering offset**, not baked letterbox bars. The FT wire protocol places a `W×H` image at `(x, y)` on the larger display canvas (`FTDisplayClient.send(offset:)`), so a `fitW×fitH` grid sent at the centering offset is exactly the primitive needed — no wasted black pixels, and it reuses the live layer/x/y send plumbing from #0056/#0057.
- **No schema migration**: `Variant` already stores `targetWidth/targetHeight`, `pixelGridData`, and `associatedDisplayId`. The centering offset is a pure function of (display dims, variant dims), computed when seeding the send controls — nothing new to persist.
- **Keep manual creation** as a secondary path; the display-first flow becomes the primary one.

Acceptance criteria for the umbrella:
- From a gallery item, the user can pick an FT display and, with no dimension entry, see the source pixelated aspect-fit + centered for that display and send it.
- No image distortion: a non-matching aspect ratio letterboxes via offset, not stretching.
- The auto-created variant is a normal saved variant (editable, deletable, re-sendable) tagged with its display.
- Works on iOS and macOS.
- Manual custom-dimension creation still available.

Out of scope (candidate future follow-ons, file only if wanted): batch "create variants for all displays" (PRD nice-to-have); persisting a per-variant manual offset override.
