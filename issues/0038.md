## Description

Integrate Sparkle 2.x so users on direct downloads get in-app updates instead of manual reinstalls: the SPM dependency, an updater controller, and a "Check for Updates…" menu item in the app menu.

## Acceptance criteria

- Sparkle (sparkle-project/Sparkle, 2.x) added as an SPM dependency **linked into the macOS build only** — the app target is multiplatform and Sparkle is macOS-only, so use a platform-conditioned target dependency (`.when(platforms: [.macOS])`) if it goes through `PixelArtGalleryKit`, or attach it directly to the app target for the macOS SDK. The iOS build must not link or embed Sparkle.
- An `UpdaterController` (wrapping `SPUStandardUpdaterController`, `#if os(macOS)`) with an `isConfigured` check on `SUFeedURL` in Info.plist — `checkForUpdates()` is a logged no-op and the menu item is disabled until the feed URL is wired (#0039), so this issue can land before hosting exists.
- A `Commands` block on the `WindowGroup` scene replaces `.appInfo` with "About Pixel Art Gallery" plus "Check for Updates…" invoking the controller (macOS only).
- Package tests still pass; both platform builds succeed; the built iOS app contains no Sparkle framework.

## Notes

- `PixelArtGalleryApp.swift` currently has no `Commands`; add them under `#if os(macOS)`.
- Sparkle ships as a binary XCFramework via SPM; its helper tools (`generate_keys`, `sign_update`) land under the build's SPM artifacts directory — #0039 and #0041 use them.
- Watch transitive-linking: nothing outside the macOS app should pick up Sparkle via the Kit (unit tests run on macOS and will link the Kit — keep the dependency scoped so `swift test` still works).

## Root cause

Feature gap: the direct-download macOS build had no in-app update mechanism — no Sparkle dependency, no updater controller, and no `Commands` block in `PixelArtGalleryApp.swift`, so users had to reinstall manually for every release.

## Fix

- `PixelArtGalleryKit/Package.swift`: added `sparkle-project/Sparkle` (`from: "2.6.0"`, resolved 2.9.4) with the product attached to the Kit target as `.product(name: "Sparkle", package: "Sparkle", condition: .when(platforms: [.macOS]))` — iOS never links it. `Package.resolved` is now tracked.
- New `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UpdaterController.swift`, wholly inside `#if os(macOS) && canImport(Sparkle)`: public final class (package-default `@MainActor`) wrapping `SPUStandardUpdaterController`. `public static let shared`; `isConfigured` = `Bundle.main` `SUFeedURL` is a non-empty string; `canCheckForUpdates` = `isConfigured && updater.canCheckForUpdates`; `checkForUpdates()` early-returns with an `AppLog.updates` info line when unconfigured. The controller is created with `startingUpdater: false` and `startUpdater()` is called only when configured — starting an unconfigured updater would surface Sparkle's automatic-check permission prompt and later feed errors (verified in Sparkle 2.9.4 source: `startUpdater` requires no feed URL, but scheduling does).
- `AppLog.swift`: new `Updates` category + `AppLog.updates` logger; `AppLogTests` and the PRD "Logging" category list updated to match (the test asserts the exact PRD set).
- `PixelArtGallery/PixelArtGalleryApp.swift`: `#if os(macOS)` `.commands { CommandGroup(replacing: .appInfo) { … } }` on the WindowGroup — "About Pixel Art Gallery" (`NSApplication.shared.orderFrontStandardAboutPanel`), Divider, "Check for Updates…" invoking `UpdaterController.shared.checkForUpdates()` with `.disabled(!UpdaterController.shared.isConfigured)`.

## Gotchas

- **MainActor default isolation vs Sparkle**: the Kit's `.defaultIsolation(MainActor.self)` makes `UpdaterController` implicitly `@MainActor`, which matches Sparkle's expectation that `SPUStandardUpdaterController` is driven from the main thread — no `nonisolated` escape hatches were needed. Sparkle 2.9.4's ObjC API imports without Sendable friction under Swift 6.2 here.
- **Do not start the updater while unconfigured**: `SPUStandardUpdaterController(startingUpdater: true, …)` with no `SUFeedURL` starts fine (feed URL is only required at check time) but schedules update sessions and shows Sparkle's "check automatically?" permission prompt. Keep `startingUpdater: false` + conditional `startUpdater()` until #0039 lands. Also note: if the host app were ever unsigned *and* keyless, `startUpdater()` fails with `SUNoPublicDSAFoundError` and Sparkle shows an error alert ~1s after launch (it does not crash).
- **Sparkle helper tools for #0039/#0041** (`generate_keys`, `sign_update`, `generate_appcast`, `BinaryDelta`) land at both:
  - `PixelArtGalleryKit/.build/artifacts/sparkle/Sparkle/bin/` (from `swift test`/`swift build`)
  - `~/Library/Developer/Xcode/DerivedData/PixelArtGallery-*/SourcePackages/artifacts/sparkle/Sparkle/bin/` (from `xcodebuild`)
- **`log show` won't find the Updates lines**: `AppLog.updates` uses `.info`, which is memory-only by default; use `log stream --info` while the app launches instead.
- Resolved version is Sparkle **2.9.4**; `Package.resolved` pins it and is now committed so app and CI builds resolve identically.
