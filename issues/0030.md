## Description

On macOS, clicking + to import presents `ImagePickerView` in a `.sheet`, but the macOS implementation is an `NSViewControllerRepresentable` with an empty view controller that fires `NSOpenPanel.runModal()` after a 0.1-second `asyncAfter` delay. The user sees an empty sheet flash up, then a modal open panel over it, and `runModal()` blocks the main thread while the panel is up. This is both visually broken (blank sheet behind the panel) and the wrong presentation pattern for macOS.

## Steps to reproduce

1. Run the macOS app.
2. Click the + toolbar button in the Gallery.
3. A blank sheet appears, then the open panel; cancel or confirm and note the empty sheet dismissing behind it.

## Expected behavior

The + button opens the file chooser directly with no intermediate blank sheet and no main-thread blocking — e.g. SwiftUI's `.fileImporter` on macOS (or a non-modal `NSOpenPanel.begin`), while iOS keeps the PHPicker sheet.

## Actual behavior

Blank sheet + `runModal()` open panel layered on top; main thread blocked while the panel is open.

## Notes

- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/ImagePickerView.swift` (macOS branch) and the `.sheet(isPresented: $coordinator.showImagePicker)` presentation in `GalleryListView.swift`.
- `.fileImporter(isPresented:allowedContentTypes:)` keeps the coordinator flow intact: read the data from the security-scoped URL and hand it to the same pending-import naming path.

## Root cause

The macOS branch of `ImagePickerView` was an `NSViewControllerRepresentable` wrapping an empty `NSViewController`, presented in a `.sheet`. It fired `NSOpenPanel.runModal()` from a `DispatchQueue.main.asyncAfter(0.1)` — so the user saw a blank sheet, then a modal panel over it, with the main thread blocked for the panel's lifetime. The sheet was the wrong presentation vehicle on macOS; the panel should be presented directly.

## Fix

- `GalleryListView.swift`: split the picker presentation platform-specifically. iOS keeps the existing `.sheet` + `ImagePickerView` (PHPicker) path under `#if os(iOS)`. macOS now uses `.fileImporter(isPresented: $coordinator.showImagePicker, allowedContentTypes: [.image])` under `#elseif os(macOS)`. On success it reads the file inside `startAccessingSecurityScopedResource()` / `stopAccessingSecurityScopedResource()` (sandbox has user-selected-file read-only entitlement) and sets `pendingImport` with the base filename as the suggested name, driving the same ImportNamingView flow. Read failures and importer errors surface via `coordinator.currentError`. Added `import UniformTypeIdentifiers` for `UTType.image`.
- `ImagePickerView.swift`: deleted the macOS `NSViewControllerRepresentable` branch. The shared `defaultImportedImageName` / `effectiveImportedImageName` helpers stay (used by ImportNamingView and unit-tested), and the iOS PHPicker branch stays under `#if canImport(UIKit)`, so the package still compiles for macOS.

## Gotchas

- `PendingImport` is `private` to GalleryListView.swift; the fileImporter callback lives in the same file, so no access-level change was needed.
- `.fileImporter`'s failure case only fires on real errors — user cancellation just dismisses the panel, which is the desired behavior (no error alert on cancel).
