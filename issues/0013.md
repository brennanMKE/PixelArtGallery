## Description

The PRD's typical workflow lets the user create a variant using a discovered FT display's native dimensions (Option A) instead of typing custom width/height. `VariantCreationView` only offers free-form numeric fields — there is no way to pick a display and prefill its `displayWidth`/`displayHeight`, and no `associatedDisplayId` is recorded on the resulting variant.

## Expected behavior

Variant creation lets the user optionally choose a stored FT display; selecting one prefills the target dimensions from that display and tags the created `Variant` with `associatedDisplayId`. Manual entry remains available.

## Notes

- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/UI/VariantCreationView.swift` — add a display picker that writes into the width/height fields.
- `PixelArtGalleryKit/Sources/PixelArtGalleryKit/Models/Variant.swift` — `associatedDisplayId` field already exists.
- PRD §3 "Dimension Matching" and User Workflow step 5 Option A. Depends on 0011 for the display list.

## Root cause

`VariantCreationView` only exposed free-form numeric width/height fields. There was no way to pick a discovered FT display to prefill its native dimensions, and the variant-creation path (`onCreateVariant` → `coordinator.createVariant(for:width:height:)`) had no parameter for `associatedDisplayId`, so the `Variant.associatedDisplayId` field (already present on the model) was never populated.

## Fix

Added an optional FT display picker to `VariantCreationView`, populated via `@Query` over the persisted `FlaschenTaschenDisplay` records (available since #0011). The section appears only when displays exist and defaults to a "Custom" option. Selecting a display prefills the width/height fields from its `displayWidth`/`displayHeight` (PRD Option A); manual entry stays fully available. The recorded `associatedDisplayId` is computed only when the typed dimensions still match the selected display's native size, so editing a field after picking gracefully reverts to a custom (un-associated) variant. The `onCreateVariant` closure signature gained a trailing `UUID?`, `GalleryListView` threads it into `coordinator.createVariant(...)`, and the coordinator's `createVariant` gained an `associatedDisplayId: UUID? = nil` parameter that it sets on the new `Variant`. Scope was kept to dimension matching — send (#0012) and batch creation were untouched.

## Gotchas

`VariantCreationView`'s prefill must not fight manual entry. Rather than clearing the picker selection from the text-field `onChange` (which risks an update loop with the picker's `onChange`), the associated id is derived at create time by comparing the typed dimensions against the selected display's native size — simpler and loop-free.
