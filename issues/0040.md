## Description

The Mac app ships via direct download, so it needs a public site serving three purposes: a marketing/landing page with screenshots and a download link, the Sparkle appcast endpoint (`appcast.xml`) the app polls for updates, and the DMG downloads themselves. Static HTML/CSS in a new `website/` folder at the repo root — no framework, no build step — plus a deploy script.

## Acceptance criteria

- `website/` contains:
  - `index.html` — what Pixel Art Gallery is (import → pixelate → send to Flaschen Taschen displays), screenshots, a prominent Download button pointing at the latest DMG, system requirements (macOS 15+), link to changelog and privacy pages.
  - `changelog.html` — versioned release notes (one `<article id="vX-Y-Z">` per release), linked from the appcast items.
  - `privacy.html` — privacy policy (the app talks to local-network FT displays and optionally saves to Photos on iOS; no analytics/tracking).
  - `appcast.xml` — Sparkle feed, newest item first; items are generated by #0041, not hand-typed.
  - `downloads/` — versioned DMGs (`PixelArtGallery-<X.Y.Z>.dmg`).
  - `assets/` — app icon renders (512/1024), OpenGraph card, and real app screenshots (the grid gallery, a variant with pixel preview, send-to-display).
- `scripts/deploy-website.sh` pushes `website/` to the host via rsync over SSH, driven by env vars (e.g. `PAG_EC2_KEY`, `PAG_EC2_HOST`, `PAG_EC2_PATH`, optional `PAG_EC2_PORT`), validating env vars and key permissions before pushing.
- Site works as plain static files (opens correctly from `file://` or any static host), light/dark friendly, presentable on mobile.

## Notes

- Domain: something under the existing `sstools.co` infrastructure (suggest `pixelartgallery.sstools.co`); the user confirms DNS + hosting. The chosen URL is what #0039 bakes into `SUFeedURL` — record the final URL in this issue when decided.
- The macOS icon artwork in `PixelArtGallery/Assets.xcassets/AppIcon-macOS.appiconset` (512/1024px) supplies the visual identity.
- Screenshots can be captured from the running app; the issue resolver should capture current-state shots rather than reusing stale ones.

## Root cause

Not a bug — a distribution gap. The direct-download macOS build (#0039) bakes `SUFeedURL = https://pixelartgallery.sstools.co/appcast.xml` into Info-macOS.plist, but nothing existed to serve that appcast, host the DMGs it points at, or tell anyone what the app is. The repo had no `website/` and no deploy path.

## Fix

New `website/` at the repo root — plain static HTML/CSS, no framework, no build step, no external resources:

- `index.html` — hero (icon, name, tagline, prominent Download button → `downloads/PixelArtGallery-1.0.0.dmg`, "macOS 15+ · Apple silicon & Intel · Free"), six feature cards (gallery with pin/sort, variants 1×1–512×512, zoom/pan/pixel-select preview, PNG/HEIC/PPM/JSON export, mDNS + manual Flaschen Taschen sending, Sparkle auto-updates), three real app screenshots in a hero shot plus two alternating showcase rows, system requirements, second download CTA, footer links to changelog/privacy/appcast. Full OpenGraph/Twitter meta pointing at `assets/og-image.png` with absolute `https://pixelartgallery.sstools.co/` URLs.
- `changelog.html` — `<article id="v1-0-0">` summarizing the initial feature set, badged "Upcoming" (appcast items link to `changelog.html#vX-Y-Z`).
- `privacy.html` — honest policy: nothing collected (no analytics/telemetry/accounts), images stay local, FT displays contacted only on user-initiated sends over the local network, Sparkle checks fetch the appcast as a plain HTTPS request, iOS Photos saving is user-initiated, static site with no cookies.
- `appcast.xml` — valid empty Sparkle RSS skeleton (channel title/link/description/language + a comment saying items are prepended by the release flow per #0041).
- `css/site.css` — single shared stylesheet: system font stack, light/dark via `prefers-color-scheme` custom properties, responsive grid/flex layouts (feature grid and showcase rows collapse at narrow widths), pixel-quad accents derived from the app icon's four colors.
- `assets/` — `icon-512.png`/`icon-1024.png` copied from `AppIcon-macOS.appiconset` (mac-512 / mac-512@2x), `favicon.png` (64px sips downscale), `og-image.png` 1200×630 rendered via SwiftUI `ImageRenderer` (icon on a dark pixel-mosaic background with title/tagline), and `assets/screenshots/` with three real current-app captures (gallery grid, item detail, variant detail).
- `downloads/README.md` — placeholder documenting that versioned DMGs land here via the release flow (#0041); no DMG committed.
- `website/README.md` — folder map, how the release flow updates appcast/downloads/changelog, deploy instructions.

`scripts/deploy-website.sh` — rsyncs `website/` to `$PAG_EC2_HOST:$PAG_EC2_PATH` over SSH with `-i $PAG_EC2_KEY` (optional `$PAG_EC2_PORT`, default 22). Validates each env var with an actionable per-variable error, requires the key file to exist with 600/400 perms, runs a `--dry-run` preview, then prompts for confirmation (`--yes`/`-y` skips). Uses `--delete` with `--filter='P downloads/*'` so a website deploy can never delete release DMGs on the server. **Not run against any host** — the user deploys.

Also: anchored the `.gitignore` `Screenshots/` rule to `/Screenshots/` — the unanchored form was (case-insensitively) swallowing `website/assets/screenshots/`.

Screenshots were captured per the #0033/#0035 technique: temporary `WebsiteShotHarness` executable target in the package (`swift build --product WebsiteShotHarness -Xswiftc -enable-testing`, `@testable import`), in-memory `ModelContainer`, seeded default FT display (`FlaschenTaschenDisplay.makeDefault()`), 8 generated-pattern gallery items imported through `GalleryCoordinator.createGalleryItem` (two pinned, hero item given 45×35/32×24/16×12 variants, the 45×35 associated with the default display), real views (`GalleryListView`, `GalleryDetailView`, `VariantDetailView`) in dark-aqua `NSWindow`s captured with `screencapture -x -l <windowID>`. Harness removed afterwards.

## Gotchas

- Redirecting `HOME` (the #0035 harness isolation trick) does **not** work on this macOS: `NSHomeDirectory()` and `FileManager.urls(for: .applicationSupportDirectory)` both ignore the `HOME` env var and return the real home. The harness fell back to the #0033 approach — track every imported `originalImagePath` and delete those files from the real Application Support images directory on exit (verified count-neutral).
- The unanchored `.gitignore` rule `Screenshots/` silently excluded `website/assets/screenshots/` (ignore matching is case-insensitive here), so the first commit was missing the screenshots; root-anchoring to `/Screenshots/` fixed it.
- The macOS app icon is literally a 2×2 four-color quad with square corners; presented raw it reads as a color test pattern. The OG card clips it to a macOS-style continuous rounded rect, and the site rounds it with CSS `border-radius`.
- `screencapture -x -l` keeps the native window shadow with transparent margins — the site's CSS deliberately adds no frame/shadow around screenshots.
- xmllint's `--html` mode predates HTML5; `header`/`nav`/`section`/`main`/`footer` warnings are parser noise, not markup errors.
